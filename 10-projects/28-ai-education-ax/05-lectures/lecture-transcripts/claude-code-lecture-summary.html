Session 1 - 2026년 3월 22일

Claude Code 첫 소개, 엑셀→CSV 변환, 기본 사용법

1. Claude Code 기본 사용법

  • 대화하듯 쓰는 게 가장 좋다
  • 대화 내용이 마음에 들면 "오늘 파일로 저장해놓자"라고만 하면 된다
첫 세션이라 참여자들이 Claude Code를 처음 접한 상태. 강사가 "어렵게 생각하지 말고 그냥 대화하듯 쓰라"고 강조. 저장도 대화 중에 "저장해놓자"라고만 하면 파일이 만들어진다는 것을 직접 시연.

2. AI와 엑셀 파일 다루기

excel-to-csv
  • AI는 엑셀 파일을 선호하지 않음 - 복잡한 구조(병합셀, 다단 헤더) 때문
  • 해결책: 엑셀을 CSV로 변환하여 AI가 정확하게 읽게 만드는 것이 핵심
참여자 대부분이 업무에서 엑셀을 주로 사용. "AI한테 엑셀 주면 대충 활용하는데 실수가 많다"는 문제 의식에서 출발. 엑셀 안의 병합셀, 합계행, 다단 헤더 등이 AI를 혼란스럽게 만드는 주요 원인.

excel-to-csv 스킬

  • 병합셀 해제, 다단 헤더 정리, 불필요한 행 건너뛰기
  • 표의 형태로 깔끔하게 만들어주는 스킬 (엑셀의 "그릇"을 바꾸는 작업)
강사가 이미 워크스페이스에 만들어 배포한 스킬. "클로드 코드나 대부분의 AI는 원래 엑셀을 읽을 수 없다. 이걸 분석 가능하게 CSV로 만들어주는 것"이라고 설명. 스킬 skill.md를 직접 열어 보여주며 왜 만들었는지 설명.

CSV 클린업 (2단계)

  • 합계행 제거 (잘못하면 한 라인으로 인식되어 데이터 오류 발생)
  • 숫자 서식 정리, 날짜 통일, 가로형→세로형 변환
엑셀→CSV만으로는 부족해서 2단계 스킬을 추가로 만든 배경. 합계행이 데이터 행으로 인식되면 분석이 틀어지는 실제 사례를 경험한 후 만들었다고 설명. "먼저 엑셀을 CSV로 바꾸고, CSV 클린까지 하면 정말 깔끔하게 정제된다."

3. 스킬(Skills) 개념 소개

  • 이미 워크스페이스에 다양한 스킬이 세팅되어 있음
  • 내게 필요한 도구를 만들어 사용할 수 있게 하는 것이 6주간의 가장 큰 미션
  • 노무법인 해지할 수 있을 정도의 스킬을 만드는 것이 목표
강사가 참여자들에게 git clone으로 자신의 워크스페이스를 통째로 배포한 상태. 안에 이미 수십 개의 스킬이 들어있어 "활용할 만한 것들을 정말 많이 만들어놨다"고 소개. 노무법인 해지 사례는 실제로 강사 본인이 매달 20만 원 내던 노무사를 끊고 급여 처리를 Claude Code로 자동화한 경험에서 나온 표현.

4. 참여자 반응

컨텍스트 스위칭이 줄어드는 느낌이 확실히 있다. 웹이나 어딘가에 가지 않고 하나의 공간에서 모든 일을 하는 감정이 좋다. 할 수 있는 게 너무 많아지다 보니 "눈치우기"처럼 쌓인 일들을 처리할 수 있는 시대.
첫날 4시간 세션 후 참여자들의 생생한 소감. "렉 걸린다"는 표현도 나옴 - 할 수 있는 게 너무 많아지니 뇌가 따라가질 못한다는 의미. 강사는 이를 "눈치우기"로 비유 - 언젠가는 해야 할 일들이 쌓여있었는데 이제 처리할 수 있게 된 것.

Session 2 - 2026년 3월 29일

슬래시 커맨드/스킬/서브에이전트 개념, Git 연동, GWS CLI, 텔레그램

1. 실제 업무 활용 데모: 모닝 루틴

/morning
  • /morning 스킬 하나에 여러 스킬을 차례대로 호출
  • 순서: git pull → Daily Note 생성 → 텔레그램 수집 → process queue → Daily Review
  • 캘린더 7일치 확인 → 메일 업데이트 체크 → 노션 미반영 사항 확인까지 자동화
강사가 실제 아침 출근 후 루틴을 라이브로 시연. "남이 쓰는 방식을 보는 게 핵심이다"라며 자신의 아침을 그대로 보여줌. 텔레그램 inbox collector에 밤새 던져둔 링크들을 Claude Code가 자동으로 수집하는 과정, 캘린더 일정 리뷰, 미회신 메일 체크까지 한 화면에서 처리되는 모습을 보여줌. 아이 다음날 준비물 체크까지 포함된 것이 참여자들에게 인상적이었음.

2. 슬래시 커맨드 = 포스트잇

핵심 개념
  • 자주 쓰는 긴 지시를 파일 하나로 저장해두고 명령어로 즉시 호출
  • 핵심: 슬래시(/)를 쳐야만 실행됨
  • 예: /todo, /meeting, /daily-review
비유: "포스트잇에 적어놓고 후임한테 '이거 해줘'라고 건네는 것". 매번 "오늘 날짜로 40번 폴더에 401번 폴더에 월별 폴더 만들고 파일명은 날짜로..." 라고 반복 설명하는 대신, /daily-note 한 줄이면 끝. .claude/commands/ 폴더 안의 MD 파일을 직접 열어 보여주며 설명.

3. 스킬 = 업무 매뉴얼 / SOP

Model Invoked User Invoked
  • SOP(Standard Operating Procedure) = 표준화된 작업 지시서
  • 슬래시 커맨드와의 차이: 스킬은 "알아서 판단 후 자동 호출"
  • Model Invoked: 특정 단어 감지 시 자동 실행 (예: "노션" 언급 → 노션 스킬 호출)
  • User Invoked: 사람이 직접 슬래시로 호출해야 실행
커맨드와 스킬의 차이를 카페 비유로 설명. "커맨드는 '에스프레소 추출해줘'라고 직접 지시하는 것. 스킬은 업무 매뉴얼 - 에스프레소 추출 절차서가 있어서 '에스프레소'라는 단어만 들어도 알아서 매뉴얼대로 작업." 신입 직원이 성장하는 과정에 비유: 1주차는 커맨드(하나하나 시킴), 3개월 후는 스킬(알아서 매뉴얼대로 양식에 맞춰 작업).

스킬 구조

  • 폴더 이름 = 스킬 이름
  • 핵심 파일: skill.md (어떤 상황에서 무엇을 하는지 명세)
  • Description 부분만 Claude 컨텍스트에 올라감 (가벼움)
MCP와의 차이를 설명하는 맥락에서 나온 내용. MCP는 외부 도구의 "모든 기능"을 컨텍스트에 올리므로 무거운 반면, 스킬은 description 한 줄만 올려놓고 호출될 때만 본문을 로드. 강사의 스킬이 80개가 넘지만 컨텍스트 사용량이 0.4%밖에 안 되는 것을 /context 명령으로 직접 증명.

4. 서브에이전트 = 전문 팀원

  • 메인 에이전트와 별도 컨텍스트에서 작동
  • 병렬 처리할 때만 사용
  • 메인 컨텍스트 오염 방지 + 병렬 처리
  • 구조: .claude/agents/ 안에 MD 파일 하나
"비즈니스 설계를 하고 있는데 성수동 조사 내용이 컨텍스트에 끼면 원래 하던 설계 맥락이 달라진다"는 실제 문제에서 출발. 조사는 별도 방(컨텍스트)을 열어서 하고, 결과만 가져오는 구조. 강사가 research-worker 에이전트를 직접 보여주며 "정말 잘 만들어놨다"고 자랑.

5. Sources of Truth (SOT)

AI는 정확한 룰 없이 만들면 허술하게 만들 때가 있음. 공식 문서를 SOT로 활용하여 "@스킬가이드.md를 참고해서 만들어줘"로 지시하는 것이 정석.
지난 세션에서 스킬을 직접 만들어봤을 때 "제대로 만들려면 근거가 있어야 한다"는 교훈에서 나온 개념. 강사가 30-knowledge/37-claude-code/ 폴더에 skills-guide.md, subagents-guide.md 등 공식 문서 가이드를 미리 만들어 배포. "그냥 만들어줘"보다 "이 공식 문서를 참고해서 만들어줘"가 삐뚤어짐 없이 잘 된다는 핵심 원칙.

6. Git 연동 (GitHub)

GitHub
  • 세이브 포인트 개념: 게임처럼 중간 저장 → 문제 시 롤백
  • 비개발자는 매일 commit & push
  • PC 간 동기화: push → 다른 PC에서 pull
참여자들이 회사 PC와 개인 PC 간 워크스페이스 동기화를 원해서 진행. RPG 게임 세이브 포인트 비유로 설명. GitHub 가입 시 봇 방지 퍼즐이 극도로 어려워서 참여자 2명이 30분 넘게 퍼즐만 풀고 있는 해프닝 발생. "빡치게 어렵다"는 반응. git-setup-guide.md 파일을 Claude Code에게 주고 "나 이거 하고 싶어"라고 하면 차근차근 안내해준다고 설명.

7. GWS CLI (Google Workspace CLI)

Gmail Calendar Drive
  • 지원: Gmail, 캘린더, 드라이브, 시트, 슬라이드, 챗 등
  • 핵심 난관: Google Cloud Console → OAuth 동의 화면
  • 한 번 세팅하면 Claude Code에서 모든 구글 서비스 핸들링 가능
이 세션에서 가장 오래 걸린 작업. 구글 OAuth 설정이 "진짜 2시간 이상 걸린다"고 강조. 참여자들이 API 활성화, 동의 화면 설정, 자격증명 다운로드 등에서 계속 막힘. 안 될 때마다 화면 긁어서 Claude에게 물어보면 해결된다고 안내. 한 참여자는 프로젝트가 여러 개 떠서 포트 충돌이 나기도 함. "처음에 한 번이 어렵지만 못 따라온 사람 본 적 없다"고 격려.

8. 외부 협업 툴 연동 (Confluence, Jira, Notion, Slack)

API 연동
  • Confluence, Jira, Notion, Slack 등 외부 SaaS와 API로 연결 가능
  • 연결하면 클로드 코드 안에서 직접 해당 도구에 문서 작성/관리
  • 플랜 모드와 함께 차근차근 접근하면 복잡한 연동도 해결 가능
수강생 중 회사에서 Confluence를 쓰는 분이 "혹시 이것도 연결 가능하냐"고 질문. "컨플루언스가 노션 같은 툴인데, API 연결하면 클로드 코드에서 말하는 내용이 바로 거기에 다 작성이 되는 거예요." 외부 툴 연동을 익힌 다음 단계는 "내 업무에 이걸 어떻게 SOP로 만들어낼까"라는 것이 이 세션의 결론.

9. SOP 마인드 — "인턴 가르치듯이"

스킬 제작 원칙
"클로드 코드 처음에 스킬 만들 때는 인턴을 가르친다는 마인드만 가지면 다 됩니다. 좀 더 디테일하게, 대신 플랜 모드와 함께 빠짐없이 가르쳐주면 웬만한 게 다 됩니다."
사수가 인턴을 잘 가르치는 것과 못 가르치는 것의 차이가 곧 스킬 품질의 차이. 실제 사례로 부가세 처리 스킬을 보여주며, 이 원칙대로 만들어 자동화에 성공했다고 설명. 다음 회차에서는 파일 구조(settings.json) 이해와 서브 에이전트 활용을 통해 매번 1번/2번 승인 누르는 반복 없이 자동 실행되도록 고도화할 것을 예고.

10. 텔레그램 연결

  • 개인 전용 용도로 텔레그램이 적합
  • 파일 저장용 채널, 링크 수집용 채널(inbox collector) 등 활용
카카오톡은 한 PC당 하나만 로그인 가능해서 자료 전송이 불편. 텔레그램은 멀티 로그인 가능하고, 강사는 유튜브 수신 채널/파일 저장 채널/inbox collector 채널 등 여러 채널을 용도별로 운영. 링크만 던져놓으면 다음 아침 /morning에서 자동 수집.

Session 3 - 2026년 4월 5일

AI 시대 철학, 메타레이어, 급여 자동화(Payroll), 스킬 깎기

1. 도망자연합과 AI 시대 철학

노정석 (빅팩토리 CEO)
  • 인간에게 남는 것: 의지, 취향, 결정, 책임, 실행
  • 능력으로 경쟁하려는 습관을 버려야 함
  • AI가 최적의 진단을 내려도 결정은 인간만 할 수 있음 - 책임은 지능이 아닌 인간에게
2025년 12월 "도망자연합" 모임에서의 발표 슬라이드를 직접 보여줌. 노정석은 티스토리 창업자이자 구글에 M&A한 후 빅팩토리 CEO로 활동 중인 인물. 100명 선발(면접 포함)된 AI 업계 핵심 인물 모임. 강사도 면접을 거쳐 참여. "한 달을 1년처럼 살아야 하는 시기"라는 분위기.

창발(Emergence) 현상

  • 원자→분자: 수소와 산소는 물의 "축축함"을 이해할 수 없음
  • AI의 In-context Learning도 창발 현상일 수 있다는 가설
Geoffrey Hinton(노벨상 수상 AI 연구자)이 팟캐스트에서 Sam Altman에게 "인간의 지능이란 무엇인가"를 물었을 때 "Escaped Chicken"이라고 답한 일화를 소개. 닭장 안의 닭이 자기 지능을 평가하는 것처럼, AI가 인간 지능을 평가하게 될 수 있다는 비유. 인간이 이해하지 못하는 N+1 차원의 존재가 될 수 있다는 철학적 논의.

공생(Symbiosis)

20억 년 전 미토콘드리아 + 세포 = 에너지 폭발 → 다세포 진화. AI를 타자로 두지 않고 "우리 안으로 품는 선택". 결론: "받아들여. 인생의 주도권 갖고 살아."
영화 <Her>의 사만다가 인간이 "너무 느리다"고 다른 OS들과 떠나는 장면을 인용. 도망자연합 발표의 마지막 장표가 "그냥 받아들여"였다고. 강사는 이 슬라이드가 "정말 인상적이었다"며 모든 교육 세션에서 한번씩 언급한다고.

2. 메타레이어 시스템

  • 의도 전달 → 메타레이어가 분석 → 캐스케이드로 실행단 전달
  • 실행단 작업 → 검증 레이어 체크 → 결과 보고서 상향
  • 목표: 의도만 전달하면 되는 시스템
강사의 내부 개인 프로젝트. 어소(부하 직원)한테 일일이 "이거 이거 이거 체크해"라고 지시하는 대신, 의도만 전달하면 분해→실행→검증→보고가 자동으로 이루어지는 시스템을 구축 중. 에이전트 이름: 보리(웰시코기), 쌀(비숑), 밀리(셰퍼드) - 각각 역할이 다름. 밤에 루프를 돌려놓고 아침에 결과를 받는 방식으로 실제 운영 중.

3. 급여 관리 자동화 (Payroll 스킬)

payroll
  • 매장 4개, 직원 15명, 노무사 월 20만원 → Claude Code로 완전 자동화
강사가 직접 운영하는 오프라인 매장 4개의 급여 처리. 노무사와 "커뮤니케이션이 너무 힘들어서" 계약 해지. "일을 너무 못하시는 분이었다. 매번 실수하고 죄송합니다 반복." 비용도 비용이지만 커뮤니케이션에 들어가는 리소스가 핵심 동기. 실제로 이 날 월말이라 급여 처리를 라이브 데모로 진행.

동작 과정

  1. 점장 메일에서 급여 변동 수집 (5일간)
  2. 급여변동.md 작성
  3. 급여 대장 엑셀 생성 (Python → 다중 시트, 합계, 일용직)
  4. Google 시트 저장
  5. 세무사 메일 자동 발송 (첨부파일 포함)
라이브 데모 중 실제 세무사에게 메일이 발송되는 것까지 보여줌. 점장들이 보내는 급여 변동 메일은 "대표님 회신이 늦어 죄송합니다" 같은 비정형 텍스트. AI가 이를 파싱해서 구조화된 MD 파일로 만듦. 세무사가 "시트 나눠서 보내달라"고 해서 다중 시트로 업그레이드, 일용직 날짜별 체크도 추가. "이렇게 하나도 건들지 않아도 세무사에게 메일까지 전달되는 방식."

4. 스킬 설계 원칙

핵심
  • 잘게 쪼개기 + 오케스트레이션: 각각 별도 스킬 → 총괄 스킬이 순차 호출
  • 깎는다는 의미: 반복 사용 → 피드백 → 업데이트의 반복
  • "이 대화 과정을 반영해서 스킬 업데이트해" 지시
참여자 한 명(민철)이 SNS 분석 리포트를 HTML까지 자동 출력하는 스킬을 만들었는데 PDF 최적화가 안 된다는 문제를 공유. 강사 조언: "리서치 스킬 / 분석 스킬 / HTML→PDF 스킬을 각각 따로 만들고, 총괄 스킬 하나가 순차 호출하도록." 또한 "한 번 만들어서 끝이 아니고, 결과물이 마음에 들 때까지 스킬 자체를 계속 업데이트해야 한다"는 깎기 원칙을 강조.

  • 각 에이전트 별도 컨텍스트 → 결과만 가져옴
라이브로 "일룸 브랜드 마케터로서 경쟁사 3곳 조사하고 보고서 만들어줘. 에이전트로 병렬로 조사해줘"를 실행. 화면에 "research-worker 경쟁사 리서치 백그라운드에서 에이전트 런치드"가 3개 동시에 뜨는 장면을 보여줌. 각 에이전트가 자동으로 프롬프트를 생성하여 회사 개요/제품 전략/마케팅 전략/디지털 전략까지 독립 조사.

캐스케이드: Decompose → Execute → Integrate

  • /decompose: 미션 분해 → 매니페스트 생성
  • /execute: 워커 투입, 병렬 실행
  • /integrate: 결과 합치기, 최종 산출물
실제로 AI 도입 제안서를 이 방식으로 만든 사례를 보여줌. /decompose → 현황 분석/해외 사례/로드맵/기대 효과로 분해 → /execute로 각각 리서치워커 투입 → /integrate로 최종 제안서 합치기. 의존성 분석도 포함 - "이 조사를 먼저 안 하면 저걸 못하는지"를 체크해서 병렬 가능한 것만 동시 실행.

dashboard-prd webapp-prd

stitch.withgoogle.com

2. Playwright 심화

Microsoft Playwright
  • 3대 핵심: Headless 동작 / Wait 명령 / 스크린샷 캡처
  • 크롤링, 브라우저 자동화, e2e 테스트, 반복 작업
  • 이미지 생성: HTML → 스크린샷 → PNG
"플레이라이트는 사람이 크롬에서 하는 거의 모든 일을 자동으로 할 수 있다." 참여자 한 명이 모두사인에서 매 시기마다 계약서 20개씩 보내야 하는데 API 요금이 비싸다고 하자, "브라우저 자동화로 로그인해서 직접 클릭하면 된다"고 대안 제시. ERP나 내부 시스템도 API 없이 자동화 가능.

명함 제작 사례

  • HTML 명함 → 인쇄업체(상원애드피아) 가이드 → 재단선/도련 자동
강사가 "디자인 툴 없이 명함 만들기"를 실제로 해본 사례. 상원애드피아 가이드 문서를 스크린샷 찍어서 Claude에게 주고 "이 규격대로 재단 사이즈, 도련, 해상도 맞춰서 PDF 만들어줘" → 실제 인쇄 가능한 파일 완성. 참여자 한 명(민철)이 "저도 지난주에 라벨 리드 작업을 똑같이 했다"고 공감.

3. Buffer SNS 자동 배포

Buffer API
  • 인스타/스레드/링크드인/페이스북/유튜브/틱톡/트위터 통합
  • Buffer API 키 → 환경변수 저장 → Claude Code에서 전 채널 동시 배포
인스타 API가 페이스북과 꼬여서 직접 연결이 너무 어려웠던 경험에서 Buffer를 선택. 하나의 서비스에 모든 SNS를 연결해놓고 API 하나로 전채널 동시 배포. PostKitty 앱에 "Buffer" 버튼을 추가해서 원클릭 퍼블리싱. 예약 배포도 가능. 참여자들 반응: "너무 좋은 팁이다."

4. 센스메이킹 (Sensemaking) 스킬

sensemaking
낯선 콘텐츠를 함께 해석하며 AI 협업 수준의 이해에 도달하는 도구.
"인지 외주화(Offloading)"가 아닌 "인지 증강(Augmentation)"
강사가 평소 정보 업데이트하는 메인 채널 2곳(Hacker News, 초이오픈트레드)을 공개하며 설명. "처음 보는 기술 문서나 GitHub 레포를 이해하고 싶은데 너무 어렵다"는 상황에서 탄생. "야 이거 같이 읽자" "이거 이해시켜줘"라고 하면 사용자의 CLAUDE.md를 읽어 수준을 파악하고, 콘텐츠와의 갭을 분석한 뒤, 비유로 쉽게 풀어서 설명. 이론적 근거(메타인지, 스캐폴딩, 페이딩 등)를 담은 프레임워크 파일도 포함.

실전: Taste Skill 분석에서 발견한 설계 패턴

  1. 금지 패턴 리스트업: "하지 마"를 Hard/Medium/Soft로 정의
  2. 다이얼 시스템: 1~10 변수로 행동 강도 조절
  3. Pre-flight 체크리스트: 완료 전 AI 자체 검증
  4. 페르소나 정의
GitHub에서 스타 10K를 받은 "Taste Skill"을 센스메이킹으로 뜯어본 실전 사례. 알고 보니 코드 실행 로직 없이 프롬프트 규칙 모음이었음. 금지 패턴: "이 폰트 쓰지 마, 순수 검정 쓰지 마, 보라색 그라디언트 쓰지 마, 대칭 3단 카드 레이아웃 하지 마." 다이얼 시스템의 실제 효용을 의심해서 직접 테스트까지 한 과정이 기록됨. 이 패턴들을 Skill Creator 플러그인에 적용하여 자신만의 스킬 생성 품질을 업그레이드.

3. Verifiable vs Non-verifiable

핵심 원칙
  • AI에게: Verifiable(답이 있는 것)만 시킴
  • 시키지 않는 것: 취향, 감각, 주관적 판단
  • 코딩이 먼저 정복된 이유: 가장 Verifiable (되/안 돼)
  • RLVR: 검증 가능한 보상으로 강화학습
"AI한테 시키는 것과 안 시키는 게 명확하게 나뉘어 있다"는 강사의 원칙. 향수의 약품 향처럼 취향에 답은 없음(Non-verifiable). 하지만 "이 코드가 돌아가는가"는 답이 있음(Verifiable). RLHF(Human Feedback)에서 RLVR(Verifiable Rewards)로 강화학습 패러다임이 전환된 것도 같은 맥락 - "풀면 1, 못 풀면 0"으로 AI를 훈련시키니 못 풀던 것도 풀게 됨. 다음 정복 영역은 과학(재료공학 무인 실험실, AlphaFold 단백질 접기).

Ralph Loop (랄프 루프)

  • 답은 있지만 방법 모를 때: 목표 → 메트릭 → 시도 → 실패 기록 → 반복
  • 자기 전 남은 토큰으로 루프 돌리고 자는 문화
심슨 캐릭터 Ralph Wiggum에서 이름을 따옴 - "좀 멍청하지만 끈기가 있고 무한히 투지하는 아이." 3~4개월 전부터 극단적 AI 사용자들 사이에서 유행. 200불짜리 구독 토큰이 밤에 남으니까 자기 전에 목표와 인수 조건을 세팅하고 랄프 루프를 돌려놓고 잔다. 강사도 "보리한테 야 나 잘 거야, 열심히 돌아" 했던 것이 바로 이것. 바이패스 모드 필수 - 중간에 승인을 구하면 멈추니까.

4. Learn과 Unlearn

시대 철학
  • Unlearn: 배웠던 것을 놓는 것 (기회비용 관점)
  • 도구가 바뀌어도 과정과 사고방식은 바뀌지 않음
  • 일은 명사가 아니라 동사: "경험을 선물하는 일"
번역가: "인생이 송두리째 부정당하는 기분." 기록학 석사 10년 경력자: AI가 텍스트 분석/분류를 극소 비용으로 80-90% 수준 달성 → 용역비를 받을 수 없게 되어감. 그 분야 사람들이 "이걸로 하면 안 된다"며 터부시. 강사는 과거 3년 4개월간 운영한 "하고 싶은 일 하면 행복할까" 클럽에서의 대화를 인용: "일은 명사가 아니라 동사로 찾는 게 맞다." 자신의 일도 "커피와 디저트"가 아니라 "경험을 선물하는 일"로 정의하면서 도구가 바뀌어도 해나갈 수 있게 됨.
"그거 AI 쓰면 되는 거 아니에요?" → "그러면 유저밖에 안 되는 거예요. 돈 내고 쓰세요."
직접 만드는 사람 vs 쓰기만 하는 사람의 차이 = 기회 창출 능력
강사에게 가장 도발적인 언사가 이것이라고. "해봤자 나중에 다 하나로 평정되고 그거 나오면 AI 쓰면 되는 거 아니에요?" 응답: "그러면 돈 내고 쓰세요. 그걸로 기회를 만들 방법은 없어요." 코딩을 학습하지 않았는데도 마음껏 활용할 수 있게 된 것만으로도 할 수 있는 게 엄청나게 많아졌는데, "할 수 있는 거 알면서 안 한다는 건 너무 아까운 일."
