## 2. 본 강의 — 클로드 코드의 파일 구조

### 2-1. 숨김 파일 보기

**macOS:**
- Finder에서 `Command + Shift + .` → 숨김 파일 표시

**Windows:**
- 탐색기 → 보기 → 숨김 항목 체크

### 2-2. `.claude` 폴더 = 클로드 코드의 설치 위치

> "여러분들은 클로드 코드를 설치를 했잖아요. 근데 그 클로드 코드가 어디에 설치되어 있는지도 아직 몰랐던 거예요."
— 강사

- 사용자 홈 디렉토리 아래 `.claude` 폴더가 클로드 코드의 본체
- 경로: `사용자/[이름]/.claude/`

### 2-3. 본사와 지점 비유 — 전역 vs 프로젝트 설정

> "`.claude`라고 하는 큰 클로드 코드 설치 폴더가 **본사**라고 생각하시면 돼요. 그러면 do-better 워크스페이스라는 곳은 **지점**입니다."
— 강사

| 구분 | 위치 | 역할 |
|---|---|---|
| **본사** (전역) | `~/.claude/CLAUDE.md` | 모든 프로젝트에 적용되는 설정 |
| **지점** (프로젝트) | `do-better-workspace/CLAUDE.md` | 해당 워크스페이스에서만 적용 |

- 본사에 룰이 있고, 지사에도 룰이 있을 수 있음
- 지사(워크스페이스)에서만 작업하면 지사 CLAUDE.md만 읽음
- **다른 폴더에서 클로드 작업 시** → 지사 CLAUDE.md를 읽지 않으므로 "내가 누군지 몰라"

> "더 좋은 거는 본사에 있는 CLAUDE.md를 잘 만드는 게 굉장히 좋다."
— 강사

### 2-4. 전역 CLAUDE.md 만드는 법

> "지금 내 do-better 워크스페이스 안에 있는 CLAUDE.md를 **전역, 즉 전 지역에서 사용하는 전역 CLAUDE.md로 바꿔줘** 하면 알아서 바꿔줄 거구요."
— 강사

### 2-5. 스킬/에이전트/커맨드도 전역 vs 프로젝트

> "do-better 워크스페이스 안에 있는 에이전트, 커맨드, 스킬들... 이것도 역시 지점의 스킬, 커맨드, 에이전트인 거예요. 이 do-better 워크스페이스 안에 있을 때에만 사용할 수 있는 스킬과 에이전트와 커맨드였던 거예요."
— 강사

- 전역으로 바꾸려면: "전역 스킬로 바꿔줘", "전역 커맨드로 바꿔줘", "전역 에이전트로 바꿔줘"
- 강사는 대부분의 스킬을 전역 스킬로 설정해 둠
- **팁**: do-better 워크스페이스 안에서만 일한다면 굳이 전역으로 바꿀 필요 없음

### 2-6. 컨텍스트 효율성 고려

> "스킬 이런 것들은 전부다 AI가 맨 처음에 로딩을, 로드를 시켜요. 컨텍스트를 먹고 있는데..."
— 강사

- 전역 스킬이 많으면 컨텍스트를 많이 소비
- **특정 프로젝트에서만 필요한 스킬**은 해당 프로젝트 폴더(지점)에만 두는 것이 효율적

### 2-7. 프로젝트별 스킬 사례 — 웹노블 프로젝트

> "웹소설 프로젝트에는 지점 `.claude` 폴더가 하나 더 있어요. 여기서만 쓰는 스킬들이 있을 거 아니에요."
— 강사

웹노블 프로젝트 전용 스킬/에이전트:
- **챕터 에디터**: 챕터 구성
- **캐릭터 보이스 키퍼**: 캐릭터 말투 일관성 유지
- **컨티뉴이티 체커**: 스토리 일관성 검증
- **플롯 트래커**: 줄거리 추적
- **세계관 빌더**: 세계관 설정

> "섹션을 3개로, 에이전트를 3개로 나누면 한쪽에서는 계속 글을 쓰고 있어도 됩니다."
— 강사

### 2-8. CLAUDE.md 계층 구조 정리

```
~/.claude/CLAUDE.md              ← 전역 (본사): 모든 곳에서 읽음
├── 프로젝트/CLAUDE.md           ← 지점: 해당 프로젝트에서 추가 읽음
│   └── 프로젝트/하위폴더/CLAUDE.md  ← 분점: 더 세부적으로 추가
```

> "전체 다 읽고 있고, 없는 내용을 이렇게 추가 추가 하면 되는 거예요."
— 강사

## 3. CLAUDE.md 작성법 — 강사의 전역 CLAUDE.md 실례

### 3-1. GWS CLI 연동 명시

> "나는 구글 분명히 연결을 해놨는데 얘가 그거를 까먹는 이유는, 그걸 네가 명시적으로 표시해놓은 게 없잖아. 즉, 'GWS CLI로 구글 다 연결하는데 이 내용 CLAUDE.md에 업데이트해놔라'고만 해보면 한 줄 들어가요. **그 한 줄이 들어가서 중요합니다.**"
— 강사

- 연동 서비스(구글, 텔레그램, 노션 등)를 CLAUDE.md에 명시해야 클로드가 "딴짓"을 안 함

### 3-2. 강사의 전역 CLAUDE.md 구성 요소

| 섹션 | 내용 |
|---|---|
| **환경 정보** | macOS + Windows 사용, 각 폴더 이름/경로 차이 명시 |
| **사용자 프로필** | 1인 비즈니스, 핵심 전문성(오프라인 비즈니스 경험 + AI 실무), 현재 역할 |
| **비즈니스 현황** | 마케팅, 블로그, 사이트 URL 등 |
| **핵심 서비스** | 컨설팅, 파이프라인, 오프라인 운영 등 |
| **내부 프로젝트** | 멀티패션 자동 프레임워크, 시크루프 등 |
| **작업 스타일** | 클로드 코드 중심, 모든 업무를 워크스페이스에서 처리 |
| **인프라** | 서버(오픈클로 보리 상시 운영), 자동화, 커뮤니케이션 채널 |
| **폴더 구조** | Johnny Decimal 구조 (00~90) |
| **파일 명명 규칙** | YYYY-MM-DD 형식 |
| **작업 규칙** | 기존 파일 편집 우선, 요청 없이 커밋 금지 |
| **이모지 규칙** | "명시적으로 요청하면 이모지 쓰지 마" |
| **문제 해결 방식** | 증상만 해결 금지 → 원인 먼저 파악, "왜"를 먼저 물어볼 것 |
| **연쇄 문제 감지** | 임시 우회보다 근본 해결을 기본으로 선택 |

### 3-3. 핵심 작업 규칙 직접 인용

> "에러나 문제 발생 시 제발 증상만 해결하지 말고 원인을 먼저 파악해줘."
— 강사 CLAUDE.md

> "제발 '왜'라고 물어봐. 표면적 수정 전에 왜 이 문제가 발생했는지 분석 좀 해."
— 강사 CLAUDE.md

> "임시 우회보다 근본 해결을 기본으로 선택해줘. 자꾸 '이렇게 해도 넘어가도 될까요'라고 너무 쉬운 길을 택하려고 한단 말이에요."
— 강사 CLAUDE.md

## 4. Source of Truth & 공식 가이드 파일

### 4-1. Source of Truth 개념

> "스킬 만들 때는 Source of Truth 해서, 클로드 가이드.md를 바탕으로 스킬 만들어줘."
— 강사

- 클로드 코드 공식 문서를 로컬에 저장해두면 **정확한 기준**으로 스킬/에이전트 생성 가능

### 4-2. 보관할 가이드 파일 목록

- `CLAUDE-MD-BEST-PRACTICES.md` — CLAUDE.md 작성 가이드
- `skill-guide.md` — 스킬 만들기 가이드
- `sub-agent-guide.md` — 서브 에이전트 가이드
- `claude-code-guide.md` — 클로드 코드 전체 가이드

> "여러분들이 CLAUDE.md를 만든 다음에, 이 CLAUDE.md 가이드에 맞춰서 제대로 작성되는지 체크해봐, 미진한 게 있으면 보완하자 하면 같이 CLAUDE.md를 잘 만들 수 있는 겁니다."
— 강사

### 4-3. 스킬/에이전트 생성 공식

```
"@[가이드 파일]을 기준으로 [스킬/에이전트] 만들어줘"
```

예시:
- "서브 에이전트 가이드.md를 기준으로 서브 에이전트 만들고 싶어"
- "스킬 가이드.md 참고해서 스킬 만들고 싶어"
