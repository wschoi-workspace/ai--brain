## 3. Playwright — 브라우저 자동화 실무 사례

> "ERP 쓰시는 팀 있으십니까? API를 주지 않으면 로그인도 없고... 예를 들면 모두사인에서 매 특정 시기마다 계약서를 20개씩 보내야 된다는 거예요. 근데 API 요금제가 너무 비싸고. 그래서 **브라우저 자동화**로 어디에 내 로그인하고 이런 것들만 다 가르쳐주면 얘가 알아서 할 수 있도록도 가능하거든요."

- Playwright = 웹 핸들링 도구

- ERP, 전자서명, 반복적 웹 작업에 활용

## 5. 서브 에이전트 — 핵심 개념

### 5-1. 서브 에이전트란?

> "원래 클로드 코드에 뭔가를 시키면 기본적으로 한 에이전트가 모든 걸 한 명이 다 하는 구조죠. 질문도 받고 조사도 하고 분석도 하고 글도 쓰고. 서브 에이전트라는 거는 **한 가지 특정 역할만 하는 전문 팀원을 추가하는 거**라고 생각하시면 됩니다."
— 강사

### 5-2. 서브 에이전트를 쓰는 이유 = 병렬 처리

> "유일하게 저는 **병렬 처리를 위해서** 서브 에이전트를 활용합니다. 병렬 처리만입니다."
— 강사

**순차 처리 (서브 에이전트 없이)**:
```
경쟁사 A 조사 → B 조사 → C 조사 → 비교 분석 → 보고서 작성
```

**병렬 처리 (서브 에이전트 활용)**:
```
리서치 워커 A ─┐
리서치 워커 B ─┼→ 결과 종합 → 보고서 작성
리서치 워커 C ─┘
(동시에 백그라운드 실행)
```

### 5-3. 핵심: 별도 컨텍스트 윈도우

> "서브 에이전트의 가장 핵심적인 기능은, 이 대화창 메인 창에서 컨텍스트가 돌아가는 게 아니고, **백그라운드에서 각각 리서치 워커 하나당 컨텍스트가 하나씩 별도로 생깁니다.** 이게 가장 핵심이에요."
— 강사

- 메인 컨텍스트에 오염되지 않음
- 조사한 내용만 결과로 가져옴
- 메인 대화창에서는 다른 작업 계속 가능

> "서브 에이전트를 쓰면 내 메인 컨텍스트에 오염되지 않는 별도의 개별 컨텍스트들이 백그라운드에서 떠서, 여기서 조사한 내용만 갖고 오는 거라고 생각하시면 됩니다."
— 강사

### 5-4. 토큰 효율성

> "메인 대화창에 토큰을 쓰다 보면 대화를 하다 보면 점점 점점 금방 토큰이 소모가 되거든요. 클로드는 이 앞에 있는 컨텍스트를 다시 다 그대로 갖고 와서 계속 대답을 하다 보니까 금방 금방 차거든요. 서브 에이전트는 별도로 하니까 훨씬 더 토큰도 더 잘 쓰는 게 좋습니다."
— 강사

## 6. 리서치 워커 — 조사 전문 에이전트

### 6-1. 역할 정의

> "외부 정보를 수집하고 검증하여 의사 결정에 필요한 팩트를 전달한다."
— 리서치 워커 정의

규칙:
- 한국어/영어 병행 검색
- 최소 3개의 독립 소스 체크
- 근거 없는 주장 전달 금지
- 모르는 건 모른다고 답변
- 결과는 짧고 핵심적으로

### 6-2. 리서치 방법론 (에이전트에 내장)

> "리서치 요청을 받으면 그냥 검색, 바로 검색하지 말고 일단 질문을 하위로 쪼개라고 요청을 했습니다."
— 강사

| 단계 | 내용 |
|---|---|
| 1. 질문 분해 | 하나의 질문 → 공식 소스 / 리뷰·커뮤니티 / 시계열 등 하위 질문으로 |
| 2. 탐색 확장 | 초기는 넓게, 좋은 소스 발견 시 거기서 확장 |
| 3. 모순 처리 | 상충되는 정보 발견 시 교차 검증 |
| 4. 빈칸 채우기 | 빙고처럼 누락된 영역 체크 |
| 5. 종합 정리 | 핵심만 정리하여 전달 |

### 6-3. 실제 시연 — 경쟁사 분석

> "경쟁사 3곳을 병렬로 조사하겠습니다 하면서 리서치 워커라는 애가 뜨죠. 백그라운드에서 에이전트 런치드. 3개의 리서치를 병렬로 실행했습니다. 에이전트 완료되면 알람이 오고 종합 분석 보고서 작성을 시작하겠습니다."
— 강사 시연

- "에이전트로 병렬로 조사해줘"라고 명시하면 확실
- 에이전트가 자체적으로 상세 프롬프트 생성 (회사 개요, 제품 전략, 마케팅 전략, 온라인/오프라인 전략 등)

## 8. Decompose 스킬 — 미션 분해

### 8-1. 개념

> "큰 프로젝트를 하려고 해. 인간의 인지할 수 있는 범위는 한계가 있으니까. 다들 내 경험 안에서만 고민을 하잖아요. 근데 세상에는 훨씬 더 다양한 생각들이 있을 수 있으니까, 먼저 뭔가 미션을 받으면 Decompose를 씁니다."
— 강사

### 8-2. Decompose 프로세스 (7단계)

| 단계 | 내용 |
|---|---|
| **1단계** | 최종 산출물 정의 (문서? 코드? 데이터?) |
| **2단계** | 산출물에 필요한 입력 정보 파악 |
| **3단계** | 각 입력을 어떻게 얻을지, 누가 할지 결정 (워커 선택) |
| **4단계** | 의존성 분석 (먼저 해야 할 것 vs 동시 가능한 것) |
| **5단계** | 작업 분류 (병렬 가능 vs 순차 필수) |
| **6단계** | 저장 경로 미리 설정 |
| **7단계** | 매니페스트(작업 지시서) 생성 |

### 8-3. 워커 선택 가이드 (Decompose 내장)

```
리서치 → 리서치 워커
분석   → 어낼리시스 워커
콘텐츠 → 콘텐츠 워커
디자인 → 디자인 워커
개발   → 개발 워커
```

### 8-4. 의존성 분석의 중요성

> "코딩을 할 때 만약에 한 파일을 두 에이전트가 동시에 붙이고 있으면 완전 문제가 생기거든요. 조사를 할 때도 한 회사에 대해서 두 개가 조사하면 잘못될 수가 있거든요."
— 강사

- 의존성이 있는 작업끼리는 순차 처리
- 의존성이 없는 작업은 병렬 처리

### 8-5. 실제 사례 — 경쟁사 보고서

미션: "경쟁사 3곳 조사하고 비교 분석 보고서 만들어줘"

Decompose 결과:
1. 최종 산출물 = 경쟁사 비교분석 보고서
2. 필요 입력 = 각 경쟁사 정보(가격, 핵심 서비스, 타겟 고객, 온라인/오프라인 전략)
3. 워커 배정 = 리서치 워커 × 3 (병렬)
4. 의존성 = 개별 조사는 독립적 → 병렬 가능, 비교 분석은 조사 후 → 순차
5. 매니페스트 생성 → 실행

## 핵심 요약

1. **서브 에이전트 = 병렬 처리 도구**: 메인 컨텍스트 오염 없이 백그라운드에서 독립 실행
2. **별도 컨텍스트 윈도우**: 각 서브 에이전트마다 독립된 컨텍스트 → 토큰 효율적
3. **5대 워커**: 리서치 / 어낼리시스 / 콘텐츠 / 디자인 / 개발
4. **리서치 워커**: 질문 분해 → 다중 소스 탐색 → 모순 처리 → 종합 방법론 내장
5. **Decompose 스킬**: 큰 미션 → 7단계 분해 → 의존성 분석 → 매니페스트 생성
6. **Playwright**: API 없는 서비스도 브라우저 자동화로 처리
7. **Bitwarden**: API 키/비밀번호 통합 관리 (무료)
8. **의지력 절약**: AI에 정리/조사 위임 → 의사결정에 의지력 집중
