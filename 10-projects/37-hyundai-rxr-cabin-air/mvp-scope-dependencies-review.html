검증 1 — RFP F1~F7 역매핑 정합성

RFP에서 정의된 F1~F7 각 기능이 협조 항목과 올바르게 연결되어 있는지, 누락된 의존성은 없는지 점검합니다.

F# 기능 문서 내 의존 항목 검증 결과 판정
F1 웹 UI C1 정확. 웹 UI 자체는 독립 구현 가능하고, 배포 환경(C1)만 의존. B3(보고 대상) 추가 고려 가능하나, UI 기능과 직결은 아님 — 현행 유지 적절 PASS
F2 VoC 전처리/DB A1A2A4 정확. 인제스트 레이어가 데이터 소스 3종(인터뷰·VoC·SNS)에 직접 의존. A5(도메인 확정)도 전처리 토픽 사전에 영향하나, 문서 내 F3에 이미 반영되어 중복 회피 — 적절 PASS
F3 Latent Desire 모델 A1A5 정확. 인터뷰 원문이 핵심 입력이고, 도메인 확정이 토픽 사전 품질 결정. 누락 없음 PASS
F4 정성/정량 교차검증 A2A3 정확. VoC+VCRM/IBD가 교차검증의 양 축. 둘 다 미제공 시 프레임워크만 남는다는 한계 설명도 RFP P3~P4와 일치 PASS
F5 Core Experience 도출 A1A5B2 정확. Latent Desire(F3) 위에 경험 구조화가 쌓이므로 A1·A5 의존 계승. 기획팀 검증(B2)이 Core Experience 품질에 직결 — 역매핑 적절 PASS
F6 User Story 초안 B2B3 보완 권고. User Story는 F5(Core Experience)의 하위 출력이므로, F5 의존 항목(A1, A5)도 간접 의존. 다만 문서에서는 "직접 의존"만 표기하는 원칙이 일관적이므로 현행도 허용 범위 NOTE
F7 Use Case 2개 A1A5B1 정확. Use Case가 전체 파이프라인의 종합 출력이므로 MUST 3종에 의존하는 것이 맞음. 실무 커뮤니케이션(B1) 없이 Use Case 방향이 맞을 수 없음 — 적절한 의존성 PASS
✓
역매핑 정합성: PASS
F1~F7 전체 의존성이 RFP 요구사항(P1~P8)과 정확히 대응. F6에 간접 의존 표기를 추가하면 더 완전하나, 현행도 "직접 의존만 표기"하는 원칙으로 일관성 유지 — 킥오프 제시에 문제 없음.

검증 2 — 3대 핵심 의존성 현실성 체크

현대차 입장에서 "합리적으로 제공 가능한 수준"인지 점검합니다.

  • ✓
    A1 · 고객 인터뷰 102건 (익명화 원문)
    RFP P4에 "고객인터뷰 102건" 명시. 현대차가 이미 보유한 데이터를 익명화하여 전달하는 것이므로 신규 수집 부담 없음. 보안 심사만 통과하면 S0 제공 현실적.
    리스크: 익명화 작업에 내부 법무·보안 승인 소요 → S0 내 수령 불가 시 S1 초로 유예 가능. 이 경우 cabin-air로 S1 전반 커버 후 전환.
  • ✓
    A5 · 분석 대상 도메인/차종 확정
    킥오프 미팅에서 구두 확정 가능한 수준. RFP에서 이미 "공조/공기질" 도메인이 암시되어 있고, 조정현 책임이 즉시 결정 가능한 범위. 제공 장벽 거의 없음.
    리스크: 거의 없음. 확정 지연 시에도 자체 "공조/공기질" 진행으로 대체 가능.
  • ✓
    B1 · 실무 담당자 1인 상시 커뮤니케이션 채널
    외주 프로젝트의 기본 요건. 조정현 책임 또는 팀 내 1인 지정은 현대차 프로젝트 관리 표준 절차. 제공 부담 최소.
    리스크: 담당자 업무 과중 시 응답 지연 가능. 대체안(격주 정기회의)이 적절한 폴백.
✓
현실성 체크: PASS
MUST 3종 모두 현대차가 합리적으로 제공 가능한 수준. A1(인터뷰 102건)은 RFP에 명시된 데이터이고, A5(도메인 확정)는 구두 합의, B1(담당자)은 프로젝트 기본 요건. "무리한 요구"로 인식될 소지 없음.

검증 3 — 등급 분류 과도성 체크

MUST를 남발하면 신뢰가 하락합니다. 12개 항목 중 MUST 비율과 각 등급의 타당성을 점검합니다.

  • ✓
    MUST 비율: 3/12 = 25%
    적정 수준. 일반적으로 MUST가 30% 이하면 과도하지 않다고 판단. "진짜 없으면 안 되는 것"만 MUST로 한정한 절제된 분류.
  • ✓
    MUST 3종의 자격 검증
    A1(인터뷰) — 없으면 현대차 고유 분석 자체 불가 → MUST 자격 충분
    A5(도메인 확정) — 없으면 토픽 사전 구성 불가, 모든 분석이 범용으로 전락 → MUST 자격 충분
    B1(담당자) — 없으면 방향 정렬 불가, 3개월 전체 리스크 → MUST 자격 충분
  • ⚠
    B2(기획팀 검증)가 SHOULD인데, 3대 핵심 의존성에 포함
    Executive Summary에서 B2를 3대 핵심 의존성으로 꼽았지만, 등급은 SHOULD. 의도적 구분인지 검토 필요.
    판단: 이것은 의도적이고 적절한 구분. B2는 "없으면 KPI 측정 불가"이지만, 프로토타입 자체는 구현 가능(기능 무력화 아님). MUST의 정의("핵심 기능 무력화")와 SHOULD의 정의("품질 유의미 저하")에 정확히 부합. Executive Summary는 "중요도"를 강조한 것이고, 등급은 "기능 영향도"를 기준한 것 — 관점 차이를 킥오프에서 설명하면 오히려 신뢰 상승.
  • ✓
    NICE 4종의 적절성
    A4(SNS 범위), B4(RACI), C2(mock 협업), C3(API 보안) — 모두 대체안이 명확하고 MVP 기능에 직접 영향 없음. '27년 연계에만 영향 → NICE 분류 적절.
✓
등급 분류: PASS
MUST 25%(3/12)로 절제된 분류. 각 등급의 정의와 항목이 정확히 대응. B2의 "핵심 의존성이지만 SHOULD" 설계는 의도적 차별화로 오히려 신뢰도를 높이는 구조.
