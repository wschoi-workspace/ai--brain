RFP 기능 × 의존성 역매핑

RFP F1~F7 각 기능이 어떤 협조 항목에 의존하며, 미제공 시 어떤 기능이 깨지는지 역추적합니다.

F# RFP 요구기능 의존 항목 미제공 시 영향
F1 웹 UI (프로토타입) C1 배포 환경 로컬 데모만 가능, 현대차 내부 접근 불가. 기능 자체는 독립 구현 가능
F2 VoC 전처리 / DB A1 인터뷰, A2 VoC, A4 SNS 범위 A1 없으면 = cabin-air만 인제스트. A2·A4 없으면 = 단일소스 시연 only
F3 Latent Desire 모델 A1 인터뷰, A5 도메인 확정 A1 없으면 = 현대차 고유 욕구 도출 불가. A5 없으면 = 범용 분석 → 깊이 부족
F4 정성/정량 교차검증 A2 VoC, A3 VCRM/IBD 둘 다 없으면 = 프레임워크만 제시, 실증 전혀 불가 (핵심 리스크)
F5 Core Experience 도출 A1 인터뷰, A5 도메인, B2 검증 A1 없으면 = cabin-air 기반 generic. B2 없으면 = 현업 적합 미검증
F6 User Story 초안 B2 기획팀 검증, B3 보고 구조 B2 없으면 = AI 생성 초안 수준, N1 95% 측정 불가
F7 Use Case 2개 프로토타입 A1, A5, B1 (전체 MUST) MUST 미충족 시 = "cabin-air 재시연" × 2로 전락. 현대차 고유 가치 0

대응 시나리오

Best Case

MUST + SHOULD 전량 제공
  • 현대차 고유 인사이트 기반 풀스펙 MVP
  • F1~F7 전 기능 실동작 + 교차검증 완료
  • N1 KPI 95% 측정·달성 가능
  • '27년 풀 프로젝트 예산 확보 근거 충분
  • Use Case 2개 = 실 경험 설계 품질

Base Case

MUST만 제공, SHOULD 일부 지연
  • 현대차 인터뷰 기반 분석은 가능
  • 교차검증(F4) = 프레임워크 + mock 시연
  • User Story 정확도는 자체 QA 수준
  • 배포는 외부 클라우드 임시 운용
  • MVP 가치는 입증되나 "완성도" 논란 가능

Worst Case

MUST 미제공 (A1 부재)
  • cabin-air 3,586건으로 파이프라인 데모만
  • 현대차 고유 인사이트 = 0
  • "기술 시연"으로 평가 격하 불가피
  • N1 KPI 측정 기준 부재
  • '27년 예산 확보 논리 약화
