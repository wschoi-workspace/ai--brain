# Wiki Ingest

소스를 분석하여 `30-knowledge/00-wiki/` 토픽 페이지에 지식을 **복리로 축적**합니다.

**복리 = 단순 축적이 아닌 통합.** 새 소스가 들어오면:
- 기존 페이지의 근거가 두꺼워지고
- 핵심 요약이 더 정교해지고
- 페이지 간 교차 참조가 강화되고
- 모순이 감지되어 플래그됨

> "The knowledge is compiled once and then kept current, not re-derived on every query." — Karpathy

## 동작 방식

```
WIKI_PATH = pkm/30-knowledge/00-wiki (git root에서 자동 감지)
```

### Input 확인

- **파일 경로**: Read
- **URL**: WebFetch
- **"이 세션"**: 현재 대화 맥락에서 파악
- **ripple에서 호출**: 세션 작업 맥락에서 파악

### Step 1: 소스 분석 및 핵심 추출

소스에서 추출:
- **주요 토픽** (1~5개): 이 소스가 다루는 핵심 주제
- **핵심 인사이트**: 축적할 가치가 있는 발견/패턴/원칙
- **구체적 사례/인용**: 인사이트를 뒷받침하는 증거
- **새로운 연결**: 기존에 별개로 보이던 개념 간 관계

**축적 가치 판단:**
- 다른 맥락에서 재사용 가능한가?
- 기존 지식에 새 근거/반례를 추가하는가?
- 반복적으로 나타나는 패턴인가?

가치 없으면 조용히 종료.

### Step 2: 기존 토픽 전수 검색

1. `$WIKI_PATH/index.md` 읽어서 전체 토픽 목록 파악
2. 추출한 토픽 키워드로 `$WIKI_PATH/*.md` Grep
3. 하나의 소스가 **여러 토픽에 관련**될 수 있음 — 모두 찾기

### Step 3: 통합 (Integration) — 핵심 단계

**각 관련 토픽 페이지에 대해:**

#### 3A. 근거 추가
- 새 근거/사례를 `## 근거` 섹션에 추가
- source 인용 형식: `[요약] [source: 파일명/URL, YYYY-MM-DD]`

#### 3B. 핵심 요약 갱신
- 새 근거가 기존 `## 핵심` 섹션의 이해를 바꾸거나 정교하게 만드는지 판단
- 바뀌면 → `## 핵심` 섹션 갱신 (기존 내용 대체가 아닌, 더 정확한 표현으로 발전)
- 안 바뀌면 → 건드리지 않음

#### 3C. 모순 감지
- 새 내용이 해당 페이지의 기존 주장과 상충하는지 확인
- 상충하면 → 해당 위치에 `[!contradiction]` 플래그 추가:
  ```
  > [!contradiction] 기존: "X가 효과적" vs 새 근거: "X는 비효과적" [source: 출처, 날짜]
  > → 검증 필요
  ```
- 모순은 삭제하지 않음. 어느 쪽이 맞는지는 사용자가 판단

#### 3D. 교차 참조 강화 (Related + Infobox)
- 이번 ingest로 업데이트된 **모든 토픽 페이지**의 Related 섹션 점검
- 토픽 A와 토픽 B가 같은 소스에서 함께 enriched 되었으면 → 서로의 Related에 없으면 추가
- 기존 Related에 이미 있으면 건드리지 않음
- **Infobox 관련 줄도 동기화**: H1 바로 아래 `> **관련**:` 줄에 새 연결을 반영. 최대 6개, indegree/의미 강도 높은 것 우선. 6개 초과 시 기존 항목과 교체 판단은 사용자에게 문의
- Infobox가 없는 페이지라면 SCHEMA 형식대로 `> **관련**:` + (선택) `> **Facets**:` 2줄을 H1 직하(YAML/인용문구 블록 아래)에 생성

**비대칭 발견 시:** 한쪽 추가했는데 대상이 꽉 차서 역방향 못 넣는 경우, 억지로 밀어 넣지 말고 그대로 둔다. 월 1회 `wiki-lint`가 비대칭을 잡는다.

#### 3D-1. Facets 갱신
- 이번 enrichment로 기존 Facets에 없던 **축(axis)/키워드**가 생겼으면 `> **Facets**:` 줄에 `·` 구분으로 append
- 추가 후 글자 수 확인: `echo -n "Facets 줄 내용" | wc -c`
- 70자 초과 시 가장 오래되거나 약한 키워드 하나를 제거(덤프 금지)
- Facets 줄이 없는(= index 한 줄 설명이 완전한 문장인) 페이지는 그대로 유지. 굳이 Facets 줄 만들지 않음

#### 3E. 적용/Open Questions 갱신
- 새 적용 사례 → `## 적용` 섹션에 추가
- 새 질문 발생 → `## Open Questions` 에 추가
- 기존 Open Question이 이번 소스로 해결되었으면 → 해당 질문에 답변 추가 또는 제거

#### 3F. footer 갱신
- `Last enriched` 날짜 갱신
- Sources 목록에 새 소스 추가

#### 3G. 근거 크기 점검 (분리 알람)

enrichment 후 `## 근거` 섹션의 항목 수를 확인:
```bash
grep -c "^- " $WIKI_PATH/파일명.md
```

- **30개 이하**: 양호
- **30~50개**: 결과 보고에 "근거 N개 — 정리 고려" 표시
- **50개 초과**: 결과 보고 상단에 **분리 제안** 블록 표시 (허브 + 서브 페이지 분리)

자동 분리하지 않음. 사용자 판단 후 `wiki-lint` Tier 2로 처리.

**root 형식 페이지 (구 roots):**
- 기존 섹션 구조를 존중 (핵심 철학, Primary Source 직접 인용, 비즈니스 적용 등)
- 적절한 기존 섹션에 추가. 강제 변환 금지

### Step 4: 새 토픽 생성 (매칭 안 될 때만)

기존 토픽과 매칭이 안 되는 내용이 있을 때만 새 파일 생성.

```markdown
# [토픽명]

> **관련**: [[관련-토픽-1]] · [[관련-토픽-2]] · [[관련-토픽-3]]
> **Facets**: 축1 · 축2 · 축3

## 핵심
[소스에서 추출한 핵심 요약]

## 근거
- [구체적 내용] [source: 소스명, 날짜]

## 적용
(향후 추가)

## Open Questions
(향후 추가)

## Related
[[관련-토픽-1]]
[[관련-토픽-2]]

---
Sources: [소스명]
Last enriched: [오늘 날짜]
```

**Infobox 2줄은 필수**. Facets가 index의 한 줄 설명으로 문장화되면 Facets 줄 생략 가능.

파일명: 케밥케이스 한글 (예: `비개발자-AI-교육.md`)

**새 토픽 생성 후 교차 참조:**
- 새 토픽과 관련된 기존 토픽의 Related에도 새 토픽 링크 추가
- 예: [[뷰티-업종-AX]] 생성 → [[AX-컨설팅-운영]]의 Related에 [[뷰티-업종-AX]] 추가

### Step 5: index.md 갱신

**현재 카테고리 구조 (2026-04-24 기준, 5개):**
- Core Values — 일과 삶의 기반 (가치·태도)
- AI & Education — AI 활용과 비개발자 교육 (AX 실무)
- Claude Code 스킬 설계 — 허브 + 서브 (스킬 구조·패턴·방어선)
- 운영 방법론 — 지식 축적과 일하는 방식 (메타 시스템)
- 비즈니스 전략 — 조직과 시장

- 새 토픽이면 위 카테고리 중 맞는 곳에 추가. 맞는 곳이 없고 유사 토픽이 3개 이상 모이면 새 카테고리 제안(자동 생성 금지, 사용자 확인)
- 기존 카테고리 중 1~2개짜리로 줄어든 경우 다른 카테고리와 통합 제안
- 상단 "읽기 가이드" 섹션의 진입 경로는 새 허브 토픽이 생겼을 때만 사용자에게 갱신 제안
- 페이지 유형 표기: concept (개념/원칙), entity (특정 대상), synthesis (비교/분석)
- **한 줄 설명은 완전한 문장 1개, 70자 이내**. 키워드 콤마 덤프 금지(키워드는 해당 페이지 Infobox Facets로). 기존 문장형 설명을 유지·미세 조정 우선, 강제 재작성 금지
- **메타데이터 갱신**: enriched된 토픽의 sources 수 증가, enriched 날짜 업데이트
- footer의 Total 수와 Last updated 갱신
- **허브 블록 재계산**(indegree 변동 시): 상단 `> **허브 토픽**` 줄의 상위 5개 wikilink를 전체 페이지 Infobox 관련 링크 indegree 기준으로 재정렬

### Step 6: log.md 기록

`$WIKI_PATH/log.md`에 append:

```markdown
## [YYYY-MM-DD] ingest | [소스명]
- source: [소스 경로 또는 URL]
- [created/enriched]: [[토픽명]] ([한 줄 요약])
- cross-refs: [[A]] ↔ [[B]] (새로 연결된 교차 참조)
- contradictions: [[토픽명]] (발견된 모순, 있을 때만)
- 근거 크기 경고: [[토픽명]] N개 (30개 초과 시에만)
```

### Step 7: 결과 보고

```
Wiki Ingest 완료:
- enriched: [[토픽A]] ← [한 줄 요약]
- enriched: [[토픽B]] ← [한 줄 요약]
- created: [[토픽C]] (새 토픽, 있을 때만)
- cross-refs: [[A]] ↔ [[B]] (새 연결)
- contradictions: 없음 (또는 발견 내용)
- 핵심 갱신: [[토픽A]]의 핵심 섹션 정교화 (있을 때만)
- 근거 크기: (30개 초과한 경우에만 표시)
```

## Query 환류

대화 중 가치 있는 분석/비교/발견이 나오면 위키에 환류:

1. 비교 분석, 의사결정 근거, 새로 발견된 연결 등 → synthesis 유형 페이지로 저장
2. 기존 토픽에 새 근거로 추가할 수 있으면 enrichment 우선
3. log.md에 `## [YYYY-MM-DD] query | [질문 요약]`으로 기록

대화에서 나온 인사이트가 사라지지 않고 위키에 남는 것이 핵심.

## Supervised vs Batch 모드

- **Supervised (기본)**: 한 소스씩 처리. 핵심 포인트를 사용자와 논의 후 반영. ripple 5단계, 수동 호출 시 이 모드
- **Batch**: 여러 소스 일괄 처리. 감독 최소화. transcript-organizer 연동 시 이 모드

## 핵심 원칙

- **통합이 축적보다 우선**: 근거 추가 → 핵심 갱신 → 교차 참조 → 모순 감지. 이 순서가 복리를 만듦
- **기존 페이지 확장 우선**: 비슷한 토픽이 있으면 새 파일 만들지 않음
- **source 인용 필수**: 모든 추가에 출처와 날짜. persistent error 방지의 핵심
- **모순은 삭제하지 않음**: 플래그만. 판단은 사용자
- **구조 강제 변환 금지**: root 등 기존 페이지의 고유 구조 존중
- **Query 환류**: 좋은 답변/분석은 다시 위키에 저장. 대화에서 사라지지 않게
- **과잉 추가 금지**: 양보다 질. 하지만 관련된 토픽은 빠짐없이 업데이트
- **정기 점검은 wiki-lint**: 비대칭·오래됨·모순은 lint가 주기적으로 잡음. ingest가 매번 다 검증하지 않음
