# Transcript Organizer

긴 녹음 텍스트 파일을 자동으로 분석하고 구조화된 문서로 정리하는 스킬입니다.

## 📌 언제 사용하나요?

- 강의 녹음 파일을 정리할 때
- 미팅 녹음을 회의록으로 만들 때
- 인터뷰 내용을 구조화할 때
- 긴 텍스트 파일(3시간 분량 등)을 디테일하게 문서화할 때

## 🔧 작동 방식

### Phase 1: 파일 확인 및 인코딩 처리

1. **파일 존재 확인**
   ```bash
   ls -la [파일경로]
   ```

2. **인코딩 감지**
   ```bash
   file -I [파일경로]
   ```

3. **UTF-16 → UTF-8 자동 변환 (필요시)**
   - UTF-16 계열 감지 시 자동 변환
   - 변환된 파일: `/tmp/[파일명]-utf8.txt`
   - `iconv -f UTF-16 -t UTF-8` 사용

4. **파일 크기 확인**
   - 110,000 토큰 이상이면 청크 처리 안내

### Phase 2: 컨텍스트 수집

사용자에게 다음 4가지 질문:

1. **유형**: 강의 / 미팅 / 인터뷰?
2. **날짜**: YYYY-MM-DD?
3. **주제**: 간단한 설명 (한 줄)?
4. **저장 위치**:
   - 특정 프로젝트 경로 입력
   - "모름" 또는 "inbox" → 기본 inbox 폴더

**경로 자동 유추 방식**:

1. **키워드 추출**: 파일명/내용에서 키워드 추출
2. **폴더 검색**: Glob으로 현재 워크스페이스에서 관련 폴더 검색
   - 예: "hfk" → `**/hfk*/**`, `**/*hfk*/`
   - 예: "meeting" → `**/meeting*/**`, `**/*meetings*/`
3. **매칭 제안**: 유사 폴더 발견 시 사용자에게 제안
4. **사용자 입력**: 매칭 실패 시 직접 경로 입력 요청

**별도 설정 파일 불필요** - 스킬이 자체적으로 폴더 구조 탐색

### Phase 3: 파일 읽기 및 구조 파악

1. **파일 읽기**
   - 변환된 UTF-8 파일 **전체** 읽기
   - **중요**: 처음 500줄은 구조 파악용 샘플링일 뿐, **반드시 전체 내용을 끝까지 읽어야 함**
   - 대용량 파일은 여러 번에 걸쳐 읽기 (예: 700줄씩 청크로 읽기)

2. **구조 분석** (샘플링 단계)
   - 화자 구분 패턴: `발화자 1 (00:00)`
   - 타임스탬프 패턴: `(00:00)`, `[00:00]`
   - 단순 텍스트 (구분 없음)

3. **전체 내용 분석**
   - **전체 파일을 끝까지 읽고 분석**
   - 키워드 추출 (자주 등장하는 단어 5-10개)
   - 주제 카테고리 식별

### Phase 4: STT 오인식 교정

> 핵심 원칙: 고정된 오류 사전이 아닌, **맥락 기반 분석**으로 교정한다.
> Phase 2에서 수집한 주제/도메인 + Phase 3에서 읽은 전체 내용을 종합하여
> "이 맥락에서 이 단어는 말이 안 된다"를 판단하는 방식.

1. **STT 여부 판단**
   - Phase 3에서 읽은 내용을 기반으로 STT 변환 파일인지 판단
   - 판단 기준: 동일 단어의 반복적 오인식, 문맥에 맞지 않는 단어 산재, 한국어 STT 특유 패턴 (조사 탈락, 띄어쓰기 오류, 음절 대체)
   - STT가 아닌 경우 (사람이 직접 작성한 회의록 등) → Phase 4 스킵

2. **맥락 파악 후 오인식 탐지**
   - Phase 2의 주제/도메인 정보를 기반으로 해당 분야 맥락을 설정
   - 전체 텍스트를 읽으며 다음 기준으로 오인식 후보를 탐지:
     - **문맥 부적합**: 해당 주제에서 등장할 이유가 없는 단어 (예: 브랜딩 세미나에서 "국토교수실" → "북토크")
     - **음운 유사 + 의미 불일치**: 발음은 비슷하지만 맥락상 다른 단어가 맞는 경우 (예: "면역" → "명료함")
     - **표기 불안정**: 같은 대상이 여러 형태로 등장 (예: "오창민/오탁미/오탁민" → 하나로 통일 필요)
     - **영어 음차 오류**: 영어 단어가 한글로 잘못 변환 (예: "롤스코즈" → "Root Cause")
     - **고유명사 왜곡**: 인명, 브랜드명, 책 제목 등이 일반 단어로 변환
   - 확신도 구분: 확실한 교정 vs 추정 교정 ("(추정)" 표기)

3. **교정 테이블 작성 및 자동 적용**
   - 발견된 오인식 후보를 테이블로 정리
   - 사용자 확인 없이 자동 적용하되, 최종 문서의 '음성인식 교정 노트' 섹션에 모든 교정 내역을 투명하게 포함한다
   - 확신도 구분 유지: 추정 교정은 "(추정)" 표기

   ```markdown
   | 오인식 | 교정 | 비고 |
   |--------|------|------|
   | 국토교수실/국토클린 | **북토크** | 문맥 부적합 - 행사 형식 |
   | 오창민/오탁미 | **오탁민** | 표기 불안정 - 저자 이름 |
   | 루트 코즈/롤스코즈 | **Root Cause** | 영어 음차 오류 |
   | 피부/피어 | **VP/Peer** (추정) | 영어 직책 - 맥락상 추정 |
   ```

4. **교정 적용**
   - 교정을 Phase 5 분석 시 자동 적용 (원본 텍스트 자체는 변경하지 않고, 분석/정리 과정에서 교정된 표현 사용)
   - 최종 문서에 "음성인식 오류 교정 노트" 섹션 필수 포함 (투명성 + 감사 추적)

### Phase 5: 내용 분석 및 정리

**스키마 기반 구조화** (선택적):
1. 저장 경로에서 `_schema*.md` 파일 검색
2. 스키마 발견 시:
   - 스키마의 섹션 구조를 우선 적용
   - 분량 가이드에 따라 섹션별 분량 조절
   - 반복 패턴이 있으면 해당 패턴으로 반복 생성
   - 스키마에 없는 섹션은 기본 템플릿에서 보충
3. 스키마 미발견 시: 기존 템플릿 그대로 적용 (하위 호환)

**유형별 템플릿 적용**:

1. **강의 (lecture-template.md)**
   - 개요 (날짜, 강사, 주제, 시간, 특이사항)
   - 핵심 내용 (시간순 또는 주제별)
   - 핵심 요약
   - 다음 작업

2. **미팅 (meeting-template.md)**
   - 개요 (날짜, 참석자, 안건, 시간)
   - 안건별 논의 내용
   - 결정 사항
   - Action Items

3. **인터뷰 (interview-template.md)**
   - 개요 (날짜, 인터뷰 대상, 주제)
   - 주요 질문 및 답변
   - 핵심 인사이트
   - 후속 작업

**정리 원칙**:
- **전체 내용 완전 분석**: 처음부터 끝까지 모든 내용을 읽고 정리
- **대용량 파일도 끝까지 읽기**: 샘플링은 구조 파악용일 뿐, 실제 정리는 전체 내용 기반
- 원본 내용 최대한 보존 (절대 임의로 생략하지 않음)

**분량 기준** (필수):
| 원본 길이 | 최소 결과물 분량 |
|----------|----------------|
| 1시간 미만 | 300줄 이상 |
| 1-2시간 | 500줄 이상 |
| 3시간 이상 | 700줄 이상 |

**인용 원칙** (필수):
- 직접 인용 **최소 10회 이상**
- `> "인용문"` 형식 사용
- 핵심 발언, 인사이트, 사례 모두 인용

**참석자/발언자 사례 기록**:
```markdown
**[이름]님**
- [사례 1]: 구체적 내용 기술
- "[직접 인용]"
- [사례 2]: 상황과 맥락 포함
```

**구조화 원칙**:
- 부(Part) 단위 구분 (주제별 또는 시간별)
- 타임스탬프 포함 (가능한 경우)
- 표(Table) 적극 활용 (비교, 정리)
- Q&A 완전 수록 (질문 + 답변 + 후속 논의)

### Phase 6: 문서 생성 및 저장

1. **파일명 생성**
   ```
   YYYY-MM-DD_[주제].md
   ```

2. **저장 경로 결정**
   - 사용자 입력 경로 우선
   - 파일명 유추 경로
   - 기본값: `pkm/00-inbox/`

3. **문서 저장**
   ```bash
   # 경로 생성 (필요시)
   mkdir -p [저장경로]

   # 파일 저장
   Write tool 사용
   ```

4. **결과 보고**
   ```
   ✅ 정리 완료!
   📄 파일: [저장경로]/YYYY-MM-DD_[주제].md
   📊 길이: [줄 수]줄
   🏷️ 유형: [강의/미팅/인터뷰]
   ```

## 🎯 사용 예시

### 예시 1: 강의 녹음 정리
```
사용자: "/path/to/downloads/lecture-week3.txt 이거 강의 정리해줘"

Claude:
1. 파일 확인 → UTF-16 감지 → UTF-8 변환
2. 질문:
   - 유형: 강의
   - 날짜: 2025-10-31
   - 주제: Claude Code 설치 실습
   - 저장 위치: (키워드 "lecture" 감지) → Glob으로 관련 폴더 검색 → 사용자에게 제안
3. 내용 분석 → 강의 템플릿 적용
4. 문서 생성: 2025-10-31_Claude-Code-설치-실습.md (최소 300줄, 인용 10회+)
```

### 예시 2: 미팅록 정리
```
사용자: "프로젝트 미팅 녹음 정리해줘"

Claude:
1. 파일 경로 요청
2. 인코딩 확인
3. 질문 4가지
4. 미팅 템플릿 적용
5. 저장 위치: Glob으로 `**/meeting*/**` 검색 → 매칭되는 폴더 제안 → 없으면 사용자 입력 요청
```

## ⚠️ 주의사항

### 1. 인코딩 문제
- UTF-16 파일은 자동 변환
- 변환 실패 시 사용자에게 알림
- EUC-KR 등 다른 인코딩은 수동 확인 필요

### 2. 대용량 파일
- 110,000 토큰 이상: 여러 번 Read로 청크 처리
- **중요**: 샘플링은 구조 파악용일 뿐, **반드시 전체 파일을 끝까지 읽어야 함**
- 700줄씩 나누어 읽되, 모든 내용을 빠짐없이 분석

### 3. 저장 위치
- 프로젝트 경로 유추 실패 시 inbox 사용
- 사용자가 나중에 수동 정리 가능

### 4. 템플릿 적용
- 화자/타임스탬프 없어도 정리 가능
- 유형별 템플릿은 가이드일 뿐, 내용에 맞게 유연하게 조정

## 📁 지원 파일

### utils/

## 🔄 워크플로우 요약

```
1. 파일 확인 → 인코딩 변환
2. 사용자 질문 (4개)

4. STT 오인식 교정 (해당 시, 자동 적용)
5. 스키마 참조 (있으면) → 템플릿 적용 및 정리
6. 문서 저장 및 보고
```
