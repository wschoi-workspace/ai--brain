# /gov-proposal — 정부사업 제안서 풀 파이프라인 오케스트레이터

> R가이드(7대 설득 원칙 + B2C 감성 설득) 기반으로 RFP→제출본까지 7단계를 순차 실행.
> 각 단계마다 중간 확인을 거치며, 정보가 부족하면 질문으로 보완.

## Core Philosophy

- **오케스트레이터**: 기존 스킬을 체인으로 연결하되, 각 단계 사이에 판단과 질문을 삽입
- **부족하면 묻는다**: 배경 정보(지난행사, 경쟁사, 사회적 문제)가 약하면 진행하지 않고 질문
- **R가이드 프레임**: 7대 원칙(정책정합성, HIRE, 3중Problem, 리스크-자산, 성과체계, Promised Land, Sparkline)을 모든 단계에 관통
- **자동이되 맹목적이지 않다**: 각 Phase 완료 후 사용자 확인 → 다음 Phase 진행

## 7-Phase Pipeline

```
Phase 1: RFP 분석 (/rfp-analysis 호출)
    ↓
Phase 2: 배경 리서치 ★질문 게이트
    ↓
Phase 3: 전략 브리프 (R-Step 2)
    ↓
Phase 4: 페이지네이션 (R-Step 3)
    ↓
Phase 5: 내러티브 아크 (/proposal-architect 호출)
    ↓
Phase 6: HTML 제작 (디자인 가이드 적용 + 마케팅 강화)
    ↓
Phase 7: QA (R가이드 체크리스트)
```

## Phase 2 — 배경 리서치 ★질문 게이트

> **이 Phase가 제안서의 설득력을 결정한다.** 정보가 부족하면 절대 넘어가지 않는다.

### 3대 리서치 축

#### 축 1: 지난 행사 분석
발주처가 이전에 수행한 동일/유사 사업의 성과와 한계를 분석.

**자동 수집 시도:**
1. RFP에서 전년도 사업명/수행사 추출
2. WebSearch로 전년도 행사 후기, 보도자료, 블로그, SNS 검색
3. 나라장터에서 이전 낙찰 업체/금액 검색

**수집할 데이터:**
| 항목 | 목적 |
|------|------|
| 전년도 사업명·수행사·금액 | 경쟁 구도 파악 |
| 방문객 수·매출·미디어 노출 | 정량 성과 기준선 |
| SNS 자발적 후기 유무 | 바이럴 성공/실패 판단 |
| 언론 보도 톤 | 긍정/부정/무관심 |
| 주최자/참여자 후기 | 구조적 한계 발견 |

**★ 질문 게이트 — 다음 중 하나라도 부족하면 질문:**
```
[필수] 전년도 행사가 있었나요? 있다면:
  - 행사명, 장소, 기간, 수행사
  - 방문객 수, 매출 등 성과 숫자
  - "잘됐다/아쉬웠다"고 느낀 점
  - 관련 기사나 블로그 링크

[선택] 발주 담당자와 미팅한 적 있나요? 있다면:
  - 담당자가 강조한 포인트
  - 담당자가 불만/아쉬움을 표현한 부분
```

#### 축 2: 사회적 문제/시장 맥락 분석
해당 산업/분야의 구조적 문제를 리서치하여 3중 Problem의 Philosophical 층위에 활용.

**자동 수집:**
1. WebSearch로 해당 분야 최신 보고서·통계 검색
2. 정부 정책 방향, 관련 법규, 예산 추이 확인
3. 실패 패턴 분석 (유사 사업의 구조적 한계)

**수집할 데이터:**
| 항목 | 3중 Problem 매핑 |
|------|-----------------|
| 시장 규모·트렌드 숫자 | External — "시장은 이렇게 변하고 있다" |
| 유사 사업 실패 통계 | External — "70~80%가 실패한다" |
| 정책 목표 vs 현실 괴리 | Internal — "담당자의 진짜 고민" |
| 세금·공공가치 이슈 | Philosophical — "실패 시 훼손되는 가치" |

**★ 질문 게이트 — 다음이 부족하면 질문:**
```
[필수] 이 사업이 풀려는 사회적 문제가 무엇인가요?
  - "왜 정부가 이 사업에 예산을 배정했는가"의 배경
  - 관련 뉴스, 보고서, 통계가 있다면 공유해주세요

[선택] 이 분야의 "고질적 실패 패턴"이 있나요?
  - 예: 지자체 팝업의 "진열→동원→보도자료→성공보고" 반복
```

#### 축 3: 경쟁사 분석
예상 경쟁사의 강점/약점을 파악하여 차별화 포인트 설계.

**자동 수집:**
1. 나라장터에서 과거 유사사업 낙찰 업체 검색
2. WebSearch로 경쟁사 포트폴리오, 실적 확인
3. RFP 자격요건에서 참여 가능 업체 유형 추론

**수집할 데이터:**
| 항목 | 용도 |
|------|------|
| 예상 경쟁사 리스트 | 포지셔닝 맵 |
| 각 사의 강점/약점 | 리스크-자산 차별화 |
| 과거 낙찰 업체·금액 | 투찰가 참고 |
| 경쟁사가 제안할 방향 예측 | "그들이 못 하는 것"을 우리 무기로 |

**★ 질문 게이트:**
```
[필수] 예상 경쟁사를 알고 있나요?
  - 직접 경쟁할 것 같은 업체명
  - 그들의 강점이라고 생각하는 것

[선택] 과거에 이 발주처와 일한 적 있나요?
  - 있다면 어떤 관계인지 (신규/기존/재입찰)
```

### 산출물
`{프로젝트명}-background-research.md` — 3축 리서치 결과 + 팩트 무기 리스트

### 완료 조건
- 팩트 무기 최소 5개 (숫자가 있는 구체적 데이터)
- 3중 Problem 3개 층위 모두 재료 확보
- 경쟁 차별화 포인트 최소 2개

### → 사용자 확인
"배경 리서치 완료. 팩트 무기 [N]개, 3중 Problem 재료 확보. 전략 브리프로 넘어갈까요?"

## Phase 3 — 전략 브리프 (R-Step 2)

### R가이드 프레임 적용
`gov-proposal-guide-R.html`의 Section 02 (감정 설계) 기준으로:

1. **SCQA 정의** — Phase 1(RFP) + Phase 2(리서치)에서 추출
   - S: 시장 현실 (팩트)
   - C: 구조적 문제 (3중 Problem)
   - Q: 어떻게 해결하는가?
   - A: 컨셉 한 줄

2. **HIRE 확정** — Phase 1에서 1차 정의한 것을 Phase 2 데이터로 구체화
   - F: 기능적 성과 (숫자)
   - E: 정서적 전환 (이미지)
   - S: 사회적 증명 (정책)

3. **3중 Problem** — Phase 2의 3축 리서치에서 구성
   - Pain 1~3 각각에 External / Internal / Philosophical

4. **Promised Land** — "솔루션 없이 비전 먼저"
   - 한 문장 + 풀 버전

5. **Wow Moment** — 정확히 1개 선정
   - 정의 + 배치(50~60%) + 예상 밖 요소

6. **Sparkline** — 감정 곡선 3피크 위치 확정
   - Peak 1 (Pain) / Peak 2 (Wow) / Peak 3 (Bliss)

7. **코어리뷰** — GO/NO-GO + 자체평가 점수표

`{프로젝트명}-R-step2-strategic-brief.md`

### → 사용자 확인
"전략 브리프 완성. SCQA·HIRE·3중Problem·PL·Wow 확정. 검토해주세요."

## Phase 5 — 내러티브 아크

### 실행
`/proposal-architect` Phase 1 (구조화 모드)를 호출하되, Phase 3의 전략 브리프를 입력으로 전달.

- SCQA, HIRE, 3중 Problem, PL, Wow → Phase 3에서 완성
- 페이지네이션 → Phase 4에서 완성

**이 Phase에서 하는 것:**
- 10단계 하이브리드 공식과 페이지네이션을 매핑
- 각 슬라이드의 후킹 메시지 1차 작성
- What Is ↔ Could Be 교차 포인트 확정
- Success ↔ Failure 암시 문구 확정

### 마케팅 후킹 강화

- 지난 행사 실패 데이터 → 3중 Problem 헤드라인에 숫자로
- 성공 사례 데이터 → Promised Land와 Could Be에
- 경쟁사 차별점 → Why Us와 차별화에

**강화 체크리스트:**
- [ ] 표지 WHY에 도발적 후킹이 있는가?
- [ ] s04(Problem)에 전년도 실패 숫자가 있는가?
- [ ] s05(PL)에 성공 사례 숫자가 있는가?
- [ ] 거점 슬라이드에 What Is↔Could Be 대비가 있는가?
- [ ] 마케팅 슬라이드에 "기존 공식을 뒤집는" 선언이 있는가?
- [ ] 클로징에 Success↔Failure 암시가 있는가?

## Phase 6 — HTML 제작

### 디자인 시스템 결정
1. `00-system/04-design/project-rent-design-guide.md` 읽기
2. 사업 성격에 맞는 디자인 톤 결정:
   - 기본: R스타일가이드 (청보라 #6C5CE7, Pretendard, 다크 배경)
   - 무드보드가 제안서 안에 있으면 해당 톤 우선 적용

### 캔버스
- **1920 × 1080px** (16:9 fullHD) — 발표 + 인쇄 겸용
- 안전영역: 104px 좌우 패딩
- 인터랙티브: ← → 네비, 편집 모드, HTML 저장, 스케일

### 컴포넌트
레퍼런스 제안서의 검증된 컴포넌트 세트 사용:
- `strip-top` / `strip-bot` (상하단 정보바)
- `slide--cover` (표지/클로징)
- `slide--divider` (챕터 간지, 대형 숫자)
- `slide--dark` (감정 피크 슬라이드)
- `card`, `stat`, `blist` (콘텐츠 카드)
- `course-grid` (매트릭스 표)
- `scene` (공간 연출 + 이미지)
- `ctable` (비교표)
- `flow`, `fstep` (매니페스토/플로우)
- `cycle`, `cyc` (번호 사이클)
- `chips` (키워드 칩)
- `gantt`, `budget` (간트차트, 예산)

### 감정 피크 디자인 규칙
- **Peak 1** (Problem): `slide--dark` — cobalt/navy 배경, 위기감 타이포, 빨간 강조
- **Peak 2** (Wow): `slide--dark` — 최대 타이포(76px+), 가장 극적인 비주얼
- **Peak 3** (Closing): `slide--cover` — cobalt 배경 + 포인트 악센트, 대형 카피

### 제작 프로세스
1. Phase 4 페이지네이션의 전체 슬라이드를 하나의 HTML 파일로 생성
2. 모든 슬라이드에 실제 콘텐츠 (플레이스홀더 텍스트 금지)
3. 이미지 영역은 `.ph` 플레이스홀더 (추후 교체)
4. 인터랙티브 JS 포함 (네비게이션, 편집, 저장)

## Phase 7 — QA

### R가이드 체크리스트 (전 슬라이드)
| 체크 | 항목 |
|------|------|
| [ ] | 모든 슬라이드에 제목이 있는가 |
| [ ] | 모든 슬라이드에 핵심 메시지 1문장이 있는가 |
| [ ] | 모든 슬라이드에 수치/팩트 최소 1개 |
| [ ] | 모든 슬라이드에 비주얼 앵커 (표/다이어그램/이미지) |
| [ ] | HIRE 연결 — 03 사업계획의 각 페이지가 F/E/S 중 어디에 기여하는가 |
| [ ] | Pain↔Solution 대응 — 01 배경의 3중 Problem과의 대응 |
| [ ] | 표지-클로징 일관성 — 같은 메시지가 양쪽 끝에서 감싸는가 |
| [ ] | **감정 곡선 3피크** — Pain(배경) + Wow(계획) + Bliss(클로징) 살아있는가 |

### 발표 설계 (발표가 있는 경우)

파이프라인 전체에서 "정보 부족 시 묻는" 포인트:

| Phase | 질문 게이트 | 없으면 |

| **2-축1** | 전년도 행사 정보 | "전년도 행사가 있었나요?" → 없으면 WebSearch로 대체 시도 |
| **2-축2** | 사회적 문제/시장 데이터 | "이 사업이 풀려는 사회적 문제가 무엇인가요?" → 없으면 WebSearch로 보완 |
| **2-축3** | 경쟁사 정보 | "예상 경쟁사를 알고 있나요?" → 없으면 유형별 추론 |
| **3** | 컨셉 방향성 | "컨셉에 대한 아이디어나 방향이 있나요?" → 없으면 3안 제시 |
| **3** | 차별화 자산 | "우리만의 무기(자산, IP, 트랙레코드)가 무엇인가요?" |
| **6** | 디자인 톤 | "디자인 톤 선호가 있나요?" → 없으면 R스타일가이드 적용 |

## 실행 흐름 예시

```
사용자: "/gov-proposal [RFP 파일 경로]"

Phase 1: /rfp-analysis 호출 → 분석 완료
→ "RFP 분석 완료. 다음으로?"

Phase 2: 배경 리서치 시작
→ 질문: "전년도 행사가 있었나요?"
→ 사용자 응답 + WebSearch 보완
→ 질문: "예상 경쟁사를 알고 있나요?"
→ 사용자 응답
→ "팩트 무기 7개, 3중 Problem 재료 확보. 전략 브리프로?"

Phase 3: 전략 브리프 생성
→ SCQA / HIRE / 3중Problem / PL / Wow / Sparkline
→ "전략 브리프 완성. 검토?"

Phase 4: 페이지네이션
→ 30P 기준 슬라이드별 명세 + Sparkline
→ "페이지네이션 완성. 3피크 확인?"

Phase 5: 내러티브 아크 + 마케팅 후킹
→ 10단계 매핑 + 후킹 메시지
→ "내러티브 완성. 핵심 메시지 확인?"

Phase 6: HTML 제작
→ 1920×1080 전체 슬라이드 생성
→ "브라우저에서 확인하세요."

Phase 7: QA + 슬라이드별 검토
→ s01부터 순차 검토 → 수정 반영
→ "전체 검토 완료. 프로그레스 저장."
```
