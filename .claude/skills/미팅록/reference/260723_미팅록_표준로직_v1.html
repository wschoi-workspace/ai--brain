미팅록 표준 로직

회의 입력 1건(전사 텍스트·거친 메모·docx)을 받아 표준 미팅록 3종(MD 원본 · 공유용 HTML 보고서 · 액션 아이템)을 만들어내는 판단·구조화·산출 로직의 정의서. 특정 프로젝트·폴더·도구에 묶이지 않은 미팅록 전용 기준으로, 누가 어떤 회의를 정리해도 같은 구조와 톤이 나오게 한다.

목적
회의록 수기 정리 시간을 0으로 — 입력만 주면 표준 산출물이 나온다
적용 범위
운영 · 콘셉트 · 정기 보고 · 협상 · 킥오프 등 모든 회의 유형 (한국어)
산출물
① 미팅록 MD  ② 공유용 HTML(A4 풀 보고서)  ③ 액션 아이템 MD — 총 3종
CORE 3원칙 — 이 셋이 미팅록의 품질을 결정한다
1제안서가 아니라 풀 보고서 — 사실만 나열하지 않는다. 사실 + 비즈니스 컨텍스트 + 인사이트 + 미해결까지 담는다.
2추론은 전부 [검토] 표기 — 입력에 없는 정보는 단정 금지. 보강한 항목은 반드시 [검토] 마커를 단다.
3내부·외부 콘텐츠 동일 — internal/external은 톤·헤더 스위치일 뿐, 정보량과 내용은 절대 달라지지 않는다.
00전체 파이프라인 — 6단계
Input → Output
1
입력 수집
전사 텍스트 · 거친 메모 · docx를 받는다. 녹음 파일은 받지 않는다(전사 텍스트로 변환 후 입력).
2
메타데이터 파싱
날짜 · 프로젝트 · 미팅명(회차) · 참석자 · 유형 · 모드를 입력에서 추출한다. 못 찾으면 정해진 fallback.
3
미팅 유형 판별
입력에 명시적 헤딩이 있으면 그대로 사용. 없으면 키워드로 유형을 추정해 본문 카테고리를 정한다.
4
5블록 구조화
개요 → Summary → 안건별 본문 → 의사결정 → To-Do의 고정 골격에 내용을 배치한다.
5
추론 보강
논의됐지만 명시 안 된 비즈니스 정보를 6개 카테고리에서 도출하고 전부 [검토] 표기한다.
6
산출 · 저장
MD + HTML(A4 doc) + actions.md 3종을 생성하고 해당 프로젝트의 미팅록 폴더에 저장한다.

04표준 5블록 골격
STEP 4 · 고정 구조
[1]
프로젝트 개요 / 배경
있을 때만. 운영 미팅=규모·범위·제약·일정, 콘셉트 미팅=배경·맥락·시장 인식, 일반 회의=생략 가능.
[2]
Summary
한 문단 요약 + 키:값 테이블. 핵심 키워드는 굵게. 이 블록만 읽어도 회의가 재구성되어야 한다.
[3]
본문 — 안건별 정리
03에서 정한 카테고리로 배치. 표·번호 매김·2컬럼을 사용하고, 논의의 맥락과 근거를 함께 적는다.
[4]
의사결정사항 ✓
이 회의에서 확정된 것만. 체크리스트 형식 — 결정과 논의를 섞지 않는다.
[5]
요청사항 / To-Do
양측 R&R 분리(우리 측 / 상대 측) + 담당 + 기한. 기한 미정이면 "미정"으로 명시한다.
05비즈니스 컨텍스트 추론 보강
전 항목 [검토] 표기

미팅에서 논의됐지만 입력 텍스트에 명시되지 않은 비즈니스 정보를 아래 6개 카테고리에서 도출해 미팅록에 포함한다. v1의 실수였던 "사실만 깔끔하게 나열"을 막는 단계다.

  • A. 매출·객단가 — 목표 수치의 달성 메커니즘 (회전율, 구성 등)
  • B. 벤치마크 — 직전 임차인·인접·동급 매장 비교
  • C. 수수료·비용 — 별도 항목까지 합산한 실질 부담
  • D. 일정 — 개인·외부 일정과의 충돌 (휴가·승인·행사)
  • E. 자금·계약 — 권한 구조 (감리·결정 권한이 어디에 있나)
  • F. 레퍼런스 — 유사 사례의 성공/실패 패턴
추론으로 보강한 항목은 인라인 [검토] 마커 또는 검토 박스로 표시하고, 문서 하단에 "[검토] 표시 안내"를 자동 삽입한다. 외부 공유 전 [검토] 항목을 일괄 점검한다.
보강 금지선 — 법적·재무적 수치 단정, 인물·기업에 대한 부정 평가, 입력에 없는 사실의 단정 서술. 이 셋은 어떤 경우에도 [검토] 없이 쓰지 않는다.

{YYMMDD}_{미팅명}_actions.md
③ 액션 아이템
체크박스 목록만 분리 — 우리 측 / 상대 측 / 일정 마커. 담당·기한 필수(미정도 명시).
{프로젝트 폴더}/ └─ 00_미팅록/ └─ {YYMMDD}_{미팅명}/ ├─ {YYMMDD}_{미팅명}.md ← ① 원본 (md 요청 시 이 파일) ├─ {YYMMDD}_{미팅명}.html ← ② 공유용 보고서 ├─ {YYMMDD}_{미팅명}_actions.md ← ③ 액션 └─ _source/ ← 원본 입력 보관
  • 저장 위치 — 미팅록은 해당 프로젝트의 00_미팅록/ 폴더에 둔다. 중앙 아카이브를 별도로 두지 않는다. 프로젝트가 없으면 미배정 폴더에 임시 보관 후 프로젝트 생성 시 이동.
  • 버전 — 재생성 시 덮어쓰지 않고 _v2, _v3 suffix. 같은 미팅의 internal/external 재생성은 같은 폴더 안에 _internal / _external로 분리.
  • 파일명 — 날짜 6자리(YYMMDD) + 짧은 미팅 식별자(1차·킥오프·실사·주간보고). 공백은 _, 슬래시·콜론 제거.
입력 1건 → 파싱 → 유형 판별 → 5블록 → [검토] 보강 → 3종 산출. 이 로직만 지키면 누가 정리해도 같은 미팅록이 나온다.
