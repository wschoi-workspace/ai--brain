# /미팅록 — 미팅록 표준 로직 v1 (FILAMENT MEETING NOTES STANDARD)

회의 입력 1건 → 판단·구조화·산출 → 표준 미팅록 3종.
누가 어떤 회의를 정리해도 같은 구조와 톤이 나오게 하는 것이 목적.
로직 정의서 원본: `reference/260723_미팅록_표준로직_v1.html`

## CORE 3원칙 (절대 준수)

1. **제안서가 아니라 풀 보고서** — 사실 + 비즈니스 컨텍스트 + 인사이트 + 미해결까지 담는다.
2. **추론은 전부 [검토] 표기** — 입력에 없는 정보는 단정 금지. 보강 항목은 반드시 [검토] 마커.
3. **내부·외부 콘텐츠 동일** — internal/external은 톤·헤더 스위치일 뿐, 정보량 차이 없음.

## 파이프라인

### STEP 1. 입력 수집
- 전사 텍스트 / 거친 메모 / docx → 그대로 사용
- 녹음 파일 → **자동 전사하지 않는다** — 전사 텍스트로 변환해 다시 입력 요청
- 100자 미만이면 내용 누락 가능성이 크므로 진행 전에 1회 확인

### STEP 2. 메타데이터 파싱
날짜·프로젝트명·미팅명·참석자·유형·모드(internal/external) 추출.
프로젝트 불명 시 1회 확인 요청. Fallback: 날짜→오늘, 유형→일반, 모드→internal.

### STEP 3. 유형 판별
→ `rules/type-detection.md` 참조. 추정 결과 한 줄 확인 후 진행.

### STEP 4. 본문 — 5블록 구조

| 블록 | 내용 |
|---|---|
| [1] 프로젝트 개요 / 배경 | **있을 때만.** 규모·범위·맥락·시장 인식 등 |
| [2] Summary | 한 문단 요약 + 키:값 테이블. **이 블록만 읽어도 회의가 재구성**되어야 한다 |
| [3] 본문 — 안건별 정리 | STEP 3 카테고리로 배치. 표·번호 매김·2컬럼 사용, 논의 맥락과 근거 함께 기록 |
| [4] 의사결정사항 | 이 회의에서 **확정된 것만**. 체크리스트 형식 — 결정과 논의를 섞지 않는다 |
| [5] 요청사항 / To-Do | 양측 R&R 분리(우리 측 / 상대 측) + 담당 + 기한. 미정이면 "미정" 명시 |

### STEP 5. 추론 보강
→ `rules/augmentation.md` 참조. 유형별 6카테고리 적용, 전 항목 [검토] 표기, 금지선 3개 준수.

### STEP 6. 산출 · 저장

| 파일 | 내용 |
|---|---|
| `{YYMMDD}_{미팅명}.md` | 미팅록 원본 — 5블록 골격 |
| `{YYMMDD}_{미팅명}.html` | 공유용 A4 보고서 (`templates/meeting-report-template.html`) |
| `{YYMMDD}_{미팅명}_actions.md` | 액션 아이템 — R&R 분리(우리 측/상대 측) |

저장: `{프로젝트 폴더}/00_미팅록/{YYMMDD}_{미팅명}/` + `_source/` 원본 보관.
프로젝트 미특정 시 `50-resources/미팅록-미배정/`에 임시 보관.
재생성 시 `_v2, _v3` suffix. internal/external은 같은 폴더 안에 분리.
