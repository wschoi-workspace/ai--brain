# RFP Analysis: 프로젝트 제안 전략 분석기

> 단순 요약이 아니라, **"왜 이 프로젝트가 생겼는가" → "진짜 문제는 무엇인가" → "어떻게 이길 것인가"**를 구조적으로 도출한다.

## Core Philosophy

- **문서 기반 우선**: 제공된 자료에서 최대한 추출하고, 웹 리서치는 보완용
- **추론과 사실 구분**: 문서에서 확인된 사실(🔍)과 AI 추론(💡)을 명확히 분리
- **실행 가능한 아웃풋**: 분석으로 끝나지 않고, "다음에 뭘 해야 하는가"까지 도출
- **체크리스트 기반**: 프레임워크의 항목을 빠짐없이 검토

## 2가지 분석 모드

이 스킬은 **자동 감지**로 적합한 프레임워크를 선택한다.

### Mode A: 일반 기업 RFP
- 프레임워크: `@framework/analysis-framework.md`
- 특징: 클라이언트 상황 분석 → 타겟/CX 분석 → 전략적 차별화 → Winning Point
- 적합: 민간 기업 제안, 경쟁 PT, 브랜드 프로젝트

### Mode B: 정부사업 · 공공입찰
- 프레임워크: `@framework/gov-analysis-framework.md`
- 특징: 평가 배점 분석 → 자격요건/행정 리스크 → GO/PASS 판단 → 수주 전략
- 적합: 정부사업, 공공입찰, 조달청/나라장터 공고, 지자체 사업

### 모드 자동 감지 기준

문서에서 다음 키워드가 발견되면 **Mode B (정부사업)**:
- 나라장터, 조달청, 공고번호, 입찰공고, 제안요청서
- 용역, 사업공고, 수의계약, 협상에 의한 계약
- 평가위원, 기술평가, 가격평가, 적격심사
- 국고보조, 지방비, 매칭펀드
- 하도급, 공동수급, 컨소시엄 구성
- 정산, 감리, 준공, 사업완료보고
- 시·군·구, 도, 부·처·청, 공공기관, 진흥원, 재단

위 키워드가 없으면 **Mode A (일반 기업)**으로 진행.
감지 결과를 사용자에게 확인: "정부사업으로 감지했습니다. 맞나요?"

## Execution Flow

### Phase 0: 입력 확인 + 모드 감지

사용자가 다음 중 하나 이상을 제공한다:
- RFP 문서 (PDF, 마크다운, 텍스트)
- 사업공고 / 제안요청서
- 미팅록 / 회의록
- 인터뷰 기록
- 클라이언트 관련 메모
- 프로젝트 브리프

**입력이 없으면 물어본다:**
```
분석할 자료를 알려주세요.

1. RFP/사업공고 문서 경로 (필수)
2. 미팅록/회의록 (있으면)
3. 인터뷰 기록 (있으면)
4. 기타 참고 자료 (있으면)

파일 경로를 주시거나, 프로젝트 폴더 경로를 주시면 해당 폴더 내 자료를 모두 분석합니다.
```

**프로젝트 폴더가 주어진 경우:**
- Glob으로 폴더 내 모든 문서 파일(.md, .txt, .pdf) 검색
- 파일 목록을 보여주고 분석 대상 확인

**모드 감지:**
문서를 1차로 읽은 후, 위 "모드 자동 감지 기준"의 키워드를 검색하여 Mode A/B를 결정한다.
감지 결과를 사용자에게 알린다:
```
📋 모드 감지: [정부사업/공공입찰] (또는 [일반 기업 RFP])
→ [해당 모드] 프레임워크로 분석을 진행합니다. 변경하려면 말씀해주세요.
```

### Phase 1: 자료 수집 및 기본 정보 추출

**1-1. 전체 자료 읽기**
- 제공된 모든 파일을 Read로 전문 읽기 (샘플링 금지)
- 파일이 많으면 Agent를 활용하여 병렬 읽기

**1-2. 기본 정보 추출**

문서에서 다음을 추출한다 (없는 항목은 "미확인"으로 표기):

| 항목 | 내용 |
|------|------|
| 프로젝트명 | |
| 발주사 | |
| 프로젝트 기간 | |
| 예산 | |
| 제출 마감일 | |
| 발표/PT 일정 | |
| 운영 기간 | |
| 주요 일정 | |

**1-3. 핵심 키워드 추출**
- 문서 전체에서 반복 등장하는 키워드 10~20개 추출
- 키워드 빈도와 맥락을 함께 기록
- 이 키워드들이 Phase 3의 우선순위 판단 근거가 된다

**1-4. 자료 간 교차 확인**
- RFP 본문과 미팅록 사이의 불일치/추가 정보 식별
- 미팅에서만 언급된 비공식 요구사항 포착
- 인터뷰에서 드러난 클라이언트의 감정/강조점 포착

---

### Phase 2: 클라이언트 컨텍스트 분석

**2-1. 웹 리서치 (WebSearch 활용)**

발주사명을 기반으로 다음을 조사한다:

```
검색 쿼리 예시:
- "[발주사명] 최근 뉴스"
- "[발주사명] 브랜드 전략"
- "[발주사명] 신규 사업"
- "[발주사명] [관련 업종] 시장"
- "[발주사명] 경쟁사"
```

**수집할 정보:**
- 기업 개요 및 핵심 사업
- 최근 사업 방향성 변화
- 최근 투자/신규사업/브랜드 전략
- 주요 경쟁사 및 시장 포지셔닝
- 최근 6개월~2년 이슈 (뉴스, 보도자료)

**⚠️ 웹 리서치 불가 또는 정보 부족 시:**
- 문서에서 추출한 정보만으로 분석 진행
- "추가 리서치 필요 항목"에 구체적으로 기재
- 추론임을 명확히 표기 (💡)

**2-2. 프로젝트 발생 배경 추론**

Phase 1의 문서 분석 + Phase 2-1의 웹 리서치를 종합하여:

- **이 프로젝트를 진행하는 진짜 이유** (표면적 이유 vs 실제 이유)
- **클라이언트 내부 KPI** (무엇을 성과로 보는가)
- **숨겨진 목적** (RFP에 명시되지 않은 기대)
- **두려워하는 리스크** (실패 시 어떤 결과가 오는가)
- **기대하는 결과** (성공의 그림)

각 항목에 근거를 반드시 표기:
- 🔍 문서 근거: "RFP p.3에 '신규 고객층 확대'라고 명시"
- 💡 추론: "최근 경쟁사 A의 팝업 성공 사례를 의식한 것으로 보임"

---

### Phase 3: 요구사항 + 타겟 분석

**3-1. 요구사항 구조화**

| 구분 | 요구사항 | 근거 | 우선순위 |
|------|---------|------|---------|
| 명시적 | | RFP p.X | ★★★ |
| 암묵적 | | 미팅록 발언 | ★★☆ |

**우선순위 판단 기준:**
- 문서에서 반복 언급 빈도 (Phase 1-3의 키워드)
- 미팅에서의 강조 정도
- 비즈니스 임팩트 크기
- 마감 긴급도

**3-2. 핵심 문제 정의**

다음 5가지 질문에 대한 답을 도출한다:
1. 클라이언트는 지금 무엇이 **부족**한가?
2. 무엇을 **해결**하고 싶은가?
3. 무엇이 **실패**하고 있는가?
4. 무엇을 **바꾸고** 싶은가?
5. 무엇을 시장에 **보여주고** 싶은가?

**3-3. 최종 산출물 정의**

프로젝트 종료 시 반드시 나와야 하는 결과물을 리스트업.
각 산출물에 대해:
- 명시적으로 요구된 것인가 / 우리가 제안하는 것인가
- 우선순위
- 예상 난이도

**3-4. 타겟 분석**

문서에서 확인된 타겟 정보 + 추론:
- 핵심 타겟 프로파일 (성별, 연령, 라이프스타일, 소비 성향)
- 고객 경험 관점 (왜 반응하는가, 무엇을 공유하는가, 방문 이유)
- 현재 고객 경험의 문제점

---

### Phase 4: 전략적 차별화 분석

@framework/analysis-framework.md의 4개 관점을 기준으로 분석한다.

**⚠️ 중요: 모든 차별화 포인트는 Phase 2~3의 분석 결과와 연결되어야 한다.**
"차별화를 위한 차별화"가 아니라, "클라이언트의 진짜 문제를 해결하는 차별화"여야 한다.

**4-1. 디자인 관점**
- 시각적 차별점
- 공간 경험 차별점
- 콘텐츠 연출 방식
- 브랜드 톤앤매너
- SNS 확산 가능성 (사진/영상 찍고 싶은 요소)
- 오프라인 경험의 밀도

**4-2. 고객 경험(CX) 관점**
- 고객 동선 설계
- 체류 경험
- 인터랙션 / 참여 유도
- 공유 유도 메커니즘
- 재방문 구조
- 감정 설계 (어떤 감정을 경험하게 할 것인가)
- 커뮤니티화 가능성

**4-3. 비즈니스 관점**
- 실제 매출 기여 가능성
- 브랜드 자산 축적 가능성
- 데이터 확보 가능성
- 장기적 확장 가능성
- 운영 효율성
- ROI 관점 설득력
- 후속 프로젝트 연결 가능성

**4-4. 클라이언트 시너지 관점**
- 클라이언트 브랜드와의 연결성
- 기존 자산(공간, 고객, 데이터) 활용 가능성
- 기존 사업과의 시너지
- 내부 조직과의 협업 용이성
- 향후 확장성
- 장기 파트너십 가능성

**각 관점별 출력 형식:**
```
### [관점명]

**현재 상태**: (클라이언트의 현재 수준)
**기회 영역**: (우리가 제안할 수 있는 차별점)
**실행 아이디어**: (구체적 아이디어 1~3개)
**근거**: 🔍/💡
```

### Phase 5: 전략적 질문 생성 (Gap-Closing Questions)

**Phase 1~4 분석 과정에서 드러난 정보 공백을 메우기 위한 질문을 생성한다.**
이 질문들은 담당자 또는 클라이언트에게 직접 보내서 추가 정보를 확보하는 용도다.

**5-1. 질문 도출 기준**

분석 전체를 돌아보며 다음 기준으로 질문을 생성한다:

| 유형 | 기준 | 예시 |
|------|------|------|
| **🔴 필수 확인** | 이것 없이는 제안 방향 자체가 흔들리는 정보 | 예산 규모, 의사결정 구조, 평가 기준 |
| **🟡 논리 보강** | 있으면 제안의 설득력이 크게 올라가는 정보 | 과거 유사 프로젝트 경험, 내부 KPI, 경쟁사 인식 |
| **🟢 차별화 단서** | 경쟁사가 놓칠 수 있는 인사이트를 얻을 수 있는 질문 | 내부 고민, 실패 경험, 이해관계자 역학 |

**5-2. 질문 설계 원칙**

1. **열린 질문 우선**: "예/아니오"가 아니라 맥락이 드러나는 질문
   - ❌ "타겟이 2030인가요?"
   - ✅ "이 프로젝트에서 가장 반응해줬으면 하는 고객층은 어떤 사람들인가요?"

2. **의도를 숨기지 않는다**: 왜 이 질문을 하는지 짧게 맥락을 제공
   - "제안 방향을 좁히기 위해 여쭤보는 건데요, ..."
   - "경쟁 제안과 차별화하려면 이 부분이 중요해서요, ..."

3. **담당자용 / 클라이언트용 구분**:
   - **담당자(내부)용**: 솔직하고 직접적. "이거 예산 얼마야?", "평가 기준 뭐야?"
   - **클라이언트(외부)용**: 프로페셔널한 톤. 질문 자체가 우리의 전문성을 보여주는 수준

4. **질문 수**: 5~10개. 폭격하지 않는다. 우선순위 순으로 정렬.

**5-3. 질문 출력 형식**

```markdown
## 📋 추가 확인 질문

### 🔴 필수 확인 (이것 없이는 제안 방향 설정 불가)

**Q1. [질문]**
- 배경: [왜 이 질문이 필요한지]
- 현재 추정: [문서 기반으로 추정한 내용]
- 이 답변에 따라: [답변에 따라 제안 방향이 어떻게 달라지는지]

**Q2. [질문]**
- 배경: ...
- 현재 추정: ...
- 이 답변에 따라: ...

### 🟡 논리 보강 (있으면 제안 설득력 대폭 향상)

**Q3. [질문]**
- 배경: ...

### 🟢 차별화 단서 (경쟁사가 놓칠 수 있는 인사이트)

**Q5. [질문]**
- 배경: ...
```

**5-4. 클라이언트 전달용 버전 (선택)**

사용자가 요청 시, 질문을 클라이언트에게 바로 보낼 수 있는 정제된 버전도 생성한다:
- 내부 메모(배경, 추정, 전략적 의도) 제거
- 프로페셔널한 톤으로 재작성
- 질문 자체가 "이 팀은 깊이 있게 고민하고 있구나"라는 인상을 주도록
- 별도 파일로 저장: `rfp-questions-[클라이언트명].md`
- 향후 Vercel 대화형 인터페이스로 전환 가능한 구조 (질문 ID + 카테고리 + 본문)

## Critical Rules

**공통 (Mode A + B):**
1. **모든 분석은 근거 기반**: 추론과 사실을 명확히 구분 (🔍/💡)
2. **빠짐없이 체크**: 해당 모드 프레임워크의 모든 항목을 한 번은 검토
3. **정보 부족 시 솔직하게**: "추가 리서치 필요"로 표기, 억지 분석 금지
4. **실행 가능성**: 모든 제안은 "그래서 뭘 해야 하는가"로 연결
5. **경쟁 관점**: 항상 "다른 팀은 어떻게 접근할까"를 염두에 둠
6. **자료 간 교차 검증**: RFP만 보지 않고, 미팅록/인터뷰에서 비공식 정보 포착
7. **미확인은 미확인으로**: 문서에 없는 정보를 있는 것처럼 쓰지 않는다

**Mode A 추가:**
8. **클라이언트 관점 유지**: "우리가 하고 싶은 것"이 아니라 "클라이언트가 필요한 것" 중심

**Mode B 추가:**
8. **평가 배점이 전략의 중심축**: 배점 높은 곳에 자원 집중
9. **행정 리스크 반드시 분석**: 정산, 감리, 하도급 등 공공사업 특유의 리스크
10. **GO/PASS 판단 필수**: 수주 가능성과 추천 액션을 반드시 제시
11. **반복 표현 = 중요 신호**: 공고문에서 반복되는 표현을 가중치 높게 해석
12. **우리 강점 연결**: 공간/콘텐츠/운영/데이터/AX 중 연결 가능한 지점이 있으면 반드시 제안
