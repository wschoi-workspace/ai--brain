# RFP Analysis: 프로젝트 제안 전략 분석기

> 단순 요약이 아니라, **"왜 이 프로젝트가 생겼는가" → "진짜 문제는 무엇인가" → "어떻게 이길 것인가"**를 구조적으로 도출한다.

## Core Philosophy

- **문서 기반 우선**: 제공된 자료에서 최대한 추출하고, 웹 리서치는 보완용
- **추론과 사실 구분**: 문서에서 확인된 사실(🔍)과 AI 추론(💡)을 명확히 분리
- **실행 가능한 아웃풋**: 분석으로 끝나지 않고, "다음에 뭘 해야 하는가"까지 도출
- **체크리스트 기반**: 프레임워크의 항목을 빠짐없이 검토

## 2가지 분석 모드

이 스킬은 **자동 감지**로 적합한 프레임워크를 선택한다.

### Mode A: 일반 기업 RFP
- 프레임워크: `@framework/analysis-framework.md`
- 특징: 클라이언트 상황 분석 → 타겟/CX 분석 → 전략적 차별화 → Winning Point
- 적합: 민간 기업 제안, 경쟁 PT, 브랜드 프로젝트

### Mode B: 정부사업 · 공공입찰
- 프레임워크: `@framework/gov-analysis-framework.md`
- 특징: 평가 배점 분석 → 자격요건/행정 리스크 → GO/PASS 판단 → 수주 전략
- 적합: 정부사업, 공공입찰, 조달청/나라장터 공고, 지자체 사업

### 모드 자동 감지 기준

문서에서 다음 키워드가 발견되면 **Mode B (정부사업)**:
- 나라장터, 조달청, 공고번호, 입찰공고, 제안요청서
- 용역, 사업공고, 수의계약, 협상에 의한 계약
- 평가위원, 기술평가, 가격평가, 적격심사
- 국고보조, 지방비, 매칭펀드
- 하도급, 공동수급, 컨소시엄 구성
- 정산, 감리, 준공, 사업완료보고
- 시·군·구, 도, 부·처·청, 공공기관, 진흥원, 재단

위 키워드가 없으면 **Mode A (일반 기업)**으로 진행.
감지 결과를 사용자에게 확인: "정부사업으로 감지했습니다. 맞나요?"

## Team Architecture: 병렬 에이전트 팀

> 이 스킬은 메인 Claude가 **오케스트레이터(팀장)** 역할을 하고,
> 독립적인 작업을 **병렬 에이전트(팀원)**에게 동시 배분하여 속도와 깊이를 모두 확보한다.

### 에이전트 역할 정의

| 에이전트 | subagent_type | 역할 | Phase |
|---------|---------------|------|-------|
| **문서분석팀** | `analysis-worker` | 문서 전문 읽기 → 기본정보 추출 → 요구사항 구조화 → 교차검증 | 1 |
| **리서치팀** | `general-purpose` | WebSearch로 발주사 배경, 시장, 경쟁사, 정책 동향 조사 | 2 |
| **키워드분석팀** | `analysis-worker` | 문서 키워드 빈도 분석 → 반복 패턴 → 우선순위 신호 추출 | 1 |
| **요구분석팀** | `analysis-worker` | Phase 1+2 결과 기반 → 핵심 문제 정의 → 타겟 분석 | 3 |
| **전략팀** | `analysis-worker` | Phase 1+2 결과 기반 → 4관점 차별화 → Winning Point | 4 |

### 실행 흐름도

```
Phase 0: 오케스트레이터 직접 실행 (입력 확인 + 모드 감지)
    │
    ▼
Phase 1+2: ── 3개 에이전트 동시 실행 ──────────────────────
    │         ├─ 문서분석팀 (문서 읽기 + 기본정보 + 교차검증)
    │         ├─ 리서치팀 (발주사 배경 + 시장 + 경쟁사)
    │         └─ 키워드분석팀 (키워드 빈도 + 패턴)
    │                          │
    ▼                          ▼
Phase 1+2 결과 통합: 오케스트레이터가 3팀 결과를 합산
    │
    ▼
Phase 3+4: ── 2개 에이전트 동시 실행 ──────────────────────
    │         ├─ 요구분석팀 (요구사항 + 타겟 + 핵심문제)
    │         └─ 전략팀 (4관점 차별화 + 전략 방향)
    │                          │
    ▼                          ▼
Phase 3+4 결과 통합: 오케스트레이터가 2팀 결과를 합산
    │
    ▼
Phase 5+6: 오케스트레이터 직접 실행 (질문 생성 + 리포트 통합)
```

### 에이전트 호출 규칙

1. **반드시 병렬 호출**: Phase 1+2의 3개 에이전트는 하나의 메시지에서 동시 호출
2. **컨텍스트 전달**: 각 에이전트에게 Phase 0에서 파악한 기본 정보(파일 경로, 모드, 발주사명)를 프롬프트에 포함
3. **결과 형식 통일**: 각 에이전트는 마크다운 형식으로 결과를 반환하며, 🔍(문서 근거)/💡(추론) 표기를 준수. 원문 인용은 최소화하고 요약 우선.
4. **통합은 오케스트레이터**: 에이전트 결과 간 충돌/보완 관계를 메인 Claude가 판단하여 하나의 분석으로 통합
5. **Mode 분기 원칙**:
   - Phase 0에서 Mode A 감지 → "Phase 1+2" 섹션의 프롬프트 사용
   - Phase 0에서 Mode B 감지 → "Phase 1+2-GOV" 섹션의 프롬프트 사용
   - 에이전트 A/B/C의 프롬프트가 모드별로 다름. **혼용 금지.**
6. **에이전트 실패 처리**:
   - 리서치팀(B) 실패 시: 오케스트레이터가 직접 WebSearch로 핵심 3개 쿼리만 실행
   - 문서분석팀(A) 실패 시: Phase 전체 중단 → 사용자에게 파일 경로 재확인 요청
   - 키워드분석팀(C) 실패 시: 오케스트레이터가 Grep으로 핵심 키워드 빈도만 확인
   - 빈 결과 반환 시: 해당 섹션을 "정보 부족 — 수동 보완 필요"로 표기하고 분석 계속
7. **서브에이전트 도구 범위**: 오케스트레이터의 allowed-tools는 서브에이전트에 상속되지 않음. 각 subagent_type은 자체 도구 세트를 사용:
   - `analysis-worker`: Read, Glob, Grep (WebSearch/Bash 없음)
   - `general-purpose`: 모든 도구 사용 가능
8. **진행 상황 안내**: Phase 1+2 시작 전 사용자에게 "3개 팀이 병렬 분석 중입니다"라고 안내

## Execution Flow

### Phase 0: 입력 확인 + 모드 감지 (오케스트레이터 직접 실행)

사용자가 다음 중 하나 이상을 제공한다:
- RFP 문서 (PDF, 마크다운, 텍스트)
- 사업공고 / 제안요청서
- 미팅록 / 회의록
- 인터뷰 기록
- 클라이언트 관련 메모
- 프로젝트 브리프

**입력이 없으면 물어본다:**
```
분석할 자료를 알려주세요.

1. RFP/사업공고 문서 경로 (필수)
2. 미팅록/회의록 (있으면)
3. 인터뷰 기록 (있으면)
4. 기타 참고 자료 (있으면)

파일 경로를 주시거나, 프로젝트 폴더 경로를 주시면 해당 폴더 내 자료를 모두 분석합니다.
```

**프로젝트 폴더가 주어진 경우:**
- Glob으로 폴더 내 모든 문서 파일(.md, .txt, .pdf) 검색
- 파일 목록을 보여주고 분석 대상 확인

**모드 감지:**
문서를 1차로 읽은 후 (핵심 문서 앞부분만 빠르게 스캔), 위 "모드 자동 감지 기준"의 키워드를 검색하여 Mode A/B를 결정한다.
감지 결과를 사용자에게 알린다:
```
📋 모드 감지: [정부사업/공공입찰] (또는 [일반 기업 RFP])
→ [해당 모드] 프레임워크로 분석을 진행합니다. 변경하려면 말씀해주세요.
```

**Phase 0 산출물** (다음 Phase에 전달):
- 분석 대상 파일 경로 목록
- 감지된 모드 (A/B)
- 발주사/기관명
- 프로젝트명 (파악된 경우)
- 프로젝트 분야 (문서에서 파악 가능한 경우 — 에이전트 B 프롬프트에 필요)

### Phase 1+2: 3개 에이전트 동시 실행

> Phase 0 완료 후, 아래 3개 에이전트를 **하나의 메시지에서 동시에** Agent tool로 호출한다.
> 각 에이전트에게 Phase 0의 산출물(파일 경로, 모드, 발주사명)을 프롬프트로 전달한다.

#### 에이전트 A: 문서분석팀 (analysis-worker)

프롬프트 템플릿:
```
RFP 문서 심층 분석을 수행하라.

대상 파일: [파일 경로 목록]
모드: [A 또는 B]

다음을 수행:

1. 전체 자료 읽기 (Read로 전문 읽기, 샘플링 금지)

2. 기본 정보 추출 (없는 항목은 "미확인"):
   - 프로젝트명, 발주사, 프로젝트 기간, 예산
   - 제출 마감일, 발표/PT 일정, 운영 기간, 주요 일정

3. 자료 간 교차 확인:
   - RFP 본문과 미팅록 사이의 불일치/추가 정보
   - 미팅에서만 언급된 비공식 요구사항
   - 인터뷰에서 드러난 클라이언트의 감정/강조점

4. 요구사항 1차 추출 (우선순위 판단 없이, 추출만):
   - 명시적 요구사항 (문서 페이지 번호 포함)
   - 암묵적 요구사항 (미팅록/인터뷰 근거)
   - ※ 우선순위 판단은 이후 단계에서 키워드 분석 결과와 교차하여 수행함

결과를 마크다운으로 반환. 🔍(문서 근거)/💡(추론) 표기 필수.
```

#### 에이전트 B: 리서치팀 (general-purpose)

프롬프트 템플릿:
```
발주사 배경 및 시장 리서치를 수행하라.

발주사: [발주사명]
프로젝트 분야: [간단 설명]
모드: [A 또는 B]

Mode A일 경우 다음을 WebSearch로 조사:
- "[발주사명] 최근 뉴스" / "브랜드 전략" / "신규 사업"
- "[발주사명] [관련 업종] 시장" / "경쟁사"
- 수집: 기업 개요, 최근 사업 방향성 변화, 투자/전략, 경쟁사, 최근 6개월~2년 이슈

Mode B일 경우 다음을 WebSearch로 조사:
- "[발주처명] 유사 사업 이력" / "기존 수행사"
- "[관련 정책] 동향" / "[해당 지역] 최근 이슈"
- 수집: 발주처 최근 유사 사업, 기존 수행사, 정책 동향, 지역 이슈

결과를 마크다운으로 반환. 정보 출처 명시. 정보 부족 시 "추가 리서치 필요"로 표기.
```

#### 에이전트 C: 키워드분석팀 (analysis-worker)

프롬프트 템플릿:
```
RFP 문서의 키워드 빈도 분석 및 우선순위 신호를 추출하라.

대상 파일: [파일 경로 목록]

다음을 수행:

1. 문서 전체에서 반복 등장하는 키워드 10~20개 추출
2. 각 키워드의 등장 빈도 + 등장 맥락(어떤 섹션에서 나오는지)
3. 반복 패턴에서 발주사가 가장 중요하게 보는 포인트 추론
4. 가장 강조되는 가치, 반복 표현, 실질적으로 원하는 결과물 정리

결과를 마크다운으로 반환:
- 키워드 빈도 테이블 (키워드 | 빈도 | 등장 맥락)
- "발주사가 진짜 원하는 것" 3줄 요약
```

#### 에이전트 결과 통합 (오케스트레이터)

3개 에이전트 결과가 돌아오면:

1. **기본 정보**: 문서분석팀 결과를 기준으로 정리
2. **발주 배경 추론**: 리서치팀 + 문서분석팀 결과를 종합하여 Phase 2-2 도출
   - 이 프로젝트를 진행하는 진짜 이유 (표면적 vs 실제)
   - 클라이언트 내부 KPI
   - 숨겨진 목적
   - 두려워하는 리스크
   - 기대하는 결과
3. **키워드 → 우선순위**: 키워드분석팀 결과를 Phase 3의 우선순위 판단 기준으로 활용
4. **충돌 해소**: 에이전트 간 분석이 다른 경우 문서 근거(🔍)를 우선

### Phase 3+4: 2개 에이전트 동시 실행

> Phase 1+2 통합 결과를 기반으로, 아래 2개 에이전트를 **동시에** 호출한다.
>
> **통합 결과 전달 방식**: 오케스트레이터가 Phase 1+2 통합 결과를 프롬프트 텍스트로 직접 전달한다.
> 프롬프트가 지나치게 길어지는 경우, 임시 파일(`_rfp-phase12-summary.md`)로 Write한 후
> "이 파일을 Read로 읽어서 시작하라"는 지시로 대체할 수 있다.

#### 에이전트 D: 요구분석팀 (analysis-worker)

프롬프트 템플릿:
```
아래는 Phase 1+2 통합 분석 결과다.
원본 파일: [파일 경로 목록] (필요 시 직접 Read로 근거 재확인 가능)

[Phase 1+2 통합 결과 전문]
```

#### 에이전트 결과 통합 (오케스트레이터)

2개 에이전트 결과가 돌아오면:

1. **요구사항 ↔ 전략 연결**: 요구분석팀의 핵심 문제와 전략팀의 차별화 포인트를 매칭
2. **Gap 발견**: 요구사항은 있는데 전략이 없는 영역, 전략은 있는데 요구와 안 맞는 영역 식별
3. **우선순위 재조정**: 두 팀의 분석을 교차하여 최종 우선순위 확정
4. **Winning Point 확정**: 전략팀의 차별화 포인트 중 요구분석팀의 핵심 문제와 직접 연결되는 3~5개를 선별하여 최종 Winning Point로 확정
5. **주의사항 및 리스크 종합**: 요구분석팀의 Gap, 리서치팀의 시장 리스크, 문서분석팀의 교차검증 불일치를 모아 리포트 섹션 11의 입력 소스로 정리

이 통합 결과가 Phase 5(질문 생성)와 Phase 6(리포트)의 직접 입력 소스다.

### Phase 5: 전략적 질문 생성 (오케스트레이터 직접 실행)

다음 4가지 소스에서 정보 공백을 수집하여 질문을 생성한다:

1. **에이전트 A가 "미확인"으로 표기한 기본정보 항목** (예산, 마감일 등)
2. **Phase 1+2 통합에서 "추가 리서치 필요"로 표기된 항목**
3. **Phase 3+4 통합 Gap 목록** (요구사항은 있는데 전략이 없는 영역)
4. **에이전트 D의 핵심 문제 정의 중 💡 추론 기반 항목** (문서 근거 없는 것)

이 4가지를 가중치 순으로 정렬하여 아래 유형 분류(🔴/🟡/🟢)에 매핑한다.
이 질문들은 담당자 또는 클라이언트에게 직접 보내서 추가 정보를 확보하는 용도다.

**5-1. 질문 도출 기준**

분석 전체를 돌아보며 다음 기준으로 질문을 생성한다:

| 유형 | 기준 | 예시 |
|------|------|------|
| **🔴 필수 확인** | 이것 없이는 제안 방향 자체가 흔들리는 정보 | 예산 규모, 의사결정 구조, 평가 기준 |
| **🟡 논리 보강** | 있으면 제안의 설득력이 크게 올라가는 정보 | 과거 유사 프로젝트 경험, 내부 KPI, 경쟁사 인식 |
| **🟢 차별화 단서** | 경쟁사가 놓칠 수 있는 인사이트를 얻을 수 있는 질문 | 내부 고민, 실패 경험, 이해관계자 역학 |

**5-2. 질문 설계 원칙**

1. **열린 질문 우선**: "예/아니오"가 아니라 맥락이 드러나는 질문
   - ❌ "타겟이 2030인가요?"
   - ✅ "이 프로젝트에서 가장 반응해줬으면 하는 고객층은 어떤 사람들인가요?"

2. **의도를 숨기지 않는다**: 왜 이 질문을 하는지 짧게 맥락을 제공
   - "제안 방향을 좁히기 위해 여쭤보는 건데요, ..."
   - "경쟁 제안과 차별화하려면 이 부분이 중요해서요, ..."

3. **담당자용 / 클라이언트용 구분**:
   - **담당자(내부)용**: 솔직하고 직접적. "이거 예산 얼마야?", "평가 기준 뭐야?"
   - **클라이언트(외부)용**: 프로페셔널한 톤. 질문 자체가 우리의 전문성을 보여주는 수준

4. **질문 수**: 5~10개. 폭격하지 않는다. 우선순위 순으로 정렬.

**5-3. 질문 출력 형식**

```markdown
## 📋 추가 확인 질문

### 🔴 필수 확인 (이것 없이는 제안 방향 설정 불가)

**Q1. [질문]**
- 배경: [왜 이 질문이 필요한지]
- 현재 추정: [문서 기반으로 추정한 내용]
- 이 답변에 따라: [답변에 따라 제안 방향이 어떻게 달라지는지]

**Q2. [질문]**
- 배경: ...
- 현재 추정: ...
- 이 답변에 따라: ...

### 🟡 논리 보강 (있으면 제안 설득력 대폭 향상)

**Q3. [질문]**
- 배경: ...

### 🟢 차별화 단서 (경쟁사가 놓칠 수 있는 인사이트)

**Q5. [질문]**
- 배경: ...
```

**5-4. 클라이언트 전달용 버전 (선택)**

사용자가 요청 시, 질문을 클라이언트에게 바로 보낼 수 있는 정제된 버전도 생성한다:
- 내부 메모(배경, 추정, 전략적 의도) 제거
- 프로페셔널한 톤으로 재작성
- 질문 자체가 "이 팀은 깊이 있게 고민하고 있구나"라는 인상을 주도록
- 별도 파일로 저장: `rfp-questions-[클라이언트명].md`
- 향후 Vercel 대화형 인터페이스로 전환 가능한 구조 (질문 ID + 카테고리 + 본문)

```
3. 기본 사업 정보 추출 (없는 항목은 "미확인"):
   사업명, 발주처, 사업기간, 예산, 마감일, 발표일정, 수행기간, 공고번호

4. 사업 목적 추론 (정책적·행정적 목적, 상위 정책 연결점)

5. 필수 자격요건 체크:
   업종 제한, 실적 조건, 컨소시엄 가능여부, 지역 제한, 인증 요구, 인력 보유 조건
   ⚠️ 각 조건별 충족 여부는 "확인 필요"로 표기

6. 제출 요구사항: 제안서 분량, PT 유무/시간, 제출 서류 목록, 예산 산출 방식

7. 운영/행정 리스크 1차 추출:
   정산 조건, 감리/검수 조건, 하도급 제한, 일정 현실성을
   문서에서 발견되는 즉시 별도 섹션으로 추출

결과를 마크다운으로 반환. 🔍(문서 근거)/💡(추론) 표기 필수.
```

### Phase 5-GOV: 제안서 양식 심층 분석

> 정부사업 RFP에는 제안서 양식이 함께 제공된다. 양식 자체가 평가 의도의 직접적 단서다.

**5G-1. 양식 구조 분석**

제안서 양식의 모든 섹션을 테이블로 구조화:
| 섹션 | 요구 항목 | 페이지/분량 제한 | 대응 평가 항목 | 추정 배점 비중 |
|------|-----------|-----------------|---------------|---------------|

- 각 섹션 ↔ 평가표 항목 매핑
- 분량 제한 자체가 우선순위 신호 (많이 쓰라는 곳 = 중요)
- 자유 기술 vs 정형화 영역 구분

**5G-2. 양식에서 읽어내는 숨은 요구**

양식의 구조적 특징에서 발주처의 진짜 의도를 추론:
- 세부 항목이 많은 섹션 → 구체적 답을 원함 (추상적 서술 금지)
- 자유 서술 섹션 → 차별화 기회 (점수 차이가 나는 곳)
- 별도 서식/표 요구 → 정량 비교 의도 (빈칸 없이, 수치 필수)
- 증빙 첨부 요구 → 실적 증명이 당락 좌우
- 양식에 없지만 RFP에는 있는 요구 → 가산점/정성 영역
- 양식에 있지만 RFP에서 안 강조한 항목 → 형식 요건 (기본만 채움)

**5G-3. 섹션별 작성 전략 가이드**

| 섹션 | 핵심 메시지 | 작성 방향 | 분량 배분 | 주의사항 |
|------|-----------|-----------|-----------|---------|

- 배점 높은 항목 연결 섹션: 지면 60%+ 투자
- 실적 섹션: 평가 키워드로 재서술
- 인력 섹션: 역할-과업 매핑 + 자격 증빙
- 일정 섹션: WBS + 마일스톤 + 리스크 대응
- 예산 섹션: 공종별 산출 근거

**5G-4. 양식 vs RFP vs 평가표 교차 분석**

3자 교차 매핑으로 Gap/불일치 발견:
| 양식 섹션 | RFP 요구사항 | 평가 항목 | 배점 | Gap/불일치 |
|-----------|-------------|-----------|------|-----------|

- Gap: RFP에서 강조하지만 양식에 전용 섹션 없는 항목 → 어디에 녹일지 전략
- 불일치: 양식과 RFP 간 용어/범위 차이 → 어느 쪽 기준인지 판단

### Phase 7-GOV: 전략 제언 (2단계 — 선택적 실행)

> 1단계 분석에서 GO 판단이 나온 경우에만 진행.
> 사용자가 "전략 제언까지", "2단계도", "기대효과 구조화", "방향성 제언" 등을 요청하면 실행.
> 요청 없으면 1단계 리포트까지만 생성하고, "2단계 전략 제언을 진행할까요?"로 안내.

프레임워크: `@framework/gov-strategy-framework.md` (v3.0 · 12 Step)

**구조 개요: 4국면 × 12 Step**

| 국면 | Step | 핵심 산출물 |
|------|------|-----------|
| 진단 심화 | 1. Pain Point 발굴 | 심사위원이 가려운 지점 |
| 진단 심화 | 2. 관통 메시지 | 제안서 전체의 방향타 (한 줄) |
| 진단 심화 | 3. Why Us | Pain ↔ 강점 1:1 매칭 |
| 전략 설계 | 4. JTBD 재정의 | 재정의된 가치 제안 |
| 전략 설계 | 5. 마케팅 전략 | STP + OMO + Flywheel |
| 전략 설계 | 6. 컨셉 3안 (A/B/C) | 비교 가능한 3개 방향 |
| 전략 설계 | 7. 예산 설계 | 투찰가 + 마진 2층 + 집행 배분 |
| 효과 구조화 | 8. 4-Layer Impact + ROI | 기대효과 1-Pager |
| 검증·전달 | 9. 설득 전략 | SCQA + Cialdini |
| 검증·전달 | 10. 자체평가 | 3안별 배점 채점 + 보완 우선순위 |
| 검증·전달 | 11. 의사결정 리뷰 | 3안 비교 + 결정 포인트 5개 |
| 검증·전달 | 12. 제안서 반영 가이드 | 배치 + 지면 + 비주얼 |

**7G-1. Pain Point 발굴** (진단 심화)
- 1단계의 발주처 정보 + WebSearch로 과거 동종 사업 딥리서치
- 반복되는 구조적 한계 포착 (성과 미공개, 일회성, 측정 불가 등)
- 산출물: "심사위원이 가려운 지점" 3개 정의

**7G-2. 관통 메시지** (진단 심화)
- Pain을 정반대로 뒤집은 한 문장
- "작년에 없던 그것을 올해 우리는 한다"
- 산출물: 제안서 전체가 수렴할 관통 메시지 한 줄

**7G-3. Why Us** (진단 심화)
- Pain의 각 갈래에 우리 강점을 1:1로 짝짓고, 증거로 뒷받침
- 경쟁사가 못 메우는 결합 차별화 (단독형 사업자 vs 컨소시엄)
- 산출물: Why Us 차별화 축 = 제안서의 뼈대

**7G-4. 사업 가치 재정의 (JTBD)** (전략 설계)
- Functional Job / Emotional Job / Social Job 3차원 재정의
- → 재정의된 가치 제안 1~2문장

**7G-5. 마케팅 전략 설계** (전략 설계)
- 마케팅 전략 캔버스 (1-Page)
- STP 분석 (Segmentation → Targeting → Positioning)
- OMO 통합 마케팅 (채널 역할, 경험 자산화, 데이터 통합)
- 가치 전달 경로 (Logic Model + CDJ)
- 확산 구조 (Growth Flywheel + Cialdini)
- 사업 포트폴리오 (Ansoff/BCG)

**7G-6. 컨셉 3안 (A/B/C) + 아이디에이션 검토** ← 핵심 원칙
- **최소 3개 베리에이션** — 의사결정자에게 선택지를 준다
- 권장 축: A안(안정형) / B안(차별형) / C안(혁신형), 사업 특성에 따라 변경 가능
- 각 안에: 한 줄 요약, 관통 메시지 연결, Rule of Three 전략, 강점/리스크/적합조건
- 3안 비교 매트릭스 + 하이브리드 옵션 제안
- RFP가 고정한 골격(예산·기간·장소)은 3안 모두 공통 전제
- **아이디에이션 검토 포함**:
  - 각 안의 마케팅/세일즈 관점 기대효과 극대화 가능성 평가
  - OMO 관점에서 온·오프라인 연결 구조 검토
  - 배점 대응력 사전 채점 (Step 10 자체평가의 초안)
  - 장단점 비교 테이블 + 실현 가능성 + 경쟁사 모방 난이도
  - **HTML 비교 리포트 자동 생성** (배점 대응력, 수주 가능성, 통합 권고 포함)

**7G-7. 예산 설계** (전략 설계)
- 투찰가: 추정가 대비 %(안정96~98 / 균형92~95 / 공격88~91)
- 마진 2층: 제출용(~13%) + 직접비 버퍼 = 내부 실질
- 컨소시엄: 마진을 출자비율로 배분, 대표사 실수령 산출
- 카테고리 배분 + 누락 점검 (해외출장, 발주처 부담 항목)
- 인력 정합: person-month 환산 → 인건비 일치 확인

**7G-8. 4-Layer Impact + ROI** ← 핵심 차별화

| Layer | 내용 | 성격 |
|-------|------|------|
| L1: 직접 성과 | 참여자수, 매출, 콘텐츠수 | 정량 (KPI 테이블) |
| L2: 행동 변화 | 인지도, NPS, 전환율, UGC | 정량+정성 |
| L3: 파급 효과 | 경제적/사회적/정책적 | 정성+추정정량 |
| L4: 지속가능성 | 자체수익, 역량내재화 | 구조적 |

- ROI 3종: 직접 ROI / 종합 ROI / 사회적 ROI(SROI)
- 시나리오: Best / Base / Worst
- **기대효과 종합 1-Pager** 생성

**7G-9. 평가위원 설득 전략** (검증·전달)
- SCQA: Pain(Step 1) → JTBD(Step 4) → Rule of Three(관통 메시지+Why Us)
- Cialdini 제안서 매핑: 권위/사회적증거/일관성/호감/희소성/통합
- 설득력 증폭: 앵커링, 프레이밍, 대비효과, 구체성효과

**7G-10. 자체평가** (검증·전달)
- 공고 배점표로 3안 각각 보수적 채점
- 감점 이유 명시 ("−4점, KPI 미확정")
- 협상적격선(85%) 통과 여부 확인
- 보완 우선순위 + [확인] 목록

**7G-11. 의사결정 리뷰** (검증·전달)
- 코어 6가지만 추출: Pain / 관통메시지 / Why Us / 3안 비교 / 예산 / 자체평가
- 3안 나란히 비교 (같은 구조)
- 결정 포인트 5개 명시: GO/PASS, 컨셉 택일, 컨소시엄, 예산, 보완

**7G-12. 제안서 반영 가이드** (검증·전달)
- 4-Layer 기대효과를 제안서 양식 어디에 배치할지
- 배점 기반 지면 재배분
- 비주얼 추천 (4-Layer 피라미드, Pain↔WhyUs 매칭, Flywheel, 3안 비교, Before/After, KPI 대시보드)

## Critical Rules

**공통 (Mode A + B):**
1. **모든 분석은 근거 기반**: 추론과 사실을 명확히 구분 (🔍/💡)
2. **빠짐없이 체크**: 해당 모드 프레임워크의 모든 항목을 한 번은 검토
3. **정보 부족 시 솔직하게**: "추가 리서치 필요"로 표기, 억지 분석 금지
4. **실행 가능성**: 모든 제안은 "그래서 뭘 해야 하는가"로 연결
5. **경쟁 관점**: 항상 "다른 팀은 어떻게 접근할까"를 염두에 둠
6. **자료 간 교차 검증**: RFP만 보지 않고, 미팅록/인터뷰에서 비공식 정보 포착
7. **미확인은 미확인으로**: 문서에 없는 정보를 있는 것처럼 쓰지 않는다

**Mode A 추가:**
8. **클라이언트 관점 유지**: "우리가 하고 싶은 것"이 아니라 "클라이언트가 필요한 것" 중심

**Mode B 추가:**
8. **평가 배점이 전략의 중심축**: 배점 높은 곳에 자원 집중
9. **행정 리스크 반드시 분석**: 정산, 감리, 하도급 등 공공사업 특유의 리스크
10. **GO/PASS 판단 필수**: 수주 가능성과 추천 액션을 반드시 제시
11. **반복 표현 = 중요 신호**: 공고문에서 반복되는 표현을 가중치 높게 해석
12. **우리 강점 연결**: 공간/콘텐츠/운영/데이터/AX 중 연결 가능한 지점이 있으면 반드시 제안
13. **제안서 양식은 평가 의도의 거울**: 양식 구조에서 발주처가 진짜 보고 싶은 것을 읽어낼 것
14. **양식 vs RFP vs 평가표 3자 교차 분석 필수**: Gap과 불일치가 경쟁사가 놓치는 포인트
