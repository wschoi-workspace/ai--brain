Part 1

수강생 근황 및 활용 사례 공유

수강생 활용 현황

강의 시작 전 수강생들의 1주일간 활용 사례를 공유하는 시간을 가졌다. 강사는 텔레그램 연동을 통한 아카이브 기능에 대해 높이 평가했다.

이야 텔레그램 잘 쓰시니까 너무 좋네요. 이게 정말 일상에 너무 좋은 점이거든요
이 아카이브가 된다는 사실이 이제 맥락을 잃기가 어렵거든요

제안서 작성 활용 사례

한 수강생이 제안서(Proposal) 작성에 Claude Code를 적극 활용한 사례를 공유했다. 혼자 진행하는 프로젝트뿐 아니라 교수님/박사님과 협업하는 프로젝트에서도 사용 중이었다.

핵심 내용

  • Claude Code로 리서치 + 제안 요청서 변동 사항 진행 + 디자인까지 완료
  • 교수님/박사님들이 "할 일이 없어진" 상황 발생
  • 제안서를 쓸 때 리서치와 디스커션 파트너로서의 가치를 실감
다 해서 제안 요청서 변동을 다 진행하고 리서치까지 해서 버전을 올리니까 교수님과 박사님들이 할 일이 없어요. 다들 숟가락을 못 얹어. 다 제안서까지 다 디자인까지 다 끝내버리는 거예요
리서치 안 할 거야, 도움을 받을 수 있지만 나랑 같이 디스커션을 일주일 정도 수술(논의)을 같이 할 수 있다고?
제가 쓴 제안서도 작년에 썼던 것까지 다 분석해서

싱킹 파트너(Thinking Partner)로서의 가치

그래서 이게 진짜 뭔가 같이 논의하는 파트너 같긴 했어요. 같이 할 때

업무 속도 향상 사례 (미용님)

미용님이 Claude Code를 활용해 업무를 빠르게 처리한 사례를 공유했다.

상황

  • 논의 → 리서치 → 결론 도출까지 매우 빠르게 완료
  • 혼자 했으면 반나절 걸렸을 작업이 1시간 이내에 완료
  • 결과물이 너무 빨리 나와서 "AI로 한 티가 날까봐" 보내기를 고민
저 혼자 했었으면 그래도 한 반나절은 걸렸을 텐데 한 1시간은 안 걸려서 끝나고
너무 빨리 보내면 AI로 하는 건 좀 티 나는데, 그거 안 좋아하시거든요 나이 있으신 분들은
그거 보자마자 '와 이거 어떻게 이렇게 빨리 잘했냐'

FOMO에 대한 조언

불필요한 FOMO 경계

강사는 Threads 등 SNS에서 Claude Code 관련 정보에 대한 FOMO를 경계할 것을 조언했다.

이거를 설치 안 하면 안 될 것 같고, 이거를 세팅 안 하면 안 될 것 같고, 이거 다 다운받아야 되고, 나도 뭐 댓글로 이거 안 하면 안 될 것 같고, 자료를 안 받으면 안 될 것 같고... 나 그거 너무 싫거든요
사실 그거 받는다고 해서 뭔가가 있다고 해서 일 되는 거 1도 없거든요. 오히려 잘 세팅된 내 시스템이 하나 있는 게 월등하게 잘 되는 거라서
핵심 메시지
다양한 도구와 자료를 무분별하게 수집하는 것보다, 하나의 잘 세팅된 시스템을 갖추는 것이 훨씬 효과적이다.

클로드 코드의 내부 구조 — 본사와 지사

숨김 파일 표시 방법

OS방법
MacFinder에서 Cmd + Shift + . (점)
Windows탐색기 > 보기 > 숨김 항목 체크

.claude 폴더 — 본사(전역) 개념

Claude Code를 설치하면 사용자 홈 디렉토리에 .claude라는 숨김 폴더가 생성된다. 이것이 Claude Code의 "본사"에 해당한다.

경로

  • Mac: /Users/[사용자명]/.claude/
  • Windows: C:\Users\[사용자명]\.claude\
여러분들은 클로드 코드를 설치를 했잖아요. 근데 그 클로드 코드가 어디에 설치되어 있는지도 아직 몰랐던 거예요
이 사용자 이름 안에 .claude라는 폴더가 설치된 겁니다

본사와 지사 구조

구분위치역할적용 범위
본사 (전역)~/.claude/모든 Claude Code 세션에 적용어디서든 적용
지사 (지역)[워크스페이스]/.claude/해당 워크스페이스에서만 적용해당 폴더 내에서만
큰 클로드 코드에 설치를 했어. 그럼 여기가 내 클로드 코드를 제어하는 모든 클로드 코드를 제어할 때는 본사라고 생각하시면 돼요
요번에 제가 세팅해서 클론해드린 Do Better 워크스페이스라는 곳은 지점입니다. 지사

CLAUDE.md의 본사/지사 적용

  • 본사 CLAUDE.md (~/.claude/CLAUDE.md): 어떤 폴더에서 작업하든 항상 읽힘
  • 지사 CLAUDE.md ([워크스페이스]/CLAUDE.md): 해당 워크스페이스에서만 읽힘
근데 만약에 여러분들이 Do Better 워크스페이스라고 하는 곳이 아닌 그 웹 폴더에서 클로드 작업을 할 거면은, 그때는 얘가 이 CLAUDE.md를 읽지 않기 때문에 내가 누군지 몰라
그렇기 때문에 더 좋은 거는 본사에 있는 CLAUDE.md를 잘 만드는 게 굉장히 좋다

전역 CLAUDE.md 만드는 방법

기존 지사(워크스페이스) CLAUDE.md를 전역으로 변환하는 것도 Claude에게 요청하면 된다.

지금 내 Do Better 워크스페이스 안에 있는 CLAUDE.md를 전역, 즉 전 지역에서 사용하는 전역 CLAUDE.md로 바꿔줘 하면은 알아서 바꿔줄 거구요
Part 4

전역 스킬 / 에이전트 / 커맨드

지역 스킬의 한계

Do Better 워크스페이스 안의 .claude/ 폴더에 있는 스킬, 에이전트, 커맨드는 해당 워크스페이스 안에서만 사용 가능하다.

이거는 이 Do Better 워크스페이스 안에 있을 때에만 사용을 할 수 있는 스킬과 에이전트와 커맨드였던 거예요

전역으로 전환하기

모든 곳에서 사용하고 싶다면 전역으로 전환 가능하다.

전역 스킬로 바꿔줘, 전역 커맨드로 바꿔줘, 전역 에이전트로 바꿔줘 하면은 얘가 이 내용을 그대로 [전역 폴더로 이동시켜줌]

지역 스킬이 유리한 경우

모든 스킬을 전역으로 올리면 컨텍스트를 소비하므로, 특정 프로젝트에서만 필요한 스킬은 지역에 두는 것이 효율적이다.

만약에 내가 하는 프로젝트가 굳이 그 전역에는 다 컨텍스트를 먹고, 정말 어떠한 폴더 안에 어떤 프로젝트 안에서만 필요한 스킬이라면 해당 지점에 스킬로 둬도 좋다는 거예요

중첩 CLAUDE.md 구조

CLAUDE.md는 중첩(nested)이 가능하다:

  1. 전역 (~/.claude/CLAUDE.md) — 전체 적용
  2. 워크스페이스 ([workspace]/CLAUDE.md) — 워크스페이스 적용
  3. 하위 폴더 ([workspace]/[project]/CLAUDE.md) — 프로젝트 적용
전역에 있는 CLAUDE.md가 있고, 그 지점 안에 있는 CLAUDE.md가 있고, 그 지점 안에 더 분점으로 CLAUDE.md가 있을 때 있습니다. 그러면 여기는 전체 다 읽고 있고, 없는 내용을 이렇게 추가 추가 하면 되는 거예요

CLAUDE.md 실제 사례 (강사의 전역 CLAUDE.md)

전역 CLAUDE.md 구성 요소

강사가 자신의 전역 CLAUDE.md를 공개하며 각 섹션의 의미를 설명했다.

1. 시스템 환경 정보

항목내용
OS 정보Mac 사용, Windows 병행
폴더 경로두 OS의 폴더 이름이 다르지만 연결되어 있음을 명시
효과git push/pull 시 오류 없이 동작
이렇게 때문에 얘가 git push를 하고 pull을 할 때 오류가 없이 항상 잘 돌아가는 겁니다

2. 사용자 프로필

  • 1인 비즈니스 컨텍스트
  • "Do better things with Sensei AI"
  • FMP 경력 15–16년

3. 비즈니스 현황

  • 블로그, 뉴스레터 등 채널 정보 포함
  • 진행 중인 서비스 및 파이프라인 정보
그러면 얘는 작업할 때 '아 얘가 이런 블로그를 하고 있고 어떤 사이트가 있다'라는 걸 알고 있는 거죠. 그러면 너무 쉬워요. 일이 너무 쉬워진다

4. 연동 서비스 명시

GWS CLI를 통한 Google 연결, Notion, Telegram 등 사용 중인 서비스를 명시해야 한다.

나 GWS CLI로 구글 다 연결을 하는데, 이 내용 그냥 CLAUDE.md에 업데이트해놔라고만 해보면 한 줄 들어가요. 그거 때문에 '아, 구글 쓸 때 꼭 GWS CLI를 써야겠구나'라는 생각을 얘가 하게 되잖아
경선이 같은 경우에는 '나는 지금 Notion, Telegram 이것들을 지금 사용 중이야, 다 API를 연결해서 사용 중이라고 하면' 얘가 딴짓할 수가 없어요. 다 연결이 되어 있다가 사실 알고 있는 거예요. 그 한 줄이 들어가서 중요합니다

5. 폴더 구조

PKM 폴더 구조를 명시하면 Claude가 파일 저장 위치를 자동으로 판단한다.

아예 안 읽어도 얘가 폴더 구조를 이해하고 있으면 '아 이렇게 해서 그냥 넣으면 되겠네'라는 생각을 하는 거죠

6. 파일 명명 규칙

YYYY-MM-DD 형식을 명시하지 않으면 Claude가 임의의 형식을 사용할 수 있다.

Q&A

Q1: 전역 CLAUDE.md는 어떻게 만드나요?

Claude에게 "지금 내 Do Better 워크스페이스 안에 있는 CLAUDE.md를 전역 CLAUDE.md로 바꿔줘"라고 요청하면 자동으로 변환해준다.

Q2: 구글 연동이 자꾸 까먹히는 이유는?

CLAUDE.md에 연동 서비스 정보를 명시적으로 기록해두지 않았기 때문이다. "나 GWS CLI로 구글 다 연결을 하는데, 이 내용 CLAUDE.md에 업데이트해놔라"고 한 줄만 추가하면 해결된다.

Q3: 숨김 파일이 안 보여요 (파인더 이슈)

  • Cmd + Shift + .을 파인더 창에서 눌러야 한다
  • 드롭박스 경로에서 열면 다른 결과가 나올 수 있으므로, 새 파인더 창에서 시도할 것

Key Concepts

핵심 개념 정리

.claude 폴더 (본사/지사 구조)

정의: Claude Code의 설정, 스킬, 에이전트, 커맨드가 저장되는 숨김 폴더

본사 (전역): ~/.claude/ — 모든 세션에 적용

지사 (지역): [workspace]/.claude/ — 해당 워크스페이스에서만 적용

핵심: 전역에서 사용할 것과 지역에서만 쓸 것을 구분하여 관리

CLAUDE.md (전역 설정 파일)

정의: Claude Code에게 "나는 누구이고, 어떻게 작업하는지"를 알려주는 핵심 설정 파일

필수 포함 항목: 사용자 프로필, 시스템 환경, 연동 서비스, 폴더 구조, 작업 규칙, 문제 해결 방식

적용: 항상 살아있는 문서로 지속적으로 업데이트 필요

전역 vs 지역 스킬/에이전트

전역 스킬: 모든 워크스페이스에서 사용 가능, 항상 컨텍스트 소비

지역 스킬: 특정 프로젝트에서만 로드, 컨텍스트 절약

판단 기준: "이 스킬이 어디서든 필요한가, 아니면 특정 프로젝트에서만 필요한가?"

Source of Truth (공식 가이드 문서)

정의: Claude Code 공식 문서에서 추출한 가이드 파일들

용도: 스킬, 에이전트, CLAUDE.md 등을 만들 때 기준점으로 참조

활용: "이 가이드를 참고해서 만들어줘"로 정확한 결과물 확보

중첩 CLAUDE.md

정의: 전역 > 워크스페이스 > 프로젝트 단위로 CLAUDE.md를 중첩 배치

동작 방식: 상위 CLAUDE.md를 모두 읽고, 하위로 갈수록 추가 정보를 적용

장점: 공통 규칙은 상위에, 프로젝트 특화 규칙은 하위에 배치
