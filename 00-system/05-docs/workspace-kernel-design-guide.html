WORKSPACE KERNEL — DESIGN GUIDE · 2026.07.22

커널을 지을 때
어긋나면 안 되는
설계 규격.

"얇은 커널 + 자라나는 방법" 철학을, 실제로 워크스페이스를 지을 때 기준이 되는 규격으로 못박은 문서. 원칙 → 구조 → 규약 → 방법 → 배포 순으로 읽는다.

설계 한 문장 — 기획자의 제2의 뇌를, 파일링이 아니라 운영체제로 설계한다.

근거 · concept: planner-workspace-kernel-concept.html · rules: CLAUDE-MD-BEST-PRACTICES.md

목적 · 철학

00

워크스페이스는 파일링 시스템이 아니라 Claude가 운영할 수 있는 외장 인지 시스템이다. 핵심은 무엇을 채워 주느냐가 아니라, 각자의 업무에서 스스로 자라나게 하는 뼈대와 방법을 주느냐다.

Pillar 01

기억

세션을 넘어 맥락이 살아남는다. progress + 메모리. 없으면 매번 백지에서 시작.

Pillar 02

규율

규칙을 CLAUDE.md에 한 번 박아 매 세션 자동 적용. 검증·보안·네이밍이 여기 산다.

Pillar 03

결정화

반복 업무가 스킬로 얼어붙는다. 단, 처음부터 주지 않는다 — 0에서 시작.

왜 구조가 Claude에게 유효한가 — 4대 메커니즘

지속성 — 세션마다 리셋. 구조 = 장기기억
컨텍스트 경제 — 필요한 파일만 읽음. 주소 = 탐색세↓
규율의 체화 — 헌법에 한 번 = 모든 세션 적용
검증 가능성 — 대조할 SSOT가 있으면 환각↓

설계 원칙 10

01

커널을 지을 때 어긋나면 안 되는 규칙. 각 원칙은 위반 시 증상과 함께 기억한다.

#원칙근거 / 위반 증상
P1헌법은 얇게CLAUDE.md < 100줄(이상 60). 길수록 무시당함 — 300줄↑이면 핵심부터 버려짐
P2지시 최소화지시문 150~200 한계. 시스템 프롬프트가 이미 ~50개 소비. 과다 시 일관성 붕괴
P3Progressive Disclosure상세는 별도 파일, 헌법엔 "어디 보라"만. 매뉴얼 전문 금지
P4스킬 0에서 시작물려받은 스킬은 언제 믿을지 모르고 만든 이 취향이 박혀 안 맞음
P53번 룰로 결정화1회차 과잉설계는 시작을 막고, 결정화 없는 반복은 매번 비용
P6SSOT 규율한 사실은 한 곳. .bak/사본 금지 — 어느 게 진짜인지 흐려짐
P7보안 기본값비밀·PIN·백업을 gitignore 선탑재. 백업 추적되면 비밀이 샘
P8활성/아카이브 분리시간축 물리 분리. 죽은 프로젝트가 활성에 섞이면 판단·검색 오염
P9검증 내장산출 프로세스에 검증 박기. "산출 완성 = 검증 완료"
P10자동생성 지양헌법은 손으로 신중히. /init 자동 덤프는 최고 레버리지 지점을 오염

커널 구조 스펙

02

최소 골격. 번호가 곧 주소다. 활성/아카이브(시간축) + 시스템/내용(목적축)의 이중 좌표계.

workspace/ ├── CLAUDE.md # 헌법 — 유일 진입점 (< 100줄) ├── README.md # 사람용 안내 + 온보딩 ├── .gitignore # 보안 기본값 선탑재 (§7) ├── 00-inbox/ # 저마찰 포집구 (분류 전 임시) ├── 00-system/ # 시스템: 템플릿·가이드·스크립트 │ ├── 01-templates/ │ └── 03-guides/ ├── 10-projects/ # 활성 (시한부) — 각 폴더에 progress ├── 20-operations/ # 지속 운영 (선택) ├── 30-knowledge/ # 지식 아카이브 (선택) ├── 40-personal/ # Daily Note · Todo ├── 50-resources/ # 참고 자료 (선택) └── 90-archive/ # 완료 프로젝트 (시간축 분리)

시작 필수 6개 — CLAUDE.md · 00-inbox · 00-system · 10-projects · 40-personal · 90-archive. 20/30/50과 99-moc는 필요가 생길 때 추가한다(안 쓰는 구조는 짓지 않는다 — P3·P4).

CLAUDE.md 스펙 + 템플릿

03

헌법은 WHAT(프로필·구조) / WHY(목적) / HOW(작업 규칙) 세 축만. 상세는 링크로 미룬다(P3).
금지 — 코드 스타일 규칙, 매뉴얼 전문, 안 쓰는 커맨드 나열.

CLAUDE.md 커널 템플릿복사용 · < 100줄
# CLAUDE.md

## 사용자 프로필
**이름**: {{이름}}
**역할**: {{직무}}
**주요 관심사**: {{3~5개 키워드}}
**접근 방식**: 인지 외주화 ❌ → 인지 증강 ✅ / "했다" 아닌 "잘했다"

## 워크스페이스 목적
{{한 문단 — 무엇을 운영하는가}}

## 폴더 구조 (Johnny Decimal)
00-inbox/ 빠른캡처   00-system/ 템플릿·가이드
10-projects/ 활성   40-personal/ 데일리·투두   90-archive/ 완료
(상세: README.md · 가이드: 00-system/03-guides/WORKSPACE-KERNEL-DESIGN-GUIDE.md)

## 작업 규칙
- 기존 파일 편집 우선, 새 파일 최소화
- 커밋은 요청 시에만
- 네이밍: `[숫자]-[설명적-이름]`
- 프로젝트마다 progress.md 유지 (세션 연속성의 핵심)
- "해줘" 뒤 반드시 "확인해줘" — 빠진 항목 체크
- 숫자·통계·인용은 3중 검증(추출→출처대조→재대조)
- 같은 작업 3회 반복 시 스킬로 결정화

## 현재 진행 중인 프로젝트
{{생기면 한 줄씩 — 프로젝트명: 상태}}

재실행 — /setup-workspace가 4문답(이름·역할·관심사·용도)으로 이 템플릿을 채운다. 단 스킬 목록·프로젝트를 미리 채우지 않는다(P4).

기억 규약 + 템플릿

04

세션 연속성은 프로젝트별 progress.md가 담당한다. 작업 내역·결정·다음 액션을 날짜별로 축적.

progress.md 템플릿복사용
# {{프로젝트명}} — Progress

> 시작: {{YYYY-MM-DD}} · 상태: 활성 · 마감: {{있으면}}

## 개요
{{한 줄 — 무엇을 왜}}

## 로그 (최신 위에)
### {{YYYY-MM-DD}}
- 한 일:
- 결정:
- 다음 액션: [ ]

## 열린 질문
-
파일명NN-progress.md(프로젝트 번호 접두) 또는 progress.md. 한 프로젝트 한 개
갱신 시점세션 종료 시(로그 한 줄 + 다음 액션). CLAUDE.md "확인해줘" 습관과 연결
전역 인덱스(선택) MEMORY.md에 프로젝트별 한 줄 포인터. 규모 커지면 도입

포집 규약 — inbox

05

구조의 엄격함이 캡처를 막지 않게 한다. 떠오른 건 일단 00-inbox에 던지고 나중에 분류.

진입 마찰 0 — 파일명 규칙 강요 안 함
정기 비우기 — 주 1회 프로젝트/지식/아카이브로 triage
모바일 포집 — 텔레그램 등 중계 경유로 설계

방법론 — "3번 하면 얼려라"

06

개인층을 스스로 자라게 하는 엔진. 스킬을 나눠주는 대신 스킬 만드는 리추얼을 준다.

1

그냥 한다 (1회차)

대화로 푼다. 구조화하지 않는다. 과잉 사전설계는 시작을 막는다.

2

패턴을 남긴다 (2회차)

프롬프트·산출 포맷을 progress/템플릿에 기록. 아직 스킬 아님.

3

얼린다 (3회차)

스킬로 결정화. 트리거·입력·검증·출력 고정. 이때부터 "운영".

4

검증을 박는다

스킬에 검증 단계 내장(P9). 산출 완성 = 검증 완료.

스킬 결정화 판정 체크리스트 — 4개↑ Yes면 얼린다

같은 작업을 3회 이상 했다

입력과 출력 형태가 반복적으로 동일하다

절차가 Verifiable하다(취향 아닌 검증 가능 단계)

매번 같은 프롬프트를 다시 쓰고 있다

남이 봐도 트리거 조건이 명확하다

트리거가 기존 스킬과 겹치지 않는다

단, 핵심이 "취향·맥락 판단"이면 스킬화하지 말고 사람이 쥔다(인지 증강 원칙).

보안 기본값 + .gitignore

07

배포판의 최소 요건은 개인정보가 새지 않는 것. 비밀·PIN·백업·런타임 패턴을 선탑재한다.

교훈 — PIN 포함 .bak이 추적돼 누출. live 파일만 ignore하고 백업(*.bak)을 놓치면 샌다.

.gitignore 보안 기본값복사용
# 비밀·환경
.env
*.env
*api_key*
*secret*

# 백업 — SSOT는 git 히스토리 (PIN 포함 .bak 추적 방지)
*.bak
*.bak-*

# PIN·계정 포함 데이터 (live + 백업 둘 다)
**/users.json
**/*-pin*.json

# 런타임·임시
__pycache__/
.DS_Store
~$*
node_modules/

# 루트 미리보기 이미지 (참조 없음)
/*.png

규칙 — 민감 파일을 ignore할 땐 live 파일과 그 백업(*.bak)을 항상 함께 넣는다.

배포 · 버전

08

커널을 "제품"으로 다루면 세 가지를 처음부터 설계한다.

Versioning

커널은 버전으로

커널만 얇게 + 버전 태그. 개인층 유지한 채 커널 업데이트만 pull.

Security

보안 기본값

§7 gitignore 선탑재. 배포 전 git ls-files로 비밀 유출 스캔.

Onboarding

10분 첫 성공

설치 후 첫 세션에 작은 루프 1회 체감 → 인지 증강 확인.

온보딩 10분 시나리오

STEP 1

clone → /setup-workspace

이름·역할·관심사·용도 4문답 → CLAUDE.md 생성

STEP 2

첫 데일리노트 자동 생성

오늘 날짜 노트가 열린다

STEP 3

할 일 3개 캡처 → 정리 한 바퀴

인지 증강을 몸으로 체감

STEP 4

"또 하면 스킬로" 안내

3번 룰로 마무리 — 스스로 자랄 씨앗

도메인 팩 (옵션)

09

기획자 공통 워크플로가 뚜렷하면, 커널 위에 검증된 공통 스킬 5~7개를 옵션 카트리지로 얹는다. 후보 — RFP분석 · 제안서 · 리서치 · 회의록정리 · 데일리/모닝 루틴.

편입 기준 — 모두 충족해야 함

특정인의 취향·클라이언트 맥락이 박혀 있지 않다(범용)

입출력이 표준화돼 있다

검증 단계가 내장돼 있다

트리거가 명확하고 충돌 없다

기준 미달 스킬은 개인층에서 각자 자라게 둔다. 기본은 커널+방법, 도메인 팩은 선택.

안티패턴 — 하지 말 것

10
✕

완성 워크스페이스 통째 배포 — 골격만 주고 근육은 못 준다. 남의 취향이 박힌다

✕

N개 스킬 물려주기 — 언제 믿을지 모르는 죽은 구조가 된다 (P4)

✕

CLAUDE.md 비대화 — 300줄 넘기면 무시당한다 (P1·P2)

✕

매뉴얼 전문을 헌법에 — Progressive Disclosure 위반 (P3)

✕

SSOT 중복·.bak 방치 — 어느 게 진짜인지 흐려진다 (P6)

✕

live만 ignore, 백업 누락 — 비밀이 백업으로 샌다 (P7)

✕

활성/완료 미분리 — 판단·검색 오염 (P8)

✕

취향 판단을 스킬화 — 인지 증강 아닌 외주화. 사람이 쥘 것

CONCLUSION

스킬을 나눠주지 말고,
스킬을 만드는 방법을 나눠줘라.
완성된 뇌가 아니라, 자라날 뇌를.

커널은 얇게 주고, 근육은 각자의 업무에서 자라게 한다 — 인지 외주화가 아닌 인지 증강.
