ARISA는 업무보고 시스템이 아니다.
조직의 사고를 운영하는 시스템이다.

Adaptive Resolution & Insight System Assistant — 직원의 사고를 번역·구조화하여 스스로 자신의 사고를 보게 만드는 Cognitive Reflection System. 보고는 목적이 아니라 인터페이스다.

최종 정리 2026-06-13 · 단일 기준(Single Source of Truth) · by Project Rent

이 문서를 "보고 정리 시스템"으로 이해하는 순간 설계는 실패한다. ARISA는 직원의 생각을 번역하는 거울이고, 대표가 조직을 읽는 창이며, 그 둘을 잇는 사고 번역 엔진이다. 모든 기술 결정은 아래 철학을 먼저 내면화한 뒤 내린다.

Part 1

철학 — 개발 불변 원칙

이 원칙들은 기능보다 우선한다. 기능이 원칙과 충돌하면 기능을 버린다.

1. 성장 코치가 아니라 사고의 거울이다.교정하지 않고 가시화한다. "아, 내가 이런 얘길 하고 있었구나"를 깨닫게 하는 것이 1차 목적. 성장은 거울 앞에서 스스로 일어난다.
2. 처리 순서는 Report → Interpretation → Reflection → Coaching.성장은 출발점이 아니라 결과물이다. 처음부터 코치가 되면 피로를 부른다. 질문 폭탄 금지.
3. 3계층 비중 — Mirror 80 / Reflector 15 / Coach 5.번역(80%) · 생각하게 만드는 질문 1~2개(15%) · 원할 때만 깊은 코칭(5%).
4. 업무량이 아니라 Output·Outcome을 추적한다."얼마나 오래"가 아니라 "무엇을 생산했고 무엇이 달라졌으며 어떤 의사결정을 가능하게 했는가".
5. 목표 순서 — 좋은 보고(1차) → 좋은 의사결정(2차) → 좋은 성장(3차).
6. 진짜 자산은 화면이 아니라 직원별 Thinking Data Log다.완성형 SaaS·대시보드를 먼저 만들지 않는다. 데이터가 쌓이는 구조부터.

5성장 미축적무엇이 늘고 무엇이 반복 부족한지직원 성장이 기록 안 됨

Part 3

시스템 구조 — 5개 엔진

직원 입력 → A. Input Parser → B. Report Completion (부족하면 질문) → C. Report Structuring (7섹션) → D. Decision Engine (대표용) → E. Growth Engine (직원 성장)

Engine A — Input Parser

직원/대표 식별, 프로젝트 추출, 보고 유형 분류: daily_report · meeting_note · research_report · decision_request · support_request · weekly_update

Engine B — Report Completion 가장 중요

보고가 "좋은 보고"인지 7기준으로 검사하고 부족한 부분만 질문한다.

  • 7기준: Task / Output / Outcome / Decision Needed / Support Needed / Blocker / Next
  • 질문 상한: Daily ≤ 3 · Weekly ≤ 5 · Monthly ≤ 7 (피로 최소화)
  • 충분하면 질문 0개로 통과. 코칭·평가·지적 금지, 사고를 비추는 질문만.

Engine C — Report Structuring

보완 답변까지 반영해 최종 7섹션 보고를 생성한다.

  1. Task — 오늘 한 일
  2. Output — 생산한 결과물
  3. Outcome — 그 결과물이 만든 변화·의미
  4. Decision Needed — 대표·팀장이 결정할 것
  5. Support Needed — 필요한 지원
  6. Next Action — 다음 액션
  7. ARISA Reflection — 잘된 점 / 부족한 점 / 다음 보완점 (평가가 아닌 관찰)

Engine D — Decision Engine

대표가 무엇을 결정해야 하는지 추출·정렬.

{ "project": "넵킨",
  "decision_type": "positioning",
  "decision_needed": "정보형/캐릭터형 결정",
  "urgency": "high",
  "owner": "최원석",
  "source_employee": "정예인" }

Engine E — Growth Engine

직원 사고 성장 6지표 축적.

{ "employee": "정예인",
  "clarity": 3, "evidence": 2,
  "structure": 3, "output_quality": 3,
  "outcome_quality": 2,
  "decision_thinking": 2,
  "growth_note": "의미·의사결정 연결 약함" }

아키텍처

파이프라인

직원 Telegram 입력 ↓ A. Input Parser (누가/프로젝트/보고유형) ↓ B. Report Completion (충분성 검사 → 부족 시 질문) ↓ 직원에게 보완 질문 (Daily ≤3 / Weekly ≤5 / Monthly ≤7) ↓ C. Report Structuring (7섹션 + Reflection) ↓ Thinking Data 저장 (파일 로그 + 구글시트) ├─ D. Decision Engine (대표용 의사결정) └─ E. Growth Engine (직원 성장 6지표) ↓ 직원 피드백(거울) + 대표 리포트(Daily/Weekly/Monthly)

입력 채널

  • 현재: 텔레그램 텍스트 / 음성 / 첨부파일(드라이브 업로드)
  • 후속: 메일(Gmail), 회의록(meeting_note — arisa-project-memory 흡수)
  • 다중 입력은 ingest/ 어댑터로 공통 포맷 정규화 후 큐 투입

저장 구조

arisa/
├── memory/
│   ├── employees/<직원>/        ← Thinking Data Log (daily 누적, growth_log)
│   ├── projects/<프로젝트>/      ← 결정·전략·진행 (arisa-memory 흡수)
│   ├── decisions/               ← Decision Engine 출력 누적
│   └── organization_patterns/   ← Monthly 패턴 분석
├── outputs/
│   ├── employee/{daily_feedback,weekly_reviews,monthly_growth}/

ARISA 2.0은 기존 일일업무보고봇을 진화시켜 시작했다. (신규 창조 아님)

MVP1 완료 — daily-report-bot.py

  • Engine B: COMPLETION_CHECK_PROMPT + WAITING_COMPLETION 상태. 부족 시 ≤3개 질문.
  • Engine C: STRUCTURE_PROMPT. 업무별 outcome + decision_needed/support_needed/reflection. 7섹션.
  • 직원 회신(거울): format_report_for_employee(). 직원 본인에게도 정리본 + Reflection 회신 → 학습 루프.

운영

  • launchd 영구 등록 com.arisa.daily-report-bot (RunAtLoad+KeepAlive, 재부팅 자동부활).
  • 주의: 텔레그램은 토큰당 polling 1개만 허용 — 수동 인스턴스 잔존 시 충돌.
  • 구글시트 헤더 신규 컬럼 반영 완료.

Part 11

ARISA 3.0 — Organizational Learning OS

3.0은 보고 시스템이 아니라 조직의 학습 속도를 측정·가속하는 운영체제다. SaaS 진화(기록→관리→자동화)의 다음, 4세대 = 학습 OS를 지향한다.

KPI 전환 — 3.0의 핵심
업무 완료율 ❌  →  조직의 학습 속도 ✅
매출·업무량·출근은 측정해도, 질문의 수준·사고의 성장·조직 이해도는 아무도 측정하지 못한다. ARISA 3.0이 그것을 측정한다.

다섯 가지 진화

  1. 질문 기반 조직(Question Report) — "오늘 무엇을 했나" → "오늘 발견한 가장 중요한 질문은? / 잘못 이해했던 것은? / 생각이 바뀐 것은?"
  2. 성장 해상도 측정(Thinking Resolution) — 사고 해상도·문제 정의 수준·의사결정 수준 추적 (개인 학습유형: 사례형/질문형/실험형/독서형)
  3. Organizational Define Cost 측정(오해 지도) — 대표 지시→팀장→직원 이해의 차이를 측정해 "어디서 오해가 발생하는가"를 지도화
  4. 집단지성 지도(조직 사고 맵) — 회의·보고·기획·응대 수집 → "이번 달 가장 많이 고민한 문제 / 성장한 영역 / 반복 실패한 영역"
  5. AI 코치 — 강사가 아니라 코치. 정답 대신 질문→실행→피드백→재정의 루프로 성장
최종 정의 — 조직의 학습 속도를 측정하고, 질문 수준을 향상시키고, 사고 해상도를 높이며, 조직의 Define Cost를 감소시키는 AI 기반 성장 운영체제(Learning OS). 진화 경로: 보고 시스템 → 성장 시스템 → 조직 학습 OS → Collective Intelligence.

시장 관점: 업무 툴 시장이 아니라 "조직 성장 시장"이라는 새 카테고리. CFD·Define Cost는 뒤에 숨은 철학, ARISA는 그 철학이 시장에 처음 나타나는 상품.

부록 A

기존 ARISA 사고프레임 계승

정량 엔진(보고·의사결정·성장) 아래에는 기존 ARISA GPT의 정성 사고프레임이 깔려 있다. Reflection·Decision의 질문 품질은 여기서 나온다.

  • 청취 / 응답 모드 — 호출될 때만 말한다. "질문 폭탄 금지"·Mirror 80% 원칙의 뿌리.
  • Desire Mapping (5단계) — Signal → Scene → Desire → Decision → Evolution. Outcome·Decision을 욕망 흐름으로 해석.
  • Double Layer — Layer A(욕망/브랜드) × Layer B(수익/비즈니스). 보고의 의미를 양면으로.
  • 패턴 감지 — 자산-정체성 역설 / 아키텍처 갈림길 / 데이터→질문 전환 / 설명 과잉 / 공간=미디어.
  • 데이터→질문 전환은 2.0 핵심 동작과 정확히 일치: 데이터를 해석으로 닫지 말고 "강화할 것인가, 재조정할 것인가"의 선택적 질문으로 전환.
2.0은 이 프레임을 "정량 파이프라인"으로 제품화한 것이며, 프레임 자체를 대체하지 않는다.
