# ARISA Daily Report 2.0 — 고도화 설계서

작성: 2026-07-11 · 기준 문서: 대표 공유 "ARISA Daily Report 2.0" 정의서
대상 코드: `00-system/02-scripts/daily-report-bot.py` (2.0-MVP1, 맥미니 launchd `com.arisa.daily-report-bot`)

---

## 1. 시스템 재정의 (기준 문서 요약)

ARISA Daily Report는 보고서를 제출받는 시스템이 아니라, 직원이 하루를 복기하며
**목표→활동→결과→사실/해석 분리→리스크→결정/지원 요청→다음 액션**을 스스로 판단하도록 돕는
AI 업무 코치다. 답변을 그대로 저장하지 않고, 누락·모호함·사실/의견 혼재를 검토해
추가 질문으로 보고 품질을 높인 뒤 최종 보고를 생성한다.

세 가지 목표:
1. 조직 현황의 정확한 파악 (대표·리더가 재질문 없이 상태/리스크/결정사안 파악)
2. 직원의 업무 사고 훈련 (결과/활동 구분, 사실/의견 구분, 리스크 조기 발견 등)
3. 조직 운영 데이터 축적 (진행 이력, 병목, 보고 품질 추이, 오류 패턴)

## 2. As-Is (현재 구현 상태)

플로우: `/report` → 업무 나열 → GPT 정리·확인 → 핵심 업무 드릴다운(프로세스/도구/산출물)
→ 이슈 → 내일 → 오늘의 질문 → **Engine B**(루브릭 채점 + 부족 항목 질문 최대 3개)
→ **Engine C**(7섹션 구조화: Task/Output/Outcome/Decision/Support/Next/Reflection)
→ 직원 회신(거울) + 관리자 전송 + 시트 저장 + Engine D(decisions.jsonl)

문서와 이미 일치하는 부분:
- 활동↔결과 구분 (Task→Output→Outcome, 더미/동어반복 탐지)
- 질문 피로 최소화 (최대 3개, 충분하면 0개)
- Report Score 100점 루브릭 (context20/objective10/evidence15/priority15/risk10/decision20/support10)
- 코치 톤 (평가·지적 금지, 사고를 비추는 질문)
- 개인 피드백 최소화 (Reflection + gap 1개)
- 창작 금지 원칙 (없는 내용 지어내지 않음, raw 원문 보존)

## 4. 로드맵

### Wave 1 — 검증 레이어 강화 (프롬프트+규칙, 플로우 불변) ← 이번 구현
질문 상한 3개·기존 대화 흐름은 그대로 두고, 탐지·질문 생성의 "눈"만 밝게 만든다.

1. **모호 표현 탐지 (G2)**: 규칙 기반 패턴("진행 중", "확인 중", "거의 완료",
   "~것 같다", "이야기했다/논의했다" 등)을 산출물·이슈·블로커·내일 필드에 적용 →
   날짜·수치·주체·결과를 묻는 질문 생성. LLM 프롬프트에도 모호 표현 금지 리스트 주입.
2. **FACT/해석 분리 (G1)**: SCORE_PROMPT evidence 규칙 강화 — 타인의 의사·감정을
   단정하는 문장("클라이언트가 원한다", "~할 것 같다")을 발견하면
   "직접 들은 발언인가요, 그렇게 해석하신 건가요?" 유형의 분리 질문을 gaps에 생성.
3. **리스크 조건부 재질문 (G3)**: "없음"이어도 보고 안에 ①마감 3일 이내 ②외부 회신
   대기 ③승인/견적 미확정 ④일정 변경 ⑤예산 초과 단서가 있으면 risk 만점 금지 +
   영향·대응을 묻는 질문 생성.
4. **다음 액션 기한 검증 (G5)**: tomorrow에 기한 표현(날짜/시간/까지/요일)이 없거나
   "계속 진행"류 추상어면 "언제까지, 어떤 행동으로?"를 질문. 규칙+프롬프트 이중.
5. **피드백 우선순위 (G8)**: gaps 정렬 기준을 문서 우선순위로 교체 —
   ①팩트·의견 혼재(evidence) ②리스크 누락(risk) ③결정 요청 누락(decision)
   ④결과 누락(context/objective) ⑤다음 액션 불명확(priority) ⑥지원(support).
   직원 회신의 "다음 보고 보완 포인트"도 이 순서를 따른다.

**결정: 루브릭은 기존 7항목 유지** (점수 데이터 연속성). 문서의 8항목 체계
(팩트정확성 20/팩트·의견분리 15)는 Wave 2에서 재검토.

### Wave 2 — 보고 유형 분기
- 보고 내용 GPT 자동분류: Type A(일반) / Type B(이슈·리스크) / Type C(의사결정 요청)
- Type B: 상황/팩트/영향/원인/대응안/추천안 심화 질문
- Type C: 옵션 A/B + 추천안 + 결정권자 + 기한 + 지연 시 영향 구조 강제
- 핵심 업무 드릴다운에 목표(왜) 질문 1개 추가
- 루브릭 8항목 체계 전환 검토

### Wave 3 — 학습 루프
- 보고별 누락 항목·반복 오류 유형을 직원별로 누적 (시트 score_detail + mem0)
- 주간 훈련 목표 1개 자동 생성 → 다음 보고 시작 시 되비춤
- 오류 감소 추이 기반 레벨(주니어/실무자/시니어) 자동 조정 → 질문 세트 난이도 분기
- 대표 브리프에 이상 신호(동일 이슈 반복, 활동만 반복, 보고 간 충돌) 섹션 추가

## 5. Wave 1 구현 상세 (코드 매핑)

| 변경 | 위치 |
|---|---|
| `_VAGUE_PATTERNS` 모호 표현 정규식 + 필드별 검사 | `_rule_based_gaps()` 확장 |
| tomorrow 기한 부재/추상어 검사 (`_DEADLINE_HINT`) | `_rule_based_gaps()` 확장 |
| evidence 분리 질문·모호 표현·리스크 조건부·기한 규칙 | `SCORE_PROMPT` |
| gaps 우선순위 = 문서 §7 순서 | `SCORE_PROMPT` + `completion_evaluate()` 정렬 |
| 폴백 프롬프트에도 동일 규칙 요약 반영 | `COMPLETION_CHECK_PROMPT` |

운영 원칙(불변):
- 질문 최대 3개, 충분하면 0개 (질문 폭탄 금지)
- 평가·지적 금지, 사고를 비추는 질문만 (거울 톤)
- LLM 실패 시에도 규칙 기반 안전망으로 최소 질문은 나감
- 없는 내용 창작 금지, 직원 원문(raw) 보존
