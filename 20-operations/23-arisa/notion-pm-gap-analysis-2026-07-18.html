0Executive Summary

한 문장 결론과 핵심 수치
ARISA는 "입력의 지능"(LLM 보고 코칭·결정 구조화)에서는 노션을 앞서지만, 노션이 검증한 "데이터의 구조"(명시적 Relation·자동 롤업·상태 표준) 층이 얇다. 현재 프로젝트↔보고↔분장은 토큰 매칭(휴리스틱)으로 연결되어 있어, 조직·데이터가 커질수록 연결 오류가 누적되는 구조다. 가장 시급한 것은 기능 추가가 아니라 ID 기반 명시적 연결 + 상태·우선순위 enum 통일이라는 기반 정비다.

1노션이 검증한 업무관리 효율 원리 7

기능 나열이 아닌 "왜 이 설계가 효율을 만드는가"의 원리 추출 · 출처: Notion Help Center
PRINCIPLE 01

단일 데이터소스 × 다중 뷰

하나의 데이터베이스를 Table·Board·Timeline·Calendar·List·Gallery·Chart 7가지 뷰로 표현. 각 뷰는 필터·정렬·그룹 설정을 독립적으로 저장하며, 어느 뷰에서 수정해도 전체에 즉시 반영.
왜 효율적인가 — 입력 1회, 소비 N회. 같은 데이터를 실무자는 칸반으로, 리더는 타임라인으로, 대표는 차트로 본다. "역할별로 다른 화면"을 데이터 복제 없이 구현해 이중 입력과 동기화 비용을 제거.
PRINCIPLE 02

Relation + Rollup 자동 집계

Relation으로 DB 간 명시적 연결(양방향 동기화, 1:N 제한 옵션), Rollup으로 연결된 데이터를 자동 집계(Count·Sum·Average·Percent 등 17종). 공식 Projects 템플릿의 "Task completion percent"가 대표 사례 — 완료 태스크 비율이 프로젝트 진행률 바로 자동 표시.
왜 효율적인가 — 집계라는 노동 자체를 소멸시킨다. 태스크 상태만 바꾸면 프로젝트 진행률·팀 부하가 저절로 갱신. 사람은 "집계"가 아니라 "판단"에만 시간을 쓴다.
PRINCIPLE 03

구조의 조직 표준화

Task DB 지정 시 Status·Assignee·Due date 3속성을 필수로 요구. Projects 템플릿은 Priority(Low/Medium/High 또는 P0/P1/P2)와 상태 파이프라인(Planning→In progress→…)을 표준 제공. "1 프로젝트 : N 태스크, 태스크는 단일 프로젝트 소속" 원칙.
왜 효율적인가 — 필드 값이 통일돼야 필터·정렬·자동화·집계가 전부 작동한다. 표준 enum이 없으면 그 위에 쌓는 모든 기능이 매칭 실패로 샌다.
PRINCIPLE 04

위계(Sub-items)와 의존성(Dependencies)

Sub-items로 태스크를 계층화(토글 중첩/평면 표시, Parents-only 필터). Dependencies는 선행-후행을 연결하고 날짜 자동 이동 3옵션("겹칠 때만 이동" / "간격 유지 이동" / "이동 안 함") + 주말 회피 지원.
왜 효율적인가 — 큰 일의 분해와 순서를 데이터로 만들면, 하나가 밀렸을 때 연쇄 일정 재계산을 시스템이 대신한다. 일정 회의가 줄어든다.
PRINCIPLE 05

자동화 트리거 (Database Automations)

"Status가 특정 값으로 변경"과 같은 트리거에 액션(속성 편집·담당자 지정·Slack 알림·페이지 생성)을 연결. Sprints는 자동 완성·자동 시작 + 미완료 태스크 처리 3옵션(다음 스프린트 이월/백로그/유지)까지 내장.
왜 효율적인가 — 상태 전이에 따르는 부수 작업(알림·다음 단계 생성·이월)의 비용을 0으로 만든다. "완료했다고 알려주는 일"이 사라진다.
PRINCIPLE 06

AI 속성 (Autofill)

DB에 AI 속성 추가 — AI Summary(자동 요약)·AI Key info(액션 아이템 추출)·AI Translation·커스텀 프롬프트. 실행 시점 선택(수동/페이지 생성 시/편집 시).
왜 효율적인가 — 사람이 쓴 비정형 내용에서 구조화된 메타데이터를 기계가 뽑는다. 입력 부담을 늘리지 않고 데이터 품질을 올리는 방향.
PRINCIPLE 07

맥락과 실행의 동거

모든 DB 행이 곧 문서 페이지 — 태스크 안에 논의·산출물·회의록이 함께 산다. Wiki·Docs·Projects가 한 워크스페이스에 있어 "less context switching"을 핵심 가치로 내세움. GitHub·Slack·Figma 연동으로 산재 정보를 허브화.
왜 효율적인가 — 실행(태스크)과 맥락(왜·어떻게·결과물)이 한 곳에 있으면 히스토리 탐색 비용이 급감한다. "그 파일 어디 있지"가 사라진다.
+ 관찰

점진적 구조화

노션은 자유 문서로 시작해 필요할 때 DB로 승격시키는 경로를 열어둔다. 템플릿·폼(Form)으로 입력 진입 장벽을 낮추고, 구조는 나중에 강화한다.
왜 효율적인가 — 처음부터 완전한 구조를 강요하면 입력이 죽는다. "일단 쓰게 하고, 구조는 시스템이 따라간다"는 순서가 채택률을 만든다. ARISA의 텔레그램 자연어 보고와 같은 철학.

2원리별 ARISA 현황 매트릭스

ARISA 현황은 실제 운영 파일 기준으로만 기술 (daily-report-bot.py · daily-brief-aggregate.py · dashboard-server.py · 프로젝트 JSON SSOT · 분장 시트 · decisions.jsonl)
노션 원리ARISA 현재 상태 (실측)판정갭 요지
01 단일소스 × 다중 뷰 역할별 자동 뷰는 이미 구현 — 대표(전체 병합 브리프+우선파악존+지연·결재), 리더(팀 홈: 팀 Todo·AI 분장·프로젝트 패널·팀원 보고), 직원(내 업무·오늘 Brief). 단일 URL 로그인으로 화면 자동 분기. 단, 표현 형식은 카드·리스트뿐 — 태스크에 start/end 날짜가 있는데도 타임라인·칸반·캘린더 렌더 없음 부분 "누가 보느냐"별 뷰는 우위, "어떤 형식으로 보느냐" 전환이 부재
02 Relation + Rollup 프로젝트 JSON에 tasks[].progress 숫자와 대시보드 프로젝트 패널 업무진행률 존재. 그러나 프로젝트↔보고↔분장 연결은 _match_project 토큰 매칭(정규화 포함 or 유의토큰 교집합 + 스톱워드) — ID 기반 명시 연결이 아님. "프로젝트 오인 방지"용 프롬프트 보정 커밋(f89b06c)이 이미 필요했던 상태 갭 연결이 휴리스틱 → 집계 신뢰도가 매칭 정확도에 종속
03 구조 표준화 상태 값 혼재: 프로젝트 JSON은 영문("Not Started"/"In Progress"), 분장·대시보드 뱃지는 한글(미착수/진행중/완료). 분장 시트에 우선순위 열이 있으나 표준 enum·정렬·강제 없음. 브리프 우선 정렬은 키워드 휴리스틱(결재·비용 등 문자열 탐지) 갭 표준 enum 부재 → 필터·집계·자동화의 기반 불안정
04 위계·의존성 프로젝트→태스크 1단 위계뿐. 분장 "프로젝트 그룹핑"(그룹 에디터)으로 묶음 입력은 해결. Sub-task·선후행 의존성·날짜 자동 shift 없음 부분 10인 조직에선 의존성은 저순위 — 마감 연쇄만 고려
05 자동화 트리거 보고 기반 자동 완료(커밋 4321f7c — 자연어 보고→분장 완료 매칭)는 노션 자동화보다 고차원. 미완료 분장 2주 윈도우 = 스프린트 이월 사상 유사. 지연 감지(exec-attn D+N)·리마인드(20/22시) 가동. 단, 상태 변경 이력(audit log) 없음 — 언제 누가 상태를 바꿨는지 추적 불가. 지각 제출 사각지대(배치 이후 제출 영구 미반영)도 미해결 부분 전이 자동화는 우위, 전이 "기록"이 없음
06 AI 속성 노션 AI autofill(요약·키정보 추출)보다 깊음 — 4-Engine 보고 코칭(팩트/해석 분리·모호 표현 차단·기한 검증), 보고 유형 A/B/C 자동분류, Report Score, 브리프 7범주 분류, 결정 4필드 구조 추출 우위 유지 — ARISA의 차별 핵심
07 맥락과 실행의 동거 태스크가 한 줄 텍스트(akey = 날짜|업무|담당자)로만 존재 — 산출물·논의·경과가 태스크에 축적되지 않음. 보고 원문은 브리프(개인별 써머리)에 있으나 날짜 축으로만 쌓여 태스크·프로젝트 축의 히스토리 탐색이 어려움 갭 "이 업무의 경과를 한 곳에서" 보기가 안 됨

3갭 3분류

이미 우위(유지) · 빠진 부분(도입 검토) · 고려 필요(그대로 이식 금지)
✓ 이미 우위 — 노션보다 나은 것, 그대로 유지
  • 입력의 지능 — 노션은 사람이 DB 필드를 채우지만, ARISA는 텔레그램 자연어 보고를 LLM이 구조화 + 코칭(팩트/해석 분리, 모호 차단, 기한 강제). 입력 마찰이 압도적으로 낮으면서 데이터 품질을 끌어올린다
  • 결정의 구조화 — decisions.jsonl(옵션·추천안·기한·지연영향 4필드 + 14일 이월)은 노션에 대응물이 없는 의사결정 전용 파이프라인
  • 역할별 자동 뷰 — 단일 URL 로그인 → 대표/리더/직원 화면 자동 분기. 노션의 "저장된 뷰" 사상을 계정 단위로 이미 실현
  • 성장 지표 — Report Score·유형 배지·보고 품질 추적은 PM 도구가 아니라 Learning OS 영역. 노션에 없음
✕ 빠진 부분 — 노션 원리 기준 부재, 도입 검토
  • G1 · 명시적 Relation (ID 연결) HIGH 일일보고·브리프·대시보드
    보고↔프로젝트↔분장을 토큰 매칭 대신 프로젝트 ID로 연결. 보고 시 봇이 "이 보고는 R-LAB 건인가요?" 후보 확인 → ID 저장. 모든 롤업·자동완료·히스토리의 기반
  • G2 · 상태·우선순위 표준 enum HIGH 전 시스템
    상태 {미착수·진행중·보류·완료} + 우선순위 {P0·P1·P2} 단일 정의를 shared/ 모듈로 두고 JSON·시트·뱃지·정렬이 전부 이것만 참조. 키워드 휴리스틱 정렬을 enum 정렬로 교체
  • G3 · 진행률 자동 롤업 HIGH 대시보드·프로젝트 JSON
    노션 "Task completion percent" 방식 — 프로젝트 progress를 수기가 아닌 태스크 완료 수/전체 수로 자동 산출. 보고 기반 자동 완료(이미 있음)와 결합하면 "보고만 하면 프로젝트 진행률이 굴러가는" 완결 루프
  • G4 · 타임라인 뷰 MEDIUM 대시보드 셸
    태스크 start/end·dday 데이터가 이미 있음 — 렌더만 추가하면 됨. 대표 우선파악존의 "무엇이 밀리고 있나"를 시각으로 전환
  • G5 · 상태 변경 이력 (audit log) MEDIUM arisa2 Decision Window
    상태 전이를 append-only 로그로 기록(누가·언제·무엇→무엇, decisions.jsonl과 동일 패턴). 인라인 편집·본인 삭제 기능이 생긴 지금(커밋 88bfb73·7978e7c) 변경 추적이 없으면 분쟁·유실 시 복원 불가
  • G6 · 이월 자동화 (carry-over) MEDIUM 브리프·주간 배치
    노션 스프린트의 미완료 처리 3옵션 사상 — 주간 경계에서 미완료 분장 자동 이월 + 지각 제출 재집계(현재 "배치 이후 지각 제출 영구 미반영" 사각지대 해소 포함)
  • G7 · 태스크 히스토리 뷰 (맥락 동거) LOW 대시보드
    태스크 클릭 → 관련 보고 원문·산출물 링크 시간순 스레드. G1(ID 연결)이 되면 쿼리만으로 구현 가능 — 선행 조건부 저순위
  • G8 · 칸반 뷰 LOW 대시보드
    상태 enum(G2) 이후 그룹핑 렌더로 저비용 추가 가능. 단, 10인 조직에서 리스트 대비 효용 검증 필요
  • G9 · 의존성 + 날짜 shift LOW 보류
    선후행 연쇄 재계산은 현재 조직 규모·업무 성격(짧은 주기 반복)에서 과잉 — 도입 보류 권고
⚠ 고려 필요 — 노션 방식을 그대로 이식하면 안 되는 것
  • 입력은 대화, 구조는 시스템 — 노션은 사람이 DB에 직접 입력하는 모델. ARISA에서 직원에게 폼·DB 입력을 요구하면 채택률이 죽는다. G1~G3 도입 시에도 입력 채널은 텔레그램 자연어 유지, 구조화는 봇 뒤에서 (확인 질문 1개까지만 허용)
  • 뷰 무한 증식 금지 — 노션의 "누구나 뷰를 만든다"는 유지보수 주체가 있는 조직의 것. ARISA는 역할별 고정 뷰 체계가 강점 — 형식 전환은 타임라인 1개만 선별 추가하고 뷰 커스터마이즈는 열지 않는다
  • Sub-task 계층 대신 프로젝트 그룹핑 — 노션식 다단 위계는 10인 조직에 관리 비용만 추가. 이미 있는 분장 프로젝트 그룹 에디터가 같은 문제를 더 싸게 푼다 — 유지
  • 자동화는 "기록 우선, 액션 절제" — 노션식 트리거→알림 남발은 텔레그램 채널 특성상 피로도 직결(이미 리마인드 20/22시 + 브리프 발송 존재). 신규 자동화는 알림 추가보다 로그 축적(G5)과 배치 반영(G6) 우선

5논의할 것

이 리포트가 답이 아니라 입력 — 아래 4가지가 후속 구현의 결정 지점
Q1. Phase 1(G1·G2·G3)을 한 묶음으로 진행할까, G2(enum)만 먼저 끊을까?
G1은 보고 봇 대화 흐름(확인 질문 1개 추가)에 손을 대므로 직원 체감 변화가 있음. G2·G3는 내부 구조라 무체감. 리스크를 나누려면 G2→G3→G1 순차도 가능.
Q2. 보고 시 프로젝트 확인 질문 — 허용 가능한 마찰인가?
"이 보고는 [R-LAB] 건으로 기록할까요?" 버튼 1탭 수준으로 설계 가능. 다만 질문 상한 3개 원칙과 충돌하지 않게 채점 질문과 별도 카운트 여부 결정 필요.
Q3. 타임라인 뷰의 1차 대상은 대표인가 리더인가?
대표 우선파악존 대체/보완이면 대표 먼저, 팀 일정 조율 목적이면 리더 팀 홈 먼저. 어느 쪽이 실사용될지에 따라 배치가 달라짐.
Q4. 상태 변경 이력(G5)의 범위 — 분장만? 프로젝트·결정까지?
최소는 분장 상태 전이만(인라인 편집·삭제 기능이 생긴 직후라 시급). 확장하면 프로젝트 상태·decisions 상태까지 동일 로그 포맷으로 통합 가능.
