0이 문서의 위치

"DB가 먼저고, 챗봇은 DB를 사용하는 인터페이스다."

대부분의 AI 운영 프로젝트가 실패하는 건 챗봇부터 만들기 때문이다. 우리가 만드는 건 챗봇 프로젝트가 아니라 운영 지식 체계 구축 프로젝트다. 그래서 순서는 운영 DB → 운영 규칙(루틴) → 운영봇 → 이벤트 로그 → 대시보드이고, 이 문서는 그중 맨 앞 두 칸을 잡는다.

전체 단계는 운영 지식 체계 → 운영 실행 체계 → 운영 인텔리전스 체계 순서로 간다. 이 문서(Phase 0~1)가 잘못되면 이후 챗봇·대시보드·리포트가 전부 흔들린다. 첫 2주는 개발 기간이 아니라 운영 OS 설계 기간이다.

1설계 원리 — REM × 스타벅스

검증된 뼈대(스타벅스)에 우리 철학(REM)을 얹는다. 둘이 만나면 "운영하면서 학습하는" 매뉴얼이 된다.

스타벅스에서 가져오는 것 (실행의 정밀함)

  • 이름 붙은 루틴 — 모든 반복 작업을 'OO 루틴'으로 표준화(우유 스티밍 루틴처럼)
  • "왜"까지 가르친다 — 단계 + 핵심포인트 + 이유(Teaching Routine)
  • 됨 / 아직 — 평가는 점수가 아니라 관찰형 이분법
  • 무한 변수 → N스텝 — 컴플레인은 LATTE처럼 공식으로 압축
  • 시간이 박힌 교육 — 누가 몇 시간 투입하는지 명시

REM에서 더하는 것 (학습의 깊이)

  • 운영 = 경험 설계 — "왜 멈추고, 왜 찍고, 왜 다시 오는가"를 현장에서 본다
  • 운영 = 학습 루프 — 실행→관찰→기록→복기→개선이 곧 교육
  • 봇은 감시가 아니라 사고 파트너 — "다음엔 뭘 바꿔볼까?"
  • 반복 질문 구조 — 정답 암기보다 좋은 질문의 반복
  • 현장형 경험 설계자 육성 — 단순 숙련공이 아니라 해석가
결합 결과: 스타벅스의 4층(철학→원칙→루틴→체크리스트)에 REM의 5번째 동작 '관찰·복기'를 실행 단계 안에 끼워넣는다. → "운영하면 자동으로 학습 데이터가 쌓이는" 매뉴얼.

2운영 언어 4층 구조 ★

매뉴얼을 두꺼운 규칙집이 아니라 위에서 아래로 연역되는 4개 층으로 짠다. L3가 REM의 차별점.

L0
철학

우리는 운영하지 않는다 — 경험을 설계한다 (REM)

Basket은 커피·상품을 파는 곳이 아니라, 사람의 인식을 설계하는 매체다. 모든 하위 규칙은 이 한 줄에서 나온다.

L1
원칙

운영 원칙 — 판단 기준

변수 많은 현장에서 직원이 스스로 결정할 기준. (예: "고객 경험 > 규정", "기록되지 않은 노하우는 자산이 아니다", YES 해도 되는 경계선)

L2
실행

루틴 & SOP — 이름 붙은 절차

오픈·마감·재고·청소·응대를 'OO 루틴'으로 표준화. 일/주/월 주기. 각 SOP는 단계+핵심포인트+왜+완료기준(됨/아직).

L3
학습

관찰 & 복기 — 운영이 곧 학습 (REM 차별점)

모든 루틴 끝에 관찰 질문 1개를 박는다(예: 마감 시 "오늘 사람들은 어디서 멈췄나?"). 이 답이 Event Log → 인사이트로 축적된다. 보고서를 쓰는 게 아니라, 운영하면 보고서가 생긴다.

가로지름 — 거버넌스(L4): 누구도 DB를 직접 수정하지 않는다. 변경 요청 → 검토 → 대표 승인 → 반영. (§7)

6루틴 태스크 DB + 생성 로직

"오늘 뭐 해야 하나요?"에 DB가 답하려면, 업무가 주기·트리거와 함께 정의돼 있어야 한다.

매일 09:00→오픈 업무 카드 생성
매주 금요일→주간 보고 생성

지금 단계에서 코드는 없다. 규칙(로직)만 DB로 먼저 정의해 두면, Phase 2에서 봇이 그대로 읽어 "오늘 할 일"을 만든다.

7DB 거버넌스 — 공동 자산 보호

운영 DB는 개인 자료가 아니라 프로젝트 렌트의 공동 자산이다. 아무나 직접 못 고친다.

직원 변경 요청→ 검토→ 대표 승인→ DB 반영 (버전↑)

요청은 봇/시트로 받고, 반영 시 SOP의 버전·변경이력이 올라간다. "나는 이렇게 해요"가 아니라 "우리의 표준은 무엇인가"를 정의하는 관점으로 참여한다.

운영팀 워크시트 (이번 주에 채워 주세요)

"왜 하는지 → 무엇을 만드는지 → 무엇을 해야 하는지"

왜 — 우리 운영 노하우가 사람의 기억에만 있으면, 한 사람이 빠지면 품질이 흔들리고 신입은 매번 처음부터 배운다. 이걸 공동 자산(DB)으로 만든다.

무엇을 — 챗봇·대시보드가 아니라, 그 전에 운영 데이터베이스 자체. 위 8개 DB.

무엇을 해야 — 운영팀이 채울 4가지:

  • ① 내 업무 리스트 — 하루/주/월에 하는 모든 일을 빠짐없이
  • ② 반복 업무 — 매일·매주·매월 꼭 하는 것 + 주기
  • ③ 자주 나는 문제 + 해결법 — 겪었던 사례(상황→대응→결과)
  • ④ 꼭 알아야 하는 정보 — 장비 사용법·AS·협력사·비상대응

참여 관점 — "나는 이렇게 해요"가 아니라 "우리의 표준은 무엇인가". 그리고 REM 한 줄: 업무를 적을 때 "이 업무에서 손님은 무엇을 느끼나"도 같이 적어 주세요. 그게 ⑧ 관찰 DB의 씨앗입니다.

9다음 단계 연결 (Phase 2~)

2주 종료 시점 — 아직 챗봇도 대시보드도 없다. 하지만 DB·SOP·루틴·거버넌스가 존재한다. 이 상태가 되면:

이 DB→ Operations Bot(인터페이스)→ Event Log→ Daily/Weekly Report→ Store / Executive Dashboard→ Operations Intelligence

REM 봇의 성격(설계 원칙) — 평가·감시 시스템처럼 느껴지면 실패한다. 봇은 사고 파트너다. "오늘 뭐 해야 하나요?"에 DB로 답하고, 마감엔 복기 3문을 묻고, "다음엔 뭘 바꿔볼까?"로 사고를 유도한다. 업무를 늘리는 게 아니라 더 깊게 보게 한다.

MVP 종료 기준 — 직원이 "오늘 뭐 해야 하나요?"를 묻고 실제 업무를 수행할 수 있으면 Phase 2 성공. 그 전제는 이 문서의 DB가 먼저 존재하는 것.
