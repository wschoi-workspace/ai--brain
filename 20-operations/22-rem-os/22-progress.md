# REM OS — Retail Experience Management Operating System

> 목적: 프로젝트렌트 현장 운영팀의 업무·사고·학습 통합 시스템 설계
> 성격: 진화형 — 논의가 깊어질 때마다 업데이트
> 시작일: 2026-06-09

---

## 핵심 정의

**REM OS** = 매장 운영 매뉴얼이 아닌, "같은 관찰 프레임을 공유하는 조직"을 만드는 시스템

> "사람들이 왜 반응하고, 왜 기억하고, 왜 소비하고, 왜 다시 찾아오는가"를 현장에서 실험하고 학습하는 시스템

### WHY LAYER — 가장 중요한 전제

- **"더 열심히가 아니라 더 깊게"** — REM은 노동량 증가가 아닌 관찰 해상도 증가
- 업무 증가처럼 느껴지면 실패. 감시 시스템처럼 느껴져도 실패
- 직원이 느껴야 하는 건 "회사 때문에 억지로"가 아니라 **"이건 내 성장에도 도움이 된다"**
- 사람은 철학 때문에 움직이지 않는다. **"내 행동이 결과를 바꿨다"를 체감할 때 움직인다**
- 참조: `rem-os-master-frame.md` (ARISA 회의 확정본, 10개 섹션)

## 교육학적 프레임워크

| 프레임워크 | REM OS 적용 포인트 |
|-----------|-------------------|
| 구성주의 (Constructivism) | 강의가 아닌 "경험→복기→해석" 루프 설계 |
| 반성적 실천 (Schön) | 업무보고 = 반성적 사고 인터페이스 |
| 디자인 싱킹 (Empathy) | "왜 멈췄는가?" 관찰→추론 훈련 |
| 스캐폴딩 (Scaffolding) | L1"뭘 봤나"→L4"다음엔 어떻게" 단계적 사고 |
| 상황기반학습 (Situated) | 실제 프로젝트 안에서 작은 실험 권한 부여 |
| 실천공동체 (CoP) | "좋은 관찰을 공유하는 문화" = 조직 감각 형성 |

---

## 설계 원칙

1. **높은 철학 → 낮은 행동 단위로 번역** — "인식을 설계하라" → "고객이 반복한 단어를 기록하라"
2. **AI = 감시자가 아닌 사고 파트너** — "점수 부족" → "가장 기억 남는 반응은?"
3. **작은 성공 경험 축적** — 좋은 관찰이 좋은 결과로 연결된다는 체험이 체화의 핵심
4. **정답 시스템이 아닌 질문 시스템** — 프로젝트마다 정답이 달라지므로 "좋은 질문을 반복하는 조직"

**C. 우선순위 — 운영부터 시작, 단계 확장**

| Phase | 범위 | 봇 역할 | 직원 기대 수준 |
|-------|------|--------|--------------|
| Phase 1 | Ch1(운영) + Ch6(프로젝트 실행) | 체크리스트, 이슈보고, 정산 가이드 | "물어보면 답해주는 봇" |
| Phase 2 | + Ch4(관찰) + Ch3(브랜드 해석) | 일일보고 질문, VOC 유도, 해석 질문 | "질문해주는 봇" |
| Phase 3 | + Ch8(복기) + Ch5(실험) + Ch9(성장) | 패턴 피드백, 실험 제안, 성장 트래킹 | "함께 생각하는 봇" |

핵심: Phase 1에서 신뢰를 얻어야 Phase 2의 질문을 받아들인다.

**D. 프로젝트별 커스터마이징**
- 두꺼운 가이드보다 **"인식 브리프 1장 + 공통 운영 매뉴얼"** 조합
- 메시지 본질이 프로젝트 형태보다 중요할 수 있음 (추후 프로젝트별 고도화)
- 인식 브리프 = 인식 목표 + 관찰 포인트 + 실험 포인트 (1장 양식)

**ARISA 분화 구조 제안**
- 경영진 ARISA: 전략 사고 파트너, 회의 정리, 브랜드 시뮬레이션
- 현장 REM Bot: 운영 지원, 관찰 질문, 일일보고, 브랜드 해석
- 데이터 레이어에서 연결: 관찰→패턴이 경영진에게 올라감, 인식 브리프가 현장으로 내려감
